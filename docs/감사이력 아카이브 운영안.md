# 감사이력 아카이브 운영안

## 1. 목적

- `om_audit_log_h`의 데이터 증가에 따른 성능 저하를 예방한다.
- BO/OM 공용 감사이력의 조회 성능과 저장 성능을 안정적으로 유지한다.

## 2. 기본 원칙

- 감사 원본 테이블: `om_audit_log_h` (월 파티션)
- 아카이브 테이블: `om_audit_log_h_archive`
- 보관 정책(권장):
  - 운영(핫): 최근 12개월
  - 아카이브(콜드): 12개월 초과 데이터

## 3. 월별 운영 작업

- 매월 말(또는 월초) 다음 월 파티션 사전 생성
  - `SELECT fn_om_audit_log_h_create_month_partition((CURRENT_DATE + INTERVAL '1 month')::date);`
- 아카이브 기준월 이전 데이터 이관(예: 12개월 초과)
  - `SELECT fn_om_audit_log_h_archive_month((CURRENT_DATE - INTERVAL '13 months')::date);`

## 4. 권장 스케줄

- 파티션 생성: 매월 25일 01:00
- 아카이브 실행: 매월 1일 02:00
- 실행 주체: DB 배치 스케줄러(pg_cron/외부 배치)

## 5. 실행 전/후 점검

- 실행 전:
  - 대상 월 건수 확인
  - 최근 슬로우쿼리/락 대기 확인
- 실행 후:
  - 이관 건수 검증(원본/아카이브 합계 일치)
  - `ANALYZE om_audit_log_h;`
  - `ANALYZE om_audit_log_h_archive;`

## 6. 장애 대응

- 아카이브 함수 실패 시:
  - 해당 월 재실행 가능(함수는 월 기준 처리)
  - 실패 월은 운영 테이블에 그대로 남아 서비스 연속성 유지
- 대량 이관으로 부하가 높은 경우:
  - 업무 비시간대 실행
  - 월 단위를 주 단위/일 단위로 쪼개서 실행

## 7. 주의사항

- 아카이브 실행은 감사 저장 트랜잭션과 분리된 배치로 운영한다.
- 법/컴플라이언스 정책 변경 시 보관 기간을 우선 조정한다.
- 운영 환경 적용 전 스테이징에서 월 단위 복원/검증 테스트를 수행한다.
