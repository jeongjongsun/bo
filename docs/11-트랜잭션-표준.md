# 트랜잭션 표준

Spring Boot 백엔드 서비스에서 데이터 정합성과 동시성 안전성을 확보하기 위한 트랜잭션 규칙입니다.

## 적용 목적

- 여러 SQL이 하나의 업무 단위를 이룰 때 원자성(모두 성공/모두 실패) 보장
- 동시 요청 환경에서 부분 반영·중복 처리·정합성 깨짐 방지
- 서비스 계층의 트랜잭션 경계를 일관되게 유지

## 기본 원칙

- 트랜잭션 경계는 **Service 계층**에서 관리한다.
- **여러 테이블에 쓰기**가 발생하면 `@Transactional`을 적용한다.
- 한 서비스 메서드에서 **조회→검증→계산→쓰기**가 이어지는 경우 `@Transactional`을 적용한다.
- 단순 조회는 `@Transactional(readOnly = true)`를 기본으로 한다.
- Controller, Mapper 계층에는 트랜잭션 경계를 두지 않는다.

## 필수 적용 기준

### 1) 쓰기(INSERT/UPDATE/DELETE)

아래 중 하나라도 해당되면 메서드에 `@Transactional`을 적용한다.

- 2개 이상 테이블에 쓰기
- 1개 테이블이라도 다건/배치 변경
- 상태 전이(예: A -> B) 처리
- 전처리 조회 결과에 따라 후속 쓰기가 결정되는 경우

### 2) 읽기(SELECT)

- 단순 목록/상세/옵션 조회 메서드는 `@Transactional(readOnly = true)` 권장
- 엑셀 full export 같은 장시간 읽기 작업도 `readOnly`를 적용한다.

## 동시성 처리 규칙

- 상태 전이 UPDATE에는 `WHERE current_status = ?` 조건을 포함한다.
- 처리 결과 메시지는 요청 건수와 실제 반영 건수를 구분해 안내한다.
- 부분 성공(경합으로 일부 미반영)은 정상 시나리오로 간주하고 사용자에게 명확히 고지한다.

## 금지/주의 사항

- Controller에 `@Transactional` 적용 금지
- 트랜잭션 메서드 안에서 장시간 외부 I/O(원격 API 호출) 최소화
- 같은 클래스 내부 자기호출(self-invocation)로 트랜잭션 적용이 누락되지 않도록 주의

## 체크리스트

- 쓰기 메서드에 `@Transactional` 적용 여부
- 조회 메서드에 `@Transactional(readOnly = true)` 적용 여부
- 상태 전이 SQL에 기대 상태 조건(`expectedCurrentStatus`) 포함 여부
- 사용자 메시지에 요청/실반영 건수 구분 여부
