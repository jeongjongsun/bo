# 감사이력 정책

## 1. 목적

- 시스템에서 발생하는 등록/수정/삭제 이력을 표준화하여 추적성과 운영 가시성을 확보한다.
- 누가, 언제, 어떤 데이터를 변경했는지 일관된 형태로 보관한다.

## 2. 적용 범위

- 백엔드 서비스 계층에서 수행되는 데이터 변경 작업
  - 등록(CREATE)
  - 수정(UPDATE)
  - 삭제(DELETE)
- 단순 조회(Read) 작업은 감사이력 대상이 아니다.

## 3. 저장 원칙

- 감사이력은 파일이 아닌 DB 테이블로 저장한다.
- 공통 감사 테이블은 `om_audit_log_h`를 기준으로 사용한다.
- BO/OM 시스템은 동일 감사 테이블을 공동 사용한다.
- 도메인별 상세 감사 테이블이 이미 있는 경우(`*_audit_h`)는 점진적으로 공통 정책과 정렬한다.

## 4. 자동 등록 원칙

- 감사이력은 사용자의 별도 입력/동작 없이 자동 등록한다.
- Controller에서 직접 저장하지 않고 Service 계층에서 기록한다.
- 공통 `AuditService`를 통해 기록 로직을 일원화한다.

## 5. 트랜잭션 정책

- 본 데이터 변경과 감사이력 저장은 동일 트랜잭션에서 처리한다.
- 감사이력 저장 실패 시 본 프로세스도 실패로 처리한다(전체 롤백).
- 실패 시도(권한 거부, 입력 검증 실패 등)는 감사이력으로 남기지 않는다.
- 시스템 에러/트랜잭션 실패 로그는 별도 요건으로 분리 정의한다.

## 6. 필수 저장 항목

- 사용자 정보: `actor_user_id`
- 시각 정보: `acted_at`
- 메뉴 정보: `menu_code`, `menu_name_ko`
- 시스템 정보: `system_main_cd`, `system_sub_cd`(BO/OM)
- 액션 정보: `action_code`, `action_name_ko`
  - `CREATE` / `등록`
  - `UPDATE` / `수정`
  - `DELETE` / `삭제`
- 엔터티 정보: `entity_type`, `entity_id`
- 요청 추적 정보(가능 시): `request_id`, `request_ip`, `user_agent`

## 7. 변경 데이터 기록 기준

- 가능하면 아래 데이터를 함께 저장한다.
  - `before_data` (변경 전)
  - `after_data` (변경 후)
  - `changed_fields` (변경된 필드 목록)
- 대용량/고빈도 작업은 전체 스냅샷 대신 변경 필드 중심 저장을 우선 고려한다.

## 8. 보안/마스킹 정책

- 현재 마스킹 기준은 미정이다.
- 다만 비밀번호/토큰/인증정보 등 명백한 민감정보는 감사이력 저장 대상에서 제외한다.
- 상세 마스킹 기준은 별도 보안 정책 문서에서 정의 후 본 정책에 반영한다.

## 9. 성능 및 운영 권장사항

- 인덱스 기준:
  - `acted_at DESC`
  - `domain_type, acted_at DESC`
  - `actor_user_id, acted_at DESC`
  - `entity_type, entity_id, acted_at DESC`
- 데이터 증가량을 고려해 보관 주기 및 아카이브 정책을 운영 정책으로 정의한다.
- 필요 시 월 단위 파티셔닝 등 저장소 최적화를 적용한다.

## 10. 구현 표준 요약

- 서비스에서 데이터 변경 수행
- 변경 전/후 데이터 수집
- `AuditService.record(...)` 호출
- 본 변경과 감사이력을 동일 트랜잭션으로 커밋

## 11. BO/OM 공통 값 규칙

- `system_main_cd`: 기본값 `SYSTEM`
- `system_sub_cd`:
  - BO 화면/서비스: `BO`
  - OM 화면/서비스: `OM`
- `action_code`: `CREATE`, `UPDATE`, `DELETE`만 허용
- `action_name_ko`: `등록`, `수정`, `삭제`만 허용
- `menu_code`:
  - BO/OM 각각의 메뉴 ID를 그대로 사용
  - 권한 테이블(`om_menu_m`)의 `menu_id`와 동일 값 사용 권장
- `menu_name_ko`: 화면 기준 한글 메뉴명 고정값 사용

## 12. OM 적용 지시 기준 (개발팀 전달용)

OM 시스템에도 동일 정책으로 적용할 때, 아래 체크리스트를 모두 만족해야 한다.

### 12.1 적용 대상 메서드

- Service 계층의 데이터 변경 메서드만 대상
  - 등록(Create), 수정(Update), 삭제(Delete)
- 조회(Read), 단순 목록 조회, 통계 조회는 대상 제외

### 12.2 구현 체크리스트

1. 변경 전 데이터(`before_data`)를 조회한다.
2. 비즈니스 변경을 수행한다.
3. 변경 후 데이터(`after_data`)를 구성한다.
4. `AuditService.record(...)`를 같은 트랜잭션에서 호출한다.
5. `system_sub_cd`를 반드시 `OM`으로 저장한다.
6. `menu_code`, `menu_name_ko`, `entity_type`, `entity_id`를 누락 없이 저장한다.
7. 실패 시도(검증 실패/권한 실패)는 저장하지 않는다.

### 12.3 금지사항

- Controller/Filter/Interceptor에서 감사 INSERT 직접 수행 금지
- 감사 저장만 별도 트랜잭션으로 분리 금지
- `before_data`를 임의 `{}`로 고정 저장 금지(실제 값 조회 필수)

## 13. OM 수용 기준 (QA)

- 동일 요청에서 본 데이터와 감사이력이 함께 커밋/롤백되는가
- OM 화면의 C/U/D 수행 시 `system_sub_cd='OM'`으로 저장되는가
- `before_data`/`after_data`가 모두 실제 값으로 저장되는가
- `changed_fields`가 한글 라벨 기준으로 저장되는가
- `actor_user_id`, `acted_at`, `menu_code`, `entity_id`가 누락되지 않는가

## 14. 배포 전 체크리스트

- `docs/ddl/om_audit_log_h.sql` 최신 스키마 반영 여부
- 파티션/아카이브 함수 반영 여부
- OM 주요 메뉴 최소 1건씩 C/U/D 샘플 테스트 완료 여부
- 운영 모니터링 항목(감사 insert 실패/지연) 준비 여부
