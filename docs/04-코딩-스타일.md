# 코딩 스타일 표준 (Coding Style Standards)

Spring Boot + React 통합 시스템(Shopeasy4)의 **코드 스타일·포맷·네이밍** 표준입니다.  
일관된 스타일로 가독성과 유지보수성을 높이고, [02 개발 표준](02-개발-표준.md)의 폴더·파일 규칙과 함께 적용합니다.

## 1. 원칙

- **가독성**: 다른 사람이 빠르게 이해할 수 있도록 이름과 구조를 명확히 한다.
- **일관성**: 팀 전체가 같은 규칙을 따르면 리뷰·머지 시 논란이 줄어든다.
- **도구 우선**: 포맷팅·일부 규칙은 ESLint/Prettier/Checkstyle 등으로 자동 적용해 수동 스타일 논의를 줄인다.

---

## 2. 공통 네이밍 규칙

### 2.1 의미 있는 이름

- **역할이 드러나게** 짓는다. `data`, `info`, `temp`, `x` 같은 모호한 이름은 피한다.
- **약어**는 팀에서 통일된 것만 사용한다. 예: `id`, `num`, `msg`, `err` — 자주 쓰는 것은 허용, 그 외는 풀네임 권장.
- **불리언**은 `is~`, `has~`, `can~`, `should~` 등으로 시작하면 의미가 분명해진다.

### 2.2 길이

- **변수·함수**: 한두 단어로 역할이 드러나면 충분. 지나치게 긴 이름은 오히려 가독성을 해칠 수 있다.
- **스코프가 넓을수록** 이름을 더 구체적으로. 루프 변수 `i`, `item` 수준은 짧게, 메서드·모듈 단위는 구체적으로.

## 3. 백엔드(Java) 스타일

### 3.1 네이밍

| 대상 | 규칙 | 예시 |
|------|------|------|
| **패키지** | 소문자, 단어 구분 없음 | `com.shopeasy.order`, `com.shopeasy.api.v1` |
| **클래스·인터페이스** | PascalCase | `OrderService`, `ApiResponse` |
| **메서드** | camelCase, 동사로 시작 | `getOrder()`, `findByUserId()` |
| **변수** | camelCase | `orderId`, `totalCount` |
| **상수** | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE`, `DEFAULT_SORT` |
| **enum 상수** | UPPER_SNAKE_CASE 또는 PascalCase (팀 통일) | `ERR_NOT_FOUND`, `OrderStatus.PENDING` |

### 3.2 파일·클래스

- **한 파일에 한 public 클래스**. 파일명은 해당 클래스명과 동일. (`OrderService.java` → `class OrderService`)
- **개발 표준 0.3**의 폴더 구조를 따른다. 패키지 경로 = 디렉터리 경로.

### 3.3 포맷팅 (Java)

- **들여쓰기**: 스페이스 **4칸**. (팀에서 2칸으로 통일해도 됨, 한 가지로 고정)
- **줄 길이**: **120자** 이내 권장. 초과 시 줄바꿈하고 들여쓰기 한 단계 추가.
- **중괄호**: K&R 스타일. `{`는 제어문·메서드 선언과 같은 줄, `}`는 별도 줄.
- **한 줄**: 제어문/메서드가 한 줄이어도 중괄호 생략하지 않음. (가독성·diff 일관성)

```java
// 권장
if (order == null) {
    throw new NotFoundException("ERR_NOT_FOUND");
}

// 비권장: 한 줄이라도 중괄호 사용
if (order == null) throw new NotFoundException("ERR_NOT_FOUND");
```

### 3.4 주석 (Java)

- **Javadoc**: public API(클래스, public 메서드)에는 간단한 용도 설명. 파라미터·반환값은 필요 시만.
- **한글/영문** 팀 규칙에 맞춰 통일. (개발 표준·로깅 표준과 동일하게 “왜” 위주로)

## 4. 프론트엔드(TypeScript/React) 스타일

### 4.1 네이밍

| 대상 | 규칙 | 예시 |
|------|------|------|
| **파일명 (컴포넌트)** | PascalCase | `UserList.tsx`, `OrderDetail.tsx` |
| **파일명 (비컴포넌트)** | camelCase | `useUserList.ts`, `dateUtils.ts`, `api.ts` |
| **폴더명** | 소문자 또는 camelCase | `features/users`, `api` |
| **컴포넌트** | PascalCase | `UserList`, `OrderDetail` |
| **훅** | camelCase, `use` 접두어 | `useUserList`, `useAuthMe` |
| **함수·변수** | camelCase | `fetchUserList`, `orderId` |
| **상수** | UPPER_SNAKE_CASE 또는 camelCase (모듈 스코프) | `MAX_PAGE_SIZE`, `apiBaseUrl` |
| **타입·인터페이스** | PascalCase | `User`, `ApiResponse`, `PagedData` |
| **enum** | PascalCase, 멤버는 PascalCase 또는 UPPER_SNAKE (팀 통일) | `OrderStatus.Pending` |

- **개발 표준 0.3**의 파일명·폴더 규칙과 동일하게 적용한다.

### 4.2 컴포넌트·JSX

- **함수형 컴포넌트** 사용. `function ComponentName()` 또는 화살표 함수 중 팀에서 하나로 통일.
- **props 타입**: `interface Props { ... }` 또는 인라인 타입을 명시. `any` 지양.
- **export**: 컴포넌트는 `export function UserList(...)` 형태로 이름 있는 export 권장. (디버깅·리팩터 시 유리)

```tsx
// 권장
interface UserListProps {
  page: number;
  onPageChange: (page: number) => void;
}

export function UserList({ page, onPageChange }: UserListProps) {
  // ...
}
```

### 4.3 포맷팅 (TypeScript/React)

- **들여쓰기**: 스페이스 **2칸**. (프론트 쪽 관례와 도구 기본값에 맞춤)
- **줄 길이**: **100~120자** 이내 권장. JSX가 길어지면 변수로 빼거나 컴포넌트로 분리.
- **따옴표**: 문자열은 **작은따옴표 `'`** 또는 **큰따옴표 `"`** 중 하나로 통일. (Prettier 기본은 작은따옴표)
- **세미콜론**: **사용** 권장. (ESLint/Prettier로 통일)
- **JSX**: 여러 줄일 때 괄호 `()` 로 감싸고, props가 많으면 한 줄에 하나씩 내려 쓴다.

```tsx
// 권장
return (
  <div className="user-list">
    <button onClick={() => onPageChange(page - 1)} disabled={isFirst}>
      이전
    </button>
  </div>
);
```

## 5. 포맷팅

### 5.1 공통

- **탭 대신 스페이스** 사용. (백엔드 4칸, 프론트 2칸 — 위와 동일)
- **파일 끝**: 마지막 줄 끝에 **줄바꿈 한 개** 유지. (많은 도구가 기본 기대)
- **공백**: 연산자 앞뒤 한 칸, `,` 뒤 한 칸. 불필요한 trailing space 제거.

### 5.2 빈 줄

- **함수·메서드 사이**: 한 줄 띄우기.
- **논리 블록 구분**: 관련 있는 코드끼리 묶고, 블록 사이에 빈 줄 하나. 과도한 빈 줄은 지양.

## 6. import / require

### 6.1 순서 (공통 원칙)

1. **외부 라이브러리** (react, axios, @tanstack/react-query 등)
2. **내부 절대 경로/별칭** (@/components, @/api 등)
3. **상대 경로** (./types, ../utils)
4. **타입만 가져오기** (가능하면 그룹으로 묶어서)

같은 그룹 안에서는 **알파벳 순** 또는 **도구 기본 정렬**을 따르면 된다.

### 6.2 예시 (TypeScript)

```ts
// 1. 외부
import { useQuery } from '@tanstack/react-query';
import React, { useState } from 'react';

// 2. 내부 (별칭)
import { Button } from '@/components/Button';
import type { ApiResponse } from '@/types';

// 3. 상대
import { useUserList } from './hooks';
import type { User } from './types';
```

### 6.3 Java

- **import**: `*`(와일드카드) 사용 여부를 팀에서 정한다. 한 패키지에서 5개 이상 쓸 때만 `import com.foo.*` 허용 등.
- **순서**: IDE 기본(패키지 알파벳 순) 또는 Checkstyle 규칙에 맞춘다.

## 7. 주석

### 7.1 원칙

- **“왜”**를 설명한다. “무엇을 하는지”는 이름으로 드러나게 하고, 비즈니스 규칙·예외 처리 이유·우회 처리 등만 주석으로 남긴다. (로깅 표준·개발 표준과 동일)
- **한글/영문** 팀에서 통일.
- **죽은 코드**는 주석 처리하지 말고 삭제. 이력은 Git에 남긴다.

### 7.2 주석 스타일

- **한 줄**: `//` (Java, TypeScript 공통)
- **여러 줄**: Java는 `/* ... */` 또는 Javadoc `/** ... */`, TypeScript는 `/* ... */` 또는 `//` 여러 줄.
- **TODO/FIXME**: `// TODO(이름): 내용`, `// FIXME: 내용` — 이슈 번호가 있으면 함께 기재.

## 8. 도구·자동화

### 8.1 백엔드 (Java)

- **Checkstyle** 또는 **Spotless**: 들여쓰기, 줄 길이, 중괄호, import 정렬 등을 빌드 시 검사·자동 정리.
- **빌드 시**: `./gradlew check` 또는 `mvn verify`에 스타일 검사 포함해, 위반 시 실패하도록 설정 권장.
- **IDE**: 팀에서 사용하는 포맷 설정(들여쓰기, 줄 끝)을 공유(프로젝트 설정 파일 또는 가이드 문서)해 로컬 저장 시 자동 포맷이 동일하게 적용되도록 한다.

### 8.2 프론트엔드 (TypeScript/React)

- **ESLint**: 문법·잠재 버그·팀 규칙(미사용 변수, React hooks 규칙 등) 검사. [02 개발 표준](02-개발-표준.md)의 규칙(타입 사용, any 지양 등)을 규칙으로 넣을 수 있다.
- **Prettier**: 따옴표, 세미콜론, 줄 길이, 들여쓰기 등 **포맷만** 담당. ESLint와 충돌하는 규칙은 `eslint-config-prettier`로 끈다.
- **저장 시 포맷**: IDE에서 “Format on Save” + Prettier 사용 권장. 커밋 전 `npm run lint`, `npm run format` 실행해 CI와 동일하게 맞춘다.
- **설정 공유**: `.eslintrc.*`, `.prettierrc` 등을 저장소에 두어 팀 전체가 같은 설정을 쓰도록 한다.

### 8.3 공통

- **CI**: 백엔드 check, 프론트 lint/format이 실패하면 머지 전에 수정하도록 한다.

## 9. 정리 체크리스트

| 구분 | 표준 요약 |
|------|-----------|
| **원칙** | 가독성, 일관성, 도구로 자동화 |
| **네이밍** | Java: PascalCase(클래스), camelCase(메서드·변수), UPPER_SNAKE(상수) / TS: PascalCase(컴포넌트·타입), camelCase(함수·변수·훅), 파일명은 개발 표준 0.3 준수 |
| **포맷** | Java: 스페이스 4칸, 줄 120자, 중괄호 항상 / TS: 스페이스 2칸, 줄 100~120자, 세미콜론 사용, 따옴표 통일 |
| **import** | 외부 → 내부 → 상대, 타입 별도 그룹, 순서 통일 |
| **주석** | “왜” 위주, 한글/영문 통일, TODO/FIXME 형식 |
| **도구** | Java: Checkstyle/Spotless, 프론트: ESLint + Prettier, CI에서 검사 |
