## Cursor 규칙에서 참조

위 표준은 **프로젝트 실행/개발 시마다 참고**되도록 `.cursor/rules/`에 등록되어 있습니다.

- **00-project-standards.mdc** (항상 적용): docs 전체 요약 및 필수 준수 사항.
- **01-backend.mdc**: `*.java`, `application*.yml` 편집 시 — 개발·로깅·보안·테스트·DB·코딩 표준.
- **02-frontend.mdc**: `*.ts`, `*.tsx` 편집 시 — 개발·타입·테스트·코딩·보안 표준.
- **03-database.mdc**: `*.sql`, 마이그레이션 폴더 편집 시 — DB 표준.
- **04-i18n.mdc**: 다국어 관련 파일·코드 편집 시 — i18n 표준.
- **05-backend-service-comments.mdc**: 백엔드 service/mapper 편집 시 — 함수별 Javadoc·주요 로직 주석(프로세스·개발 의도 파악).

AI 에이전트와 개발자가 동일한 표준을 참고할 수 있습니다. 전문은 위 표의 각 문서를 열어 확인하세요.

## 추가로 정의할 수 있는 표준 후보

아래는 **아직 별도 문서로 정의하지 않은** 영역입니다. 필요에 따라 선택해 표준 문서를 추가하면 됩니다.

### 1. 코딩 스타일·컨벤션 ✅ 정의됨

- **문서**: [04-코딩-스타일.md](04-코딩-스타일.md)  
  네이밍(Java/TS), 포맷팅, import, 주석, ESLint/Prettier, Checkstyle/Spotless.

---

### 2. Git·브랜치·커밋 규칙 (권장도: 높음)

- **목적**: 브랜치 전략 통일, 커밋/PR 이력으로 변경 이유 추적.
- **포함 후보**  
  - 브랜치 전략: main / develop, feature/xxx, hotfix/xxx 등  
  - 커밋 메시지: Conventional Commits( feat:, fix:, docs: 등) 또는 팀 규칙  
  - PR 규칙: 제목 형식, 필수 체크리스트, 리뷰 요청 기준  
  - 머지 전: 테스트·린트 통과 여부
- **문서 예시명**: `GIT_STANDARDS.md` 또는 `CONTRIBUTING.md`

---

### 3. 배포·환경 표준 (권장도: 중간)

- **목적**: 배포 절차·환경을 통일해 오류와 "내 로컬에선 되는데"를 줄임.
- **포함 후보**  
  - 환경 구분: local / dev / stage / prod 정의  
  - 환경 변수·설정: 어떤 값을 어디서 주입하는지, 비밀은 어떻게 다루는지  
  - 배포 절차: 빌드 → 아티팩트 → 서버 배포 순서, 롤백 방법  
  - 헬스체크 엔드포인트(예: /actuator/health), 모니터링 항목(선택)
- **문서 예시명**: `DEPLOYMENT_STANDARDS.md` 또는 `OPERATIONS.md`

---

### 4. 데이터베이스(DB) 규칙 ✅ 정의됨

- **문서**: [08-DB-표준.md](08-DB-표준.md)  
  테이블·컬럼·PK/FK·인덱스 네이밍, 공통 컬럼, 데이터 타입, 제약조건, Flyway/Liquibase 마이그레이션, 권한.

---

### 5. API 문서화·버저닝 (권장도: 중간)

- **목적**: API 계약을 문서로 유지, 버전 변경 시 혼선 방지.
- **포함 후보**  
  - API 문서: OpenAPI(Swagger) 작성 위치·주기·리뷰  
  - 버저닝: URL 경로 버전(/api/v1/), 호환 정책(하위 호환 유지 기간)  
  - Deprecation: deprecated API 표시 방법, 전환 기간·공지
- **문서 예시명**: `API_DOCUMENTATION_STANDARDS.md` (또는 개발 표준에 "API 문서화" 절 추가)

---

### 6. 성능·리소스 (권장도: 낮음 ~ 중간)

- **목적**: 응답 시간·용량을 일정 수준으로 유지.
- **포함 후보**  
  - API: 응답 크기 상한, 페이징 기본값·최대값(이미 개발 표준에 있음), 타임아웃  
  - DB: N+1 방지, 대용량 조회 시 배치/스트리밍  
  - 프론트: 번들 크기, 이미지/에셋 최적화, lazy load
- **문서 예시명**: `PERFORMANCE_STANDARDS.md` (또는 개발 표준 "성능" 절)

---

### 7. 문서화·주석 (권장도: 낮음)

- **목적**: README·아키텍처·주석 규칙 통일.
- **포함 후보**  
  - README: 프로젝트 소개, 실행 방법, 환경 요구사항  
  - 아키텍처 다이어그램: 갱신 주기·위치  
  - 코드 주석: "무엇"보다 "왜" 위주 (로깅 표준·개발 표준과 연계)
- **문서 예시명**: `DOCUMENTATION_STANDARDS.md` (또는 기존 문서에 "문서화" 절 추가)

---

### 8. 접근성(A11y)·국제화(i18n)

- **국제화(i18n)** ✅ 정의됨  
  **문서**: [09-다국어-표준.md](09-다국어-표준.md) — 지원 언어·기본값, 메시지 키·번역 파일, API message 키/서버 번역, 날짜/숫자/통화, react-i18next.
- **접근성(A11y)** (권장도: 요구 시): 키보드 조작, 스크린리더, 대비·포커스 등 — 법적·정책 요구가 있을 때 표준화.

## 정리

- **이미 커버된 것**: 개발(API·구조·React)·로깅·보안·테스트.
- **우선 추가를 권장하는 것**: **코딩 스타일**, **Git·브랜치·커밋** — 팀 협업과 코드 품질에 바로 영향을 줍니다.
- **프로젝트 성격에 따라 추가**: **배포·환경**, **DB**, **API 문서화** — 운영·DB 변경·외부 연동이 많을수록 유용합니다.
- **필요 시 추가**: 성능, 문서화, 접근성, 국제화.

새 표준 문서를 만들 때는 이 인덱스와 [02-개발-표준.md](02-개발-표준.md)의 "관련 문서" 링크를 서로 걸어 두면 찾기 쉽습니다.
