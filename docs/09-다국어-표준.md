# 다국어(국제화) 표준 (i18n / Internationalization Standards)

다국어를 지원하는 웹 애플리케이션의 **메시지·날짜·숫자·언어 전환** 표준입니다.  
Spring Boot + React, 동일 도메인 환경을 전제로 합니다. [02 개발 표준](02-개발-표준.md)의 API 응답·에러 메시지 규칙과 연동합니다.

## 1. 원칙

- **한 곳에서 관리**: UI 문구는 **메시지 키**로만 참조하고, 실제 문자열은 번역 파일에서만 관리한다. 코드에 하드코딩된 문구는 넣지 않는다.
- **일관된 키 체계**: 키 이름 규칙을 통일해 찾기·번역 누락 검사가 쉽게 한다.
- **fallback**: 키에 해당 언어 번역이 없으면 **기본 언어**로 표시한다. 빈 화면·키 그대로 노출을 피한다.

---

## 2. 지원 언어·기본값

- **지원 언어**: 팀에서 사용할 언어 코드를 정한다. ISO 639-1(2자리) 사용 권장. 예: `ko`, `en`, `ja`, `vi`. 본 프로젝트는 한국어(ko), 영어(en), 일본어(ja), 베트남어(vi)를 지원한다.
- **기본 언어(default locale)**: 서비스 기본 언어 하나를 정한다. 예: `ko`. 미지정·미지원 언어 요청 시 기본 언어로 응답한다.
- **언어 저장**: 선택한 언어는 **localStorage** 또는 **쿠키**에 저장해, 다음 방문 시 유지한다. (키 예: `app_locale` 또는 `locale`)

## 3. 프론트엔드(React) i18n

### 3.1 라이브러리

- **react-i18next** (i18next + react-i18next) 사용을 권장한다. React와 통합이 잘 되고, 네임스페이스·lazy load·날짜 포맷 연동이 편하다.
- **i18next** 설정에서 `fallbackLng`, `defaultNS`, `interpolation` 등을 한 곳에서 관리한다.

### 3.2 초기화·설정 위치

- **설정 파일**: `src/i18n.ts` 또는 `src/locales/index.ts` 에서 i18n 초기화.
- **언어 리소스**: `src/locales/{locale}/` 아래에 JSON(또는 팀 정책에 따라 다른 형식)으로 둔다. (상세는 [5. 번역 파일 구조](#5-번역-파일-구조))

### 3.3 사용 방법

- **컴포넌트 내**: `const { t } = useTranslation();` 후 `t('common.save')`, `t('order.list.title')` 형태로 사용.
- **네임스페이스**: 화면/기능 단위로 나눌 경우 `useTranslation('order')` 등으로 네임스페이스 지정. 키는 해당 NS 기준 상대 경로.
- **파라미터 치환**: `t('message.welcome', { name: user.name })` — JSON에는 `"message.welcome": "{{name}}님 환영합니다."` 형태로 정의.
- **플러럴**: `t('item.count', { count: n })` — i18next plural 규칙 사용. (필요 시)

## 4. 메시지 키 규칙

### 4.1 형식

- **계층 구분**: **점(.)**으로 구분. 예: `common.save`, `order.list.empty`, `error.not_found`.
- **소문자**: 키는 **소문자**와 숫자만 사용. 단어 구분은 언더스코어 또는 카멜 케이스 중 팀에서 하나로 통일. (예: `order_detail_title` 또는 `orderDetailTitle` — 본 문서는 점만 사용하고 하위는 소문자+언더스코어 권장: `order.detail.title`)

### 4.2 네임스페이스(키 접두어) 구분

| 접두어 | 용도 | 예시 |
|--------|------|------|
| **common** | 버튼·레이블·공통 문구 | `common.save`, `common.cancel`, `common.loading` |
| **menu** | 메뉴명 (서버에서 내려주는 메뉴는 서버 키와 매핑) | `menu.dashboard`, `menu.admin.users` |
| **error** | 에러 메시지 (API code와 매핑 가능) | `error.not_found`, `error.validation` |
| **validation** | 폼 검증 메시지 | `validation.required`, `validation.email` |
| **기능/도메인** | 해당 기능 전용 문구 | `order.list.title`, `user.form.name` |

- **API 에러 코드와 매핑**: 백엔드가 `code: "ERR_NOT_FOUND"` 를 주면, 프론트에서 `code` → 메시지 키 매핑 테이블로 `error.not_found` 를 찾아 `t('error.not_found')` 로 표시한다. (또는 code를 그대로 키로 쓰기: `t(\`error.${code}\`)` — 키 명명 규칙만 통일)

### 4.3 키 이름

- **역할이 드러나게** 짓는다. `msg1`, `title2` 같은 이름은 사용하지 않는다.
- **동일 문구 재사용**: 같은 문구는 같은 키를 쓴다. 복사해서 새 키를 만들지 않는다.

## 5. 번역 파일 구조

### 5.1 디렉터리

```
src/locales/
├── index.ts          # i18n 초기화, 리소스 로드
├── ko/
│   ├── common.json
│   ├── error.json
│   ├── validation.json
│   └── order.json     # 기능별 (선택)
└── en/
    ├── common.json
    ├── error.json
    ├── validation.json
    └── order.json
```

- **한 언어 한 폴더**: `ko`, `en` 등. 그 안에 **네임스페이스별 JSON** 또는 **단일 JSON** 중 팀 정책 선택.
- **네임스페이스 = 파일명**(확장자 제외). 예: `common.json` → NS `common`.

### 5.2 JSON 형식

- **1단계 깊이** 또는 **2단계 깊이**로 유지해 키가 너무 길어지지 않게 한다.  
  예: `{ "save": "저장", "cancel": "취소" }` (common.json) → 키 `common.save`.  
  또는 `{ "order": { "list": { "title": "주문 목록" } } }` → 키 `order.list.title`.
- **파라미터**: `{{paramName}}` 형태. i18next 기본 문법과 동일.

```json
{
  "welcome": "{{name}}님 환영합니다.",
  "item_count": "{{count}}개"
}
```

### 5.3 누락·검증

- **빌드 또는 CI**에서 기본 언어(예: ko) 대비 다른 언어 JSON에 **키 누락**이 없는지 검사하는 스크립트를 두면 좋다.
- **키는 기본 언어를 기준**으로 두고, 다른 언어 파일에 동일 키가 있는지 확인한다.

## 6. 백엔드·API 연동

### 6.1 API 응답의 message

- **방식 A (권장)**: API는 **메시지 키**만 내려준다.  
  예: `"message": "error.not_found"` 또는 `"messageKey": "error.not_found"`.  
  프론트는 `t(response.message)` 또는 `t(response.messageKey)` 로 표시.  
  → 언어 전환 시 서버 재요청 없이 클라이언트에서 즉시 반영.
- **방식 B**: API가 **Accept-Language** 헤더를 보고 이미 번역된 문구를 `message`에 넣어 준다.  
  → 서버에 번역 리소스가 있어야 하고, 클라이언트는 그대로 표시.  
  → 개발 표준의 "다국어 시 메시지 키만 전달하고 클라이언트에서 치환"은 **방식 A**에 해당.

### 6.2 Accept-Language

- 프론트에서 API 요청 시 **Accept-Language** 헤더에 현재 선택 언어를 넣어 준다.  
  예: `Accept-Language: ko-KR,ko;q=0.9,en;q=0.8`.  
  서버가 방식 B를 쓰거나, 로그·감사용으로 언어를 기록할 때 활용한다.

### 6.3 메뉴·권한

- **메뉴명**도 다국어일 수 있다.  
  - **방법 1**: 서버가 메뉴 **키**만 내려주고(예: `menuKey: "menu.dashboard"`), 프론트에서 `t(menu.menuKey)` 로 표시.  
  - **방법 2**: 서버가 현재 언어로 번역된 메뉴명을 내려준다.  
  팀에서 하나로 통일. (키 기반이면 메뉴 추가 시 번역 파일만 갱신하면 되어 관리가 쉬움.)

## 7. 날짜·숫자·통화

### 7.1 날짜

- **표시**: 브라우저 `Intl.DateTimeFormat` 또는 i18next와 연동된 date 라이브러리(예: date-fns + locale) 사용.  
  **locale**은 현재 선택 언어(`i18n.language`)와 맞춘다.
- **API 전송**: 서버와 주고받을 때는 **ISO 8601** 문자열(예: `2025-02-20T10:00:00Z`)로 통일. 표시할 때만 로케일 포맷으로 변환한다.

### 7.2 숫자·통화

- **숫자**: `Intl.NumberFormat` 에 현재 locale 전달. 천 단위 구분, 소수 자리수를 로케일에 맞춘다.
- **통화**: `Intl.NumberFormat(locale, { style: 'currency', currency: 'KRW' })` 등. 통화 코드는 서버에서 내려주거나 클라이언트 기본값을 둔다.
- **단위**: 단위 문구도 메시지 키로. 예: `t('common.currency.won')`, `t('common.unit.piece')`.

## 8. 새 언어·키 추가

### 8.1 새 지원 언어 추가

1. `src/locales/{새언어코드}/` 폴더 생성.
2. 다른 언어와 **동일한 구조**의 JSON 파일 복사 후 번역 채우기.
3. i18n 설정에 새 언어를 `supportedLngs`(또는 동등 설정)에 추가.
4. (선택) 언어 선택 UI에 새 언어 옵션 추가.
5. 키 누락 검사 스크립트에 새 언어 포함.

### 8.2 새 메시지 키 추가

1. **기본 언어** JSON에 키·문구 추가.
2. 나머지 언어 JSON에 **같은 키**로 번역 추가. (누락 시 fallback으로 기본 언어 표시)
3. 코드에서 `t('namespace.key')` 로 사용. 기존 문구 하드코딩이 있다면 키로 교체.

## 9. 정리 체크리스트

| 구분 | 표준 요약 |
|------|-----------|
| **원칙** | 문구는 키로만 참조, 번역은 파일에서만. fallback은 기본 언어. |
| **언어** | ISO 639-1(ko, en 등), 기본 언어 명시, 선택 언어는 localStorage/쿠키 저장 |
| **라이브러리** | react-i18next 권장, 설정은 src/i18n.ts 또는 src/locales/index.ts |
| **키 규칙** | 점(.) 구분, common/menu/error/validation/도메인 접두어, 소문자·의미 있게 |
| **파일 구조** | src/locales/{locale}/, 네임스페이스별 JSON(common.json, error.json 등) |
| **API** | message는 메시지 키(방식 A) 또는 Accept-Language로 서버 번역(방식 B). 메뉴는 키 기반 권장 |
| **날짜/숫자/통화** | Intl + 현재 locale, API는 ISO 8601 |
| **추가** | 새 언어: 동일 구조 JSON + 설정 추가. 새 키: 기본언어 먼저, 나머지 언어 동일 키 |
