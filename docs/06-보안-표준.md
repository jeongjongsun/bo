## 1. 보안 원칙

- **최소 권한**: 사용자·프로세스는 필요한 최소 권한만 부여한다.
- **검증은 서버에서**: 클라이언트 검증은 UX용이며, **접근 제어·입력 검증은 반드시 백엔드에서 수행**한다.
- **민감 정보 노출 금지**: 비밀번호·토큰·개인정보는 로그·에러 메시지·URL·클라이언트 저장소에 남기지 않는다. (로깅은 [05 로깅 표준](05-로깅-표준.md) 참고)
- **기본 거부**: 새 API·메뉴는 기본 비공개로 두고, 명시적으로 권한이 있는 대상만 허용한다.

## 2. 인증(Authentication)

### 2.1 방식

- **세션 기반 인증**을 사용한다. 동일 도메인에서 쿠키(JSESSIONID)로 세션을 유지한다.
- 로그인 성공 시 서버에서 세션을 생성하고, 응답에 **Set-Cookie**로 세션 쿠키를 내려준다. 이후 API 요청 시 브라우저가 자동으로 쿠키를 전송한다.

### 2.2 로그인·로그아웃

| 항목 | 규칙 |
|------|------|
| 로그인 실패 | 구체적인 이유(“비밀번호 오류” vs “존재하지 않는 ID”)를 구분해 노출할지 팀 정책으로 결정. 남용 방지를 위해 **동일 IP/계정에 대한 잠금·지연** 검토 |
| 로그인 성공 | 세션 생성, 필요한 최소 정보만 세션에 저장(userId, 역할 등). 비밀번호는 세션에 넣지 않음 |
| 로그아웃 | 서버에서 해당 세션 무효화. 클라이언트는 로그아웃 API 호출 후 로그인 페이지로 이동 |
| 비밀번호 | 저장 시 **bcrypt 등 적절한 해시** 사용. 평문·단순 해시(SHA-256 단독 등) 저장 금지 |
| 비밀번호 변경(프로필) | **복잡도 검증** 백엔드 필수: 최소 길이(예: 8자), 영문 1자 이상(대소문자 구분 없음)·숫자·특수문자 각 1자 이상. 구현: `PasswordPolicy` 등 유틸에서 검증 후 저장 |

### 2.3 인증 필요 API

- 인증이 필요한 API는 **모든 요청**에 대해 세션 유효 여부를 검사한다. 세션이 없거나 만료된 경우 **401 Unauthorized** + 표준 에러 코드 `ERR_UNAUTHORIZED` 또는 `ERR_SESSION_EXPIRED`를 반환한다. (02 개발 표준의 [권한 만료/세션 만료](02-개발-표준.md#4-권한-만료세션-만료-시-규칙) 참고)

### 2.4 프론트엔드

- API 호출 시 **credentials: 'include'** 를 사용해 세션 쿠키가 전송되도록 한다. (동일 도메인이면 기본이지만, fetch/axios 설정에서 명시 권장)
- 인증 여부는 **GET /api/v1/auth/me** 성공 여부로 판단한다. 실패(401) 시 로그인 페이지로 리다이렉트한다.

## 3. 인가(Authorization)

### 3.1 원칙

- **메뉴·기능 노출**은 서버에서 내려준 `menus`, `permissions`(auth/me)에만 의존한다. 클라이언트에서만 권한을 판단하지 않는다.
- **API 접근**은 요청 시마다 서버에서 **역할/권한**을 검사한다. “해당 path는 로그인만 하면 된다”가 아니라, **리소스 단위**(예: 특정 주문 조회는 본인 또는 관리자만)까지 검사할 수 있도록 설계한다.

### 3.2 구현

- URL/메서드별로 필요한 권한(역할 또는 권한 코드)을 정의하고, 필터/인터셉터 또는 메서드 시큐리티에서 검사한다.
- 권한 없음 시 **403 Forbidden** + `ERR_FORBIDDEN`, 사용자용 메시지(예: “권한이 없습니다”)를 반환한다.
- **다른 사용자 리소스 접근**: 예) “내 주문만 조회” — 세션의 userId와 요청한 리소스의 소유자(또는 조회 허용 범위)를 비교해, 불일치 시 403 처리한다.

### 3.3 클라이언트

- 버튼/메뉴 노출은 `permissions` 또는 `menus` 기준으로만 제어한다. 이는 **UX용**이며, 실제 보안은 서버 검사로 보장한다.

## 4. 전송 구간 보안

### 4.1 HTTPS

- **운영 환경**에서는 반드시 **HTTPS**를 사용한다. HTTP는 리다이렉트용 또는 개발용으로만 한정한다.
- 세션 쿠키는 **Secure** 속성을 설정해 HTTPS에서만 전송되도록 한다. (동일 도메인이라도 운영에서는 HTTPS 가정)

### 4.2 API 통신

- 프론트와 백엔드가 **동일 도메인**이면 CORS 이슈는 적다. 다른 도메인으로 API를 분리할 경우, 허용 Origin을 화이트리스트로 제한하고, credentials 사용 시 `Access-Control-Allow-Origin`을 `*`가 아닌 구체 도메인으로 설정한다.

## 5. 입력 검증 및 인젝션 방지

### 5.1 원칙

- **모든 외부 입력**(쿼리 파라미터, path variable, body, 헤더)은 신뢰하지 않고, 서버에서 타입·길이·형식·범위를 검증한다.
- 검증 실패 시 **400 Bad Request** 또는 **422 Unprocessable Entity** + 표준 에러 포맷(`ERR_VALIDATION`, `errors` 배열에 필드별 메시지)을 반환한다.

### 5.2 SQL 인젝션

- **PreparedStatement / 파라미터 바인딩**만 사용한다. 사용자 입력을 SQL 문자열에 이어 붙이지 않는다.
- JPA/MyBatis 등 사용 시에도 **네이티브 쿼리에 문자열 연결 금지**. 동적 정렬/필터는 화이트리스트(허용된 컬럼명·값)로만 구성한다.

### 5.3 파라미터 검증

- **길이 제한**: 문자열 길이 상한을 두고 초과 시 400/422.
- **타입·형식**: 숫자, 날짜, 이메일, 코드값 등은 서버에서 파싱·검증. 정규식이나 Bean Validation 활용.
- **페이징**: `page`, `size`는 개발 표준대로(예: size 최대 100). 초과 시 400 또는 상한으로 자동 보정 후 처리(팀 정책 통일).
- **정렬(sort)**: 허용 컬럼만 허용. “컬럼명,DESC” 형태를 파싱한 뒤 컬럼명을 화이트리스트와 비교한다.

### 5.4 경로·리소스 식별자

- path variable 또는 쿼리의 ID(숫자/UUID)는 **타입 검증** 후 사용. 해당 리소스 존재 여부·접근 권한을 반드시 검사한다.

## 6. XSS·CSRF 방지

### 6.1 XSS(Cross-Site Scripting)

- **서버**: API 응답에 HTML을 넣어 내려주지 않는다. 데이터는 JSON 등 구조화된 형태로만 제공한다.
- **프론트(React)**: 사용자 입력·API 데이터를 **기본적으로 텍스트로 렌더링**한다. React는 기본 이스케이프를 하므로 `dangerouslySetInnerHTML`는 **신뢰할 수 있는 HTML이 확실할 때만** 제한적으로 사용하고, 사용자 입력을 그대로 넣지 않는다.
- **파일 업로드**: 업로드 파일의 Content-Type·확장자·실제 내용을 검증한다. 실행 가능한 스크립트가 서버에 저장·서빙되지 않도록 한다.

### 6.2 CSRF(Cross-Site Request Forgery)

- **동일 도메인 + 세션 쿠키** 구성에서는, 다른 사이트에서 우리 도메인으로 요청을 보내도 브라우저가 자동으로 쿠키를 붙이므로 CSRF 가능성이 있다.
- **권장**: 서버에서 **CSRF 토큰**을 사용한다. (Spring Security의 CSRF 토큰을 폼/메타에 넣고, 상태 변경 요청(POST/PUT/DELETE 등) 시 헤더 또는 body에 토큰을 포함해 검증)
- **SameSite 쿠키**: 세션 쿠키에 **SameSite=Strict** 또는 **Lax**를 설정하면 외부 사이트에서 오는 요청에 쿠키가 붙지 않아 CSRF 위험이 줄어든다. (동일 도메인 내 이동·API 호출에는 영향 없음)

## 7. 민감 정보 및 비밀 관리

### 7.1 로그·에러 메시지

- **로그**: 비밀번호, 세션 ID 전체값, 토큰, 주민번호, 전화번호 전체 등은 로그에 남기지 않는다. (상세는 [05 로깅 표준 6. 보안 및 개인정보](05-로깅-표준.md#6-보안-및-개인정보))
- **에러 응답**: 스택 트레이스·내부 경로·DB 에러 문구를 그대로 클라이언트에 내려주지 않는다. 사용자용 `message`와 표준 `code`만 반환한다. 일괄 처리 결과의 `errors` 배열 등에도 **시스템/쿼리 예외 메시지를 넣지 않고**, 다국어 메시지 키(예: `orders.bulkImport.orderRegisterFailed`)만 담는다. 상세 원인은 서버 로그에만 남긴다.

### 7.2 비밀(DB 비밀번호, API 키 등)

- **소스 코드·설정 파일에 평문 저장 금지**. 환경 변수 또는 서버/키 관리 서비스에서 주입한다.
- Spring Boot: `application.yml`에 비밀을 직접 넣지 않고, `application-{profile}.yml`도 버전 관리에서 제외하거나, `spring.config.import` 등으로 외부 설정만 사용한다.
- 프론트: **비밀·API 키**는 브라우저에 노출되므로, 프론트 전용 키가 있다면 읽기 전용·최소 권한으로 두고, 백엔드 비밀은 절대 프론트에 두지 않는다.

### 7.3 개인정보

- 저장·전송 시 필요 최소한만 다룬다. 조회 시에도 권한이 있는 사용자에게만 노출한다.
- 화면·API에서 전화번호·이메일 등은 **마스킹** 정책을 정해 일관 적용할 수 있다. (로깅 표준의 마스킹 규칙과 통일)

## 8. 세션·쿠키 보안

| 항목 | 규칙 |
|------|------|
| **세션 타임아웃** | 비활성 시간(예: 30분) 또는 절대 시간(예: 8시간)으로 만료. 운영 환경 값은 보안·UX 균형으로 정한다. |
| **세션 고정 방지** | 로그인 성공 시 **세션 ID 재생성**(invalidate + 새로 생성). 로그인 전과 후에 동일 세션 ID가 유지되지 않도록 한다. |
| **쿠키 속성** | **HttpOnly**(스크립트 접근 불가), **Secure**(HTTPS만), **SameSite=Strict** 또는 **Lax**. Path는 필요한 경로로 제한. |
| **동시 세션** | 정책에 따라 “한 계정당 하나의 세션” 또는 “N개까지 허용” 등을 정하고, 초과 시 기존 세션 무효화 등으로 구현한다. |

## 9. HTTP 보안 헤더

서버에서 아래 헤더를 적용해 클릭재킹·스니핑·MIME 스니핑 등을 완화한다.

| 헤더 | 권장 값 | 목적 |
|------|---------|------|
| **X-Content-Type-Options** | `nosniff` | MIME 타입 스니핑 방지 |
| **X-Frame-Options** | `DENY` 또는 `SAMEORIGIN` | 클릭재킹 방지 (iframe 제한) |
| **Content-Security-Policy** | 정책 문자열 (팀 정책에 따라) | XSS·불필요한 리소스 로드 제한 |
| **Strict-Transport-Security** | `max-age=31536000; includeSubDomains` (HTTPS 서버만) | HTTPS 강제 |

- Spring Security 사용 시 `headers()` 설정으로 위 헤더를 켜 두는 것을 권장한다.

## 10. 의존성·설정 보안

### 10.1 의존성

- **라이브러리 버전**: 알려진 취약점이 있는 버전은 사용하지 않는다. 빌드 시 **취약점 스캔**(예: OWASP Dependency-Check, npm audit, Gradle dependency check)을 주기적으로 실행하고, 심각도 높은 항목은 업그레이드 또는 대체한다.
- **최소 필요 의존성**: 사용하지 않는 라이브러리는 제거해 공격 면을 줄인다.

### 10.2 설정

- **운영 프로파일**: 디버그 모드·상세 에러 페이지·관리 포트 노출 등은 운영에서 비활성화한다.
- **DB 계정**: 애플리케이션이 사용하는 DB 계정은 **필요한 권한만** 부여한다. DDL/전역 권한은 피한다.
- **기본 비밀번호**: 관리자·시스템 계정의 기본 비밀번호는 최초 로그인 시 변경을 강제하거나, 설치 시 반드시 변경하도록 한다.

## 11. 정리 체크리스트

| 구분 | 표준 요약 |
|------|-----------|
| **인증** | 세션 기반, 로그인 시 세션 생성·비밀번호 해시(bcrypt), 인증 필요 API는 401 반환, 프론트 credentials include |
| **인가** | 메뉴/권한은 auth/me 기준, API는 서버에서 역할·리소스 소유자 검사, 403 + ERR_FORBIDDEN |
| **전송** | 운영 환경 HTTPS, 세션 쿠키 Secure |
| **입력** | 서버에서 타입·길이·형식·범위 검증, SQL은 파라미터 바인딩만, sort 등 화이트리스트 |
| **XSS** | API는 JSON만, React는 dangerouslySetInnerHTML 금지(필요 시 최소한), 파일 업로드 검증 |
| **CSRF** | SameSite 쿠키 + 필요 시 CSRF 토큰 |
| **민감정보** | 로그/에러에 비밀·토큰·개인정보 미포함, 비밀은 환경변수·키 관리 |
| **세션/쿠키** | HttpOnly, Secure, SameSite, 로그인 시 세션 ID 재생성, 타임아웃 정책 |
| **헤더** | X-Content-Type-Options, X-Frame-Options, (선택) CSP, HSTS |
