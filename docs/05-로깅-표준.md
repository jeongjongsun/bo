# 로깅 표준 (Logging Standards)

Spring Boot + React 통합 시스템(Shopeasy4)의 **백엔드·프론트엔드 로깅** 표준입니다.  
장애 추적, 운영 점검, 보안을 위해 **무엇을, 어디서, 어떤 형식으로** 남길지 정의합니다.

## 1. 로깅 원칙

- **목적**: 장애 원인 파악, 감사(audit), 동작 확인. 불필요한 로그는 노이즈만 늘리므로 **의미 있는 경우에만** 기록합니다.
- **일관성**: 레벨·메시지·키 이름을 팀에서 통일해 검색·분석이 쉽게 합니다.
- **보안**: 비밀번호, 토큰, 개인정보(전화번호, 주민번호 등)는 **로그에 남기지 않습니다**. (상세는 [6. 보안 및 개인정보](#6-보안-및-개인정보))

## 2. 로그 레벨

공통으로 사용하는 레벨와 용도는 아래와 같습니다.

| 레벨 | 용도 | 예시 |
|------|------|------|
| **ERROR** | 처리 실패, 예외, 재현이 필요한 오류 | API 500, DB 연결 실패, 예외 stack trace |
| **WARN** | 비정상이지만 동작은 계속되는 상황 | 권한 없음 접근 시도, 세션 만료, 재시도 성공 |
| **INFO** | 중요한 비즈니스/시스템 이벤트 | 로그인/로그아웃, 중요 API 호출 시작·완료, 배치 시작·종료 |
| **DEBUG** | 개발·장애 추적용 상세 정보 | SQL 파라미터, 내부 상태값, 단계별 진행 (운영에서는 필요 시에만 활성화) |
| **TRACE** | 매우 상세한 추적 (기본 비활성) | 라이브러리 내부, 네트워크 바이트 단위 등 |

**운영 환경 권장**: 기본은 **INFO**, 장애 분석 시 특정 패키지만 **DEBUG**로 상향.

## 3. 백엔드(Spring Boot) 로깅

### 3.1 사용 프레임워크

- **SLF4J + Logback** (Spring Boot 기본)
- 코드에서는 **SLF4J**만 사용해 구현체를 바꿀 수 있게 합니다.

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class UserService {
    private static final Logger log = LoggerFactory.getLogger(UserService.class);
    // ...
}
```

- **Lombok** 사용 시: `@Slombok.extern.slf4j.Slf4j` 적용 시 `log` 자동 생성 가능.

### 3.2 언제 무엇을 로깅할지

| 상황 | 레벨 | 로깅 내용 |
|------|------|-----------|
| API 진입(중요 API) | INFO | 메서드/경로, 사용자 식별자(세션 ID 또는 userId), 요청 키 정보(예: orderId) |
| API 정상 완료 | INFO 또는 생략 | 필요 시 처리 건수, 소요 시간 |
| 비즈니스 예외(4xx) | WARN | 에러 코드, 메시지, 사용자 식별자, 요청 경로. **상세 파라미터는 제외** |
| 시스템 예외(5xx) | ERROR | 예외 메시지, **stack trace**, 요청 경로, requestId(있을 경우) |
| 로그인 성공/실패 | INFO / WARN | 성공: userId(또는 로그인 ID), IP 등. 실패: 로그인 ID만(비밀번호 절대 금지) |
| 로그아웃 | INFO | userId 또는 세션 식별자 |
| 권한 없음 접근 | WARN | 사용자 식별자, 요청 경로, 필요한 리소스 |
| 외부 연동 호출/응답 | INFO 또는 DEBUG | URL(쿼리 제외), 상태코드, 소요 시간. 실패 시 ERROR + 원인 |

### 3.3 메시지 형식 (일관성)

- **한글 또는 영문** 팀 규칙에 맞춰 통일. 문장형보다는 **"역할: 내용"** 형태가 검색에 유리합니다.
- 예: `"로그인 성공, userId={}, ip={}"`, `"주문 조회 실패, orderId={}, code=ERR_NOT_FOUND"`

```java
// 좋은 예: 구조화된 메시지 + placeholder
log.info("주문 조회 완료, orderId={}, userId={}", orderId, userId);
log.warn("권한 없음, path={}, userId={}", request.getRequestURI(), userId);
log.error("주문 저장 실패, orderId={}, ex={}", orderId, e.getMessage(), e);
```

### 3.4 예외 로깅

- **ERROR** 시 **원인 예외(cause)** 와 **stack trace**를 남깁니다. `log.error("메시지", exception)` 형태로 예외 객체를 마지막 인자로 넘깁니다.
- 동일 예외를 상위에서 다시 로깅하지 않습니다 (한 번만 ERROR 로그).

```java
try {
    // ...
} catch (BusinessException e) {
    log.warn("비즈니스 오류, code={}, message={}", e.getCode(), e.getMessage());
    throw e;
} catch (Exception e) {
    log.error("주문 처리 중 오류, orderId={}", orderId, e);  // e 로 stack trace 포함
    throw new InternalServerException(e);
}
```

### 3.5 설정 권장 (application.yml / logback-spring.xml)

- **포맷**: 로그 발생 시각, 레벨, 로거명, 스레드, 메시지. 운영에서는 **한 줄 per 로그** 유지.
- **파일**: 애플리케이션 로그는 파일로 남기고, 일별 롤링·보관 기간 정책 수립.
- **패키지별 레벨**: 기본 INFO, `com.shopeasy` 등 프로젝트 패키지는 DEBUG 가능하도록 구성하고, 운영에서는 INFO로 두고 필요 시만 DEBUG로 변경.

```yaml
# application.yml 예시
logging:
  level:
    root: INFO
    com.shopeasy: INFO
    org.springframework.web: WARN
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n"
```

## 4. 프론트엔드(React) 로깅

### 4.1 기본 원칙

- 브라우저 로그는 **개발·디버깅** 용도가 크고, 운영에서는 사용자가 콘솔을 열지 않으므로 **과도한 로그는 지양**합니다.
- **운영 빌드**에서는 `console.log`를 제거하거나 레벨에 따라 무시하는 방식을 권장합니다.

### 4.2 사용할 출력

| 목적 | 사용 | 비고 |
|------|------|------|
| 개발 중 확인 | `console.log` | 운영 빌드에서 제거 또는 래퍼로 비활성화 |
| 경고(잘못된 사용 등) | `console.warn` | 유지해도 됨, 개인정보 포함 금지 |
| 에러(API 실패, 예외) | `console.error` 또는 로거 | 사용자 식별자만 필요 시, 메시지는 서버 메시지 활용 |

### 4.3 로거 래퍼 권장 (선택)

- **개발**에서는 `console.*` 출력, **운영**에서는 `error`/`warn`만 남기거나 외부 로깅 서비스로 전송하도록 한 곳에서 제어합니다.
- 프로젝트에 `utils/logger.ts`(또는 `lib/logger.ts`)를 두고, 여기서만 로그를 남기면 나중에 정책 변경이 쉽습니다.

```ts
// utils/logger.ts 예시
const isDev = import.meta.env.DEV; // 또는 process.env.NODE_ENV === 'development'

export const logger = {
  debug: (...args: unknown[]) => isDev && console.log('[DEBUG]', ...args),
  info: (...args: unknown[]) => isDev && console.info('[INFO]', ...args),
  warn: (...args: unknown[]) => console.warn('[WARN]', ...args),
  error: (...args: unknown[]) => console.error('[ERROR]', ...args),
};
```

- 실제 로그는 `logger.info('API 호출', { path, status });` 형태로 **객체**를 넘기면 나중에 필터·수집 시 유리합니다.

### 4.4 언제 무엇을 로깅할지

| 상황 | 레벨 | 로깅 내용 |
|------|------|-----------|
| API 요청(개발용) | debug | method, url, params (개인정보 제외) |
| API 에러(4xx/5xx) | error | url, status, code, message (서버 표준 에러 필드) |
| 세션 만료(401) 리다이렉트 | warn | "세션 만료로 로그인 페이지 이동" 등 |
| 권한 없음(403) | warn | path 또는 기능명 |
| 라우트/페이지 전환(개발용) | debug | pathname |

- **절대 로그에 넣지 말 것**: 비밀번호, 세션 ID/토큰 값, 전화번호·이메일 등 개인정보. (상세는 [6. 보안 및 개인정보](#6-보안-및-개인정보))

### 4.5 React Query·API 레이어와 연동

- **전역**에서 처리하는 것을 권장합니다. axios 인터셉터 또는 React Query의 `defaultOptions.queries.meta` 등에서 **에러 시에만** 로깅합니다.
- API 에러는 `error?.response?.data`의 `code`, `message`만 로깅하고, 요청 body 전체는 로그에 넣지 않습니다.

```ts
// 예: axios 인터셉터에서
axiosInstance.interceptors.response.use(
  (res) => res,
  (err) => {
    const status = err.response?.status;
    const data = err.response?.data;
    logger.error('API 오류', {
      url: err.config?.url,
      method: err.config?.method,
      status,
      code: data?.code,
      message: data?.message,
    });
    return Promise.reject(err);
  }
);
```

## 5. 로그 포맷 및 출력 규칙

### 5.1 공통 규칙

- **한 줄에 한 로그**: 여러 줄로 나뉘면 수집·검색 시 불리하므로, 여러 값은 JSON 또는 `key=value` 형태로 한 줄에 넣습니다.
- **일시**: 백엔드는 로그 프레임워크가 자동 추가. 프론트는 필요 시 `new Date().toISOString()` 등으로 추가 가능.
- **키 이름 통일**: 팀에서 정한 키를 사용합니다. 예: `userId`, `orderId`, `path`, `status`, `code`, `message`, `requestId`.

### 5.2 백엔드 로그 한 줄 예시

```
2025-02-20 10:00:01.123 [http-nio-18081-exec-1] INFO  c.s.order.OrderController - 주문 조회 완료, orderId=ORD001, userId=user01
2025-02-20 10:00:02.456 [http-nio-18081-exec-2] WARN  c.s.auth.AuthFilter - 권한 없음, path=/api/v1/admin/users, userId=user02
2025-02-20 10:00:03.789 [http-nio-18081-exec-3] ERROR c.s.order.OrderService - 주문 저장 실패, orderId=ORD002, ex=Duplicate key
```

### 5.3 프론트엔드 로그 한 줄 예시 (객체)

```ts
logger.error('API 오류', { url: '/api/v1/orders', status: 500, code: 'ERR_INTERNAL', message: '일시적인 오류가 발생했습니다.' });
```

- 운영에서 외부 로깅 서비스(예: Sentry)를 쓸 경우, 같은 객체 구조로 전송하면 백엔드 로그와 연계 분석하기 쉽습니다.

## 6. 보안 및 개인정보

### 6.1 로그에 남기지 말아야 할 것

| 항목 | 설명 |
|------|------|
| 비밀번호 | 평문, 해시값 모두 로그에 넣지 않음 |
| 세션 ID / 토큰 전체 값 | 식별용으로 짧은 접미사만 허용할지 팀 정책으로 결정 (권장: 남기지 않음) |
| 주민번호, 전화번호 전체 | 마스킹 없이 로그 금지 |
| 이메일/이름 | 개인정보 정책에 따라 마스킹 또는 제외. 필요 시 `user***@***.com` 형태만 |
| API 요청 body 전체 | 특히 로그인·회원가입 등 개인정보가 들어가는 요청은 body 로깅 금지 |
| SQL 파라미터(개인정보 컬럼) | DEBUG에서도 개인정보 컬럼 값은 마스킹 또는 제외 |

### 6.2 허용 예시

- **사용자 식별**: `userId`, `loginId`(로그인 ID) 정도만. “로그인 성공, userId=user01” 등.
- **요청 추적**: path, method, status, requestId, 처리 시간.
- **에러**: 서버가 내려준 `code`, `message` (이미 사용자용 메시지이므로 과도한 상세는 제외).

### 6.3 마스킹 규칙 (필요 시)

- 전화번호: `010-****-5678`
- 이메일: `ho***@example.com`
- 주민번호: 로그에 포함하지 않는 것을 원칙으로 하고, 불가피한 경우 앞 6자리만 또는 전부 마스킹

## 7. 요청 추적(Request ID)

- **목적**: 하나의 HTTP 요청이 백엔드 여러 로그에 흩어져 있어도, **동일한 ID**로 묶어서 추적할 수 있게 합니다.
- **구현 요약**:
  - 백엔드: 필터/인터셉터에서 요청 진입 시 `requestId`(UUID 등) 생성 후 MDC에 넣고, Logback pattern에 `%X{requestId}` 포함.
  - API 응답 헤더에 `X-Request-Id: <uuid>` 로 내려주면, 프론트에서 에러 로그에 같은 값을 넣어 백엔드와 연결 가능.
- **선택 사항**: 도입 시 팀에서 “모든 API 응답에 X-Request-Id 포함” 규칙만 정하면 됩니다.

```yaml
# Logback pattern 예시 (requestId 포함)
logging:
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss.SSS} [%X{requestId}] [%thread] %-5level %logger{36} - %msg%n"
```

## 8. 정리 체크리스트

| 구분 | 표준 요약 |
|------|-----------|
| **레벨** | ERROR(실패/예외), WARN(비정상·권한 등), INFO(중요 이벤트), DEBUG(개발·추적) |
| **백엔드** | SLF4J/Logback, 예외는 ERROR+stack trace, 비즈니스 예외는 WARN, 메시지·키 일관 |
| **프론트** | logger 래퍼 권장, 운영에서 debug/log 최소화, API 에러는 error 레벨로 한 곳에서 |
| **포맷** | 한 줄 per 로그, 키 이름 통일(userId, path, code, message, requestId 등) |
| **보안** | 비밀번호·토큰·개인정보 로그 금지, body/개인정보 컬럼 로깅 금지 또는 마스킹 |
| **추적** | 선택: Request ID(MDC + X-Request-Id 헤더)로 요청 단위 추적 |
