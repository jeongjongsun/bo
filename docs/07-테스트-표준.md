## 1. 테스트 원칙

- **목적**: 수정 시 기존 동작이 깨지지 않았는지 확인(회귀 방지), 요구사항대로 동작하는지 검증. “테스트가 있다”는 것만으로는 부족하고, **의미 있는 경우를 검증**하는 것이 중요합니다.
- **우선순위**: 비즈니스 로직·API 계약·공통 유틸·권한 등 **장애 시 영향이 큰 부분**부터 테스트합니다. 화면 꾸미기만 하는 컴포넌트는 나중에 추가해도 됩니다.
- **일관성**: 테스트도 “어디에, 어떤 이름으로”를 팀에서 통일하면 찾기 쉽고, CI에서 같은 방식으로 돌릴 수 있습니다.
- **독립성**: 테스트 간 의존을 두지 않습니다. 실행 순서가 바뀌어도, 하나만 실행해도 결과가 같아야 합니다.

## 2. 테스트 종류와 범위

| 종류 | 설명 | 백엔드 | 프론트 |
|------|------|--------|--------|
| **단위(Unit)** | 한 클래스/함수/훅을 격리해 검증. 외부는 모킹 | 서비스·유틸 로직 | utils, hooks, 순수 함수 |
| **통합(Integration)** | 여러 계층/API/DB가 맞물려 동작하는지 검증 | API 엔드포인트, DB 연동 | API 함수 + 모킹, 훅+컴포넌트 |
| **컴포넌트** | UI 컴포넌트가 기대대로 렌더·동작하는지 | - | React 컴포넌트 |
| **E2E** | 브라우저에서 전체 흐름 검증 | (선택) | (선택) |

**권장 우선순위**

1. **백엔드**: API 응답 포맷·에러 코드·권한(401/403)·비즈니스 로직.
2. **프론트**: API 호출 함수·공통 훅·유틸·중요 화면(폼 제출, 목록 표시).

## 3. 백엔드(Spring Boot) 테스트

### 3.1 사용 도구

- **JUnit 5** (테스트 실행·어노테이션)
- **Mockito** (모킹)
- **Spring Boot Test** (통합 테스트 시 컨텍스트, MockMvc, TestRestTemplate)
- **AssertJ** (가독성 좋은 단언문, 선택)

### 3.2 단위 테스트

- **대상**: 서비스 클래스의 비즈니스 로직, 유틸/헬퍼. **외부 의존(Repository, 외부 API)**은 모킹합니다.
- **목적**: 입력에 대한 출력·예외가 요구사항대로인지 검증.

**예시 (서비스)**

```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock
    private OrderRepository orderRepository;

    @InjectMocks
    private OrderService orderService;

    @Test
    @DisplayName("주문 조회 시 존재하지 않으면 예외")
    void getOrder_notFound_throws() {
        when(orderRepository.findById(any())).thenReturn(Optional.empty());
        assertThatThrownBy(() -> orderService.getOrder("INVALID"))
            .isInstanceOf(NotFoundException.class)
            .hasFieldOrPropertyWithValue("code", "ERR_NOT_FOUND");
    }
}
```

- **검증 시**: 개발 표준의 **에러 코드**(`ERR_NOT_FOUND`, `ERR_FORBIDDEN` 등)를 기대값으로 사용하면, API 계약과 맞는지 확인할 수 있습니다.

### 3.3 API(컨트롤러) 통합 테스트

- **대상**: HTTP 요청 → 응답 상태·본문 구조·에러 코드. DB는 인메모리(H2) 또는 Testcontainers(PostgreSQL)로 팀 정책 통일.
- **방식**: `@SpringBootTest` + `TestRestTemplate` 또는 `@WebMvcTest` + `MockMvc`. 인증이 필요한 API는 테스트용 유저/세션을 주입해 401/403 케이스를 함께 검증합니다.

**검증 항목**

- 성공 시: **HTTP 200**, 응답 body에 `success: true`, `data` 존재.
- 실패 시: **4xx/5xx**, 표준 에러 포맷(`success: false`, `code`, `message`) 및 `ERR_*` 코드.
- 목록 API: `data.items`, `data.page`, `data.size`, `data.total` 등 개발 표준의 [API 응답 포맷](02-개발-표준.md#1-api-응답-포맷-표준) 준수 여부.

**예시 (MockMvc)**

```java
@SpringBootTest
@AutoConfigureMockMvc
class OrderApiTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    @DisplayName("미인증 시 401")
    void getOrder_unauthorized_returns401() throws Exception {
        mockMvc.perform(get("/api/v1/orders/ORD001"))
            .andExpect(status().isUnauthorized())
            .andExpect(jsonPath("$.success").value(false))
            .andExpect(jsonPath("$.code").value("ERR_UNAUTHORIZED"));
    }
}
```

### 3.4 DB 연동 테스트

- **필요 시**: Repository 또는 “DB까지 포함한 서비스 플로우”를 검증할 때 사용.
- **방식**: `@DataJpaTest`(JPA만) 또는 `@SpringBootTest` + 내장 H2/Testcontainers. **트랜잭션 롤백**으로 데이터가 쌓이지 않게 합니다.
- **데이터**: 테스트 전용 fixture를 넣고, 테스트에서 기대하는 레코드만 조회·검증합니다.

### 3.5 권장하지 않는 것

- **랜덤/시간 의존**: 같은 입력이면 같은 결과가 나오도록 고정 시드·고정 시간 또는 모킹 사용.
- **실제 외부 API 호출**: 외부 연동은 모킹. 필요 시 WireMock 등으로 HTTP 스텁.

## 4. 프론트엔드(React) 테스트

### 4.1 사용 도구

- **Vitest** 또는 **Jest** (테스트 러너·모킹)
- **React Testing Library (RTL)** (컴포넌트·훅 테스트, 사용자 행동 중심)
- **MSW(Mock Service Worker)** (선택): API를 네트워크 레벨에서 스텁할 때

### 4.2 단위 테스트

- **대상**: `utils/` 함수, **순수 로직**, **커스텀 훅**(데이터 변환·상태 로직).
- **원칙**: DOM/브라우저가 필요 없는 것은 **테스트 환경만으로** 실행. 훅은 `@testing-library/react`의 `renderHook`으로 검증합니다.

**예시 (유틸)**

```ts
import { describe, it, expect } from 'vitest';
import { formatDate } from './dateUtils';

describe('formatDate', () => {
  it('YYYY-MM-DD 문자열을 사용자 포맷으로 변환한다', () => {
    expect(formatDate('2025-02-20')).toBe('2025년 2월 20일');
  });
});
```

**예시 (훅)**

```ts
import { renderHook, waitFor } from '@testing-library/react';
import { describe, it, expect, vi } from 'vitest';
import { useUserList } from './hooks';

describe('useUserList', () => {
  it('성공 시 data.items를 반환한다', async () => {
    vi.mocked(fetch).mockResolvedValueOnce({
      ok: true,
      json: () =>
        Promise.resolve({
          success: true,
          data: { items: [{ id: '1', name: '홍길동' }], page: 0, size: 20, total: 1, totalPages: 1, first: true, last: true },
        }),
    } as Response);
    const { result } = renderHook(() => useUserList({ page: 0, size: 20 }), {
      wrapper: QueryClientWrapper, // QueryClientProvider 등
    });
    await waitFor(() => expect(result.current.isSuccess).toBe(true));
    expect(result.current.data?.items).toHaveLength(1);
    expect(result.current.data?.items[0].name).toBe('홍길동');
  });
});
```

- API 호출은 **모킹**합니다. 응답 형식은 개발 표준의 `ApiResponse`, `PagedData`와 동일하게 두면, 계약이 바뀌었을 때 테스트가 먼저 깨져서 발견하기 쉽습니다.

### 4.3 컴포넌트 테스트

- **대상**: 공통 컴포넌트(`components/`), 또는 중요 폼·목록이 있는 `features/xxx/` 컴포넌트.
- **원칙**: “사용자가 보는 동작”을 검증합니다. 내부 state 이름보다는 **라벨·버튼 텍스트·역할(role)**로 요소를 찾습니다. (RTL의 `getByRole`, `getByLabelText` 등)
- **API/훅**: 컴포넌트가 의존하는 훅·API는 모킹해 **UI 동작만** 검증합니다.

**예시**

```tsx
import { render, screen, userEvent } from '@testing-library/react';
import { describe, it, expect, vi } from 'vitest';
import { UserList } from './UserList';

describe('UserList', () => {
  it('로딩 중에는 로딩 문구를 표시한다', () => {
    render(<UserList />, {
      wrapper: ({ children }) => (
        <QueryClientWrapper overrides={{ useUserList: () => ({ data: undefined, isLoading: true, error: null }) }}>
          {children}
        </QueryClientWrapper>
      ),
    });
    expect(screen.getByText(/로딩/)).toBeInTheDocument();
  });
});
```

- **과도한 스냅샷 지양**: 스냅샷은 레이아웃이 자주 바뀌면 오히려 유지보수 부담이 됩니다. “이 요소가 있고, 클릭 시 이렇게 된다”는 식의 **행동·데이터 검증**을 우선합니다.

### 4.4 API 레이어 테스트

- **대상**: `api/` 또는 `features/xxx/` 내 API 호출 함수. URL·메서드·파라미터·응답 파싱이 표준대로인지 검증합니다.
- **방식**: `fetch`/axios를 모킹해, 호출 인자와 반환값만 검증해도 됩니다. (실제 서버 불필요)

### 4.5 E2E 테스트

- **선택** 사항입니다. Cypress, Playwright 등으로 “로그인 → 목록 조회 → 상세 이동” 같은 **핵심 플로우**만 두어도 효과가 있습니다.
- 운영과 동일한 도메인·세션 구조를 쓰면, E2E에서도 동일한 인증 방식을 적용할 수 있습니다.

## 5. 테스트 위치·네이밍

### 5.1 위치

- **백엔드**: 테스트는 **대상 클래스와 같은 패키지**에 두고, 소스는 `src/main/java`, 테스트는 `src/test/java`. 경로 구조를 동일하게 유지합니다.  
  예: `com.shopeasy.order.OrderService` → `src/test/java/com/shopeasy/order/OrderServiceTest.java`
- **프론트**: 테스트 파일을 **대상 파일과 같은 폴더**에 두고, 파일명만 `*.test.ts` 또는 `*.spec.ts`로 통일합니다.  
  예: `utils/dateUtils.ts` → `utils/dateUtils.test.ts`, `features/users/hooks.ts` → `features/users/hooks.test.ts`

팀에서 `__tests__/` 폴더를 쓰기로 했다면, 그에 맞춰 일관 적용합니다.

### 5.2 네이밍

- **클래스/파일**: `OrderServiceTest`, `useUserList.test.ts` — 대상 이름 + `Test` / `.test.ts`.
- **테스트 메서드/케이스**: “조건_동작_기대결과” 또는 “기대결과 when 조건” 형태로 읽기 쉽게 작성합니다.  
  예: `getOrder_notFound_throws()`, `미인증 시 401 반환`.
- **describe 블록**: 대상 단위(함수/컴포넌트/훅)와 시나리오를 구분해 묶습니다.  
  예: `describe('formatDate', () => { ... })`, `describe('OrderApi', () => { describe('GET /orders/:id', () => { ... }) })`.

## 6. 실행·CI

### 6.1 로컬 실행

- **백엔드**: `./gradlew test` 또는 `mvn test`. 특정 클래스만 실행할 수 있도록 IDE 또는 `-Dtest=OrderServiceTest` 등 사용.
- **프론트**: `npm run test` 또는 `pnpm test`. watch 모드로 변경 시마다 실행되게 할 수 있습니다.

### 6.2 CI

- **커밋/PR 시**: 백엔드 테스트 + 프론트 테스트를 모두 실행하고, **실패 시 빌드 실패**로 처리합니다.
- **테스트 결과**: JUnit XML / Vitest JSON 등으로 리포트를 남기고, 커버리지는 참고용으로 수집해도 됩니다. “커버리지 N% 강제”보다는 **중요 모듈·API·권한**에 대한 테스트 존재를 우선합니다.

### 6.3 환경

- 테스트 시 **별도 프로파일**(예: `test`)을 사용하고, DB는 H2 또는 Testcontainers로 운영 DB와 분리합니다. **비밀·외부 URL**은 테스트 전용 값 또는 모킹으로 처리합니다.

## 7. 새 기능 추가 시 테스트 체크리스트

개발 표준의 [새 화면/기능 추가 시 체크리스트](02-개발-표준.md#7-새-화면기능-추가-시-체크리스트)에 이어서, 테스트 측면에서 확인할 항목입니다.

1. **새 API(백엔드)**  
   - 성공 시 200 + 표준 응답 포맷(`success`, `data`) 검증하는 테스트 추가.  
   - 미인증 401, 권한 없음 403, 잘못된 파라미터 400/422 케이스가 있다면 각각 테스트 추가.
2. **비즈니스 로직(서비스/유틸)**  
   - 경계값·예외 케이스에 대한 단위 테스트 추가.
3. **새 API 호출 함수(프론트)**  
   - URL·메서드·응답 파싱이 표준과 일치하는지 테스트에서 모킹 후 검증.
4. **새 훅(useQuery/useMutation)**  
   - 성공/에러 시 반환 구조가 기대와 같은지 `renderHook`으로 검증.
5. **중요 UI(폼 제출, 목록 표시)**  
   - 컴포넌트 테스트에서 로딩/에러/정상 데이터 표시를 한 번씩 검증.

## 8. 정리 체크리스트

| 구분 | 표준 요약 |
|------|-----------|
| **원칙** | 회귀 방지·의미 있는 검증, 비즈니스·API·권한 우선, 테스트 독립성 |
| **백엔드** | JUnit 5 + Mockito, 서비스 단위 테스트, API 통합(상태·에러코드·응답 포맷), DB는 H2/Testcontainers |
| **프론트** | Vitest/Jest + RTL, utils/hooks/API 함수 단위, 컴포넌트는 행동·데이터 검증, API 모킹 |
| **위치** | 백엔드: 동일 패키지 `*Test`, 프론트: 동일 폴더 `*.test.ts` |
| **네이밍** | 조건_동작_기대결과, describe로 대상·시나리오 구분 |
| **실행** | 로컬: gradle/mvn test, npm test / CI: 모두 실행 후 실패 시 빌드 실패 |
| **새 기능** | 새 API → 응답·401/403 테스트, 새 로직 → 단위, 새 훅/API함수 → 모킹 검증 |
