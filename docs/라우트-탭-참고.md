# 라우트 vs 탭 구조 참고

> 상품 편집 등 "같은 path, 다른 데이터" 화면을 **URL 라우트**로 할지, **탭(메모리 상태)**로 할지 판단 시 참고.

---

## 현재 구조 (탭 기반)

- **라우트**: `/product`, `/product/edit` 등 path만 등록. 실제로 **어떤 상품**을 편집 중인지는 **탭 상태(tab.data.productId)**에만 있음.
- **동작**: 목록에서 상품코드 클릭 → `openTab({ path: '/product/edit', data: { productId } })` → 동일 path로 여러 탭 오픈 가능.
- **URL**: 브라우저 주소창은 `/*` 캐치라우트라서 **편집 중인 상품을 URL만으로 알 수 없음** (예: `/product/edit`만 보이고 productId는 URL에 없음).

---

## 라우트를 쓰는 경우 (예: `/product/edit/:productId`)

| 장점 | 단점 |
|------|------|
| **URL만으로 화면/리소스 특정** 가능 (딥링크, 북마크, 공유) | 같은 화면(path)을 **여러 개** 열려면 라우트만으로는 부족하고, 탭/쿼리 등 추가 구조 필요 |
| **브라우저 뒤로가기**와 자연스럽게 연동 | 목록↔편집 이동 시 **라우트 변경마다** 컴포넌트 마운트/언마운트 → 필요 시 **데이터 재조회** |
| **성능**: 특정 상품만 열 때는 **해당 상품 데이터만** 로드해도 됨 | 여러 상품을 동시에 열어두려면 **탭 UI + 라우트**를 같이 설계해야 함 |

- **성능**: “라우트 쓴다”는 것 자체가 성능을 좋게 만드는 건 아님. **필요한 데이터만 요청/캐시**하는지가 중요. 라우트로 `/product/edit/123` 쓰면 123만 로드하면 되고, 탭으로 여러 개 열면 여러 상품을 미리/동시에 로드할 수 있음. 즉, **요구사항(한 건만 볼지, 여러 건 동시 편집인지)**에 따라 다름.

---

## 탭만 쓰는 경우 (현재처럼)

| 장점 | 단점 |
|------|------|
| **같은 path에 여러 탭**을 쉽게 구현 (productId만 tab.data로 구분) | URL만으로는 “지금 어떤 상품 편집 중인지” 알 수 없음 (딥링크/공유 불리) |
| **뒤로가기**를 “탭 닫기” 등으로 직접 정의 가능 | 목록↔편집 전환 시 **라우트가 바뀌지 않아도** 되므로, SPA 내 상태만으로 동작 |

- **성능**: 탭 기반이라도 **탭이 열릴 때만** 해당 상품 상세 API 호출하면 됨. 목록 API와 상세 API가 분리돼 있으므로 “라우트를 안 쓴다”는 이유만으로 성능이 나쁘지 않음.

---

## 정리

- **성능**: “라우트 쓸까 / 탭만 쓸까”보다 **언제 어떤 API를 몇 번 부르는지**가 더 중요. 현재처럼 목록·상세 분리면 둘 다 유지해도 무방.
- **라우트를 쓰는 게 유리한 경우**:  
  - 특정 상품 편집 화면을 **URL로 공유·북마크**하고 싶을 때.  
  - **한 번에 한 상품만** 열고, 뒤로가기로 “목록으로 돌아가기”를 브라우저 기본 동작으로 쓰고 싶을 때.
- **탭만 써도 되는 경우**:  
  - **여러 상품을 동시에 탭으로 열어두고** 편집하는 UX를 유지할 때.  
  - 딥링크/공유가 중요하지 않을 때.

**결론**: 없애도(탭만 유지해도) 성능·동작상 문제 없음. URL 공유·딥링크·뒤로가기 일치가 필요하면 그때 **편집 화면만** `/product/edit/:productId` 같은 라우트를 도입하고, 탭과 병행할지(탭에서 열 때 라우트도 바꾸기) 결정하면 됨.
