# 주문관리 메뉴 기능 명세

> **대상 독자**: 유지보수 개발자, QA, 매뉴얼 작성자  
> **목적**: 프로세스·기능 파악, 테스트 시나리오, 사용자 매뉴얼 기초 자료

---

## 1. 개요

- **메뉴명**: 주문 관리 (Apps > 주문 관리)
- **역할**: 쇼핑몰·상점에서 수집한 주문(발주)을 법인·상점·상태별로 조회하고, 주문서 처리·출고 준비·발송·배송 등 단계별로 관리한다.
- **인증**: 로그인(세션) 필요. 미인증 시 API 401, 프론트는 로그인 리다이렉트 처리.
- **데이터 출처**: 1차는 OM 내부 주문 테이블(수집·수동 입력)을 전제로 목록/상세 조회를 구현한다. 쇼핑몰 API 수집 연동은 별도 단계에서 설계·적용한다.

---

## 2. 메뉴·화면 구성


| 메뉴(사이드바)  | 경로                    | 화면         | 비고                           |
| --------- | --------------------- | ---------- | ---------------------------- |
| 국내 B2C 주문 | `/order`              | 주문 목록(그리드) | sales_type_cd = B2C_DOMESTIC |
| 해외 B2C 주문 | `/order/overseas-b2c` | 주문 목록(그리드) | sales_type_cd = B2C_OVERSEAS |
| 국내 B2B 주문 | `/order/domestic-b2b` | 주문 목록(그리드) | sales_type_cd = B2B_DOMESTIC |
| 해외 B2B 주문 | `/order/overseas-b2b` | 주문 목록(그리드) | sales_type_cd = B2B_OVERSEAS |
| 기타주문      | `/order/other`        | 주문 목록(그리드) | sales_type_cd = ETC          |
| 비매칭 주문    | `/order/unmatched`    | 비매칭 주문     | 별도 화면(준비 중)                  |


- **동일 프로세스**: 국내 B2C·해외 B2C·국내 B2B·해외 B2B·기타주문 5개 메뉴는 **동일 OrderList**를 사용하며, **om_order_m.sales_type_cd** 값으로 그리드 표시 데이터만 구분한다. 프로세스(상태 탭·툴바·일괄 처리·상세 모달)는 동일.
- **진입**: 사이드바 **주문 관리 > 국내 B2C 주문**(또는 해외 B2C 등) 클릭 → 주문 목록 화면. 목록에서 **주문번호** 등 클릭 시 상세 조회(모달) 진입.
- **다음단계 > 선택 주문서 처리**: 선택한 주문(주문 단위)만 처리. **일괄 주문서 처리**: 현재 필터(법인·메뉴·날짜 등)에 맞는 **발주(접수) 전체**를 한 번에 처리(선택 무관).

## 3. 화면별 기능

### 3.1 국내 B2C / 해외 B2C / 국내 B2B / 해외 B2B / 기타주문 — `/order`, `/order/overseas-b2c` 등

- **공통**: 5개 메뉴 모두 동일 화면(OrderList). 경로별 **sales_type_cd** 로 API 목록·건수 필터만 적용 (국내 B2C=B2C_DOMESTIC, 해외 B2C=B2C_OVERSEAS, 국내 B2B=B2B_DOMESTIC, 해외 B2B=B2B_OVERSEAS, 기타=ETC). DB 컬럼명은 **sales_type_cd**(om_order_m).
- **표시**: 선택된 **법인** 기준 주문 목록(라인 단위). AG Grid, 페이징(100/1000/5000/10000), 엑셀 내보내기 파일명 `orders`.
- **정렬(목록 쿼리)**: `ORDER BY o.combined_ship_no, o.order_no, o.order_id DESC, o.regist_dt DESC, i.line_no ASC`.
- **상태 탭**: 전체, 발주(접수), 입고완료, 배송완료, **보류주문**, **삭제주문**. 탭별 건수는 `GET /api/v1/orders/counts-by-status` 로 표시.
- **툴바**
  - **발주(접수) 탭**: 주문등록(주문수집/수기등록/일괄등록), **선택처리**(선택 출고보류, 선택 삭제), 다음단계(선택 합포장 처리, 일괄 합포장 처리).
  - **합포장처리(PROCESSING) 탭**: 사은품지급(선택/일괄, 준비중), 이전단계(선택 발주주문 처리/일괄 발주주문 처리), 다음단계(선택 출고준비 처리/일괄 출고준비 처리).
  - **출고준비(SHIP_READY) 탭**: 이전단계(발주주문으로 이동, 합포장처리로 이동 - 모두 선택 처리만), 출고지시(선택/일괄, 준비중).
  - **보류주문 탭**: **선택처리**만 — 선택 보류 해제(→ 발주(접수)로 복원), 선택 삭제.
  - 그리드 **체크박스**는 선택처리가 필요한 탭에서만 표시. **전체/삭제주문 탭은 체크박스 미표시**.
- **일괄 처리 단위**
  - **출고보류 / 보류 해제**: 주문 단위(orderId + registDt). 선택된 라인에서 주문 키만 유일하게 추출해 처리.
  - **선택 삭제**: **라인(아이템) 단위**(orderId + registDt + lineNo). 선택한 그리드 행 수만큼 `om_order_item_m.is_deleted = true` 처리. 삭제 전 SweetAlert로 확인.
- **필터**: 법인(필수), 날짜 구분(등록일/주문일), 기간(From~To). 상점 필터는 API 지원 시 연동.
- **컬럼**: 순번, 주문번호, 상품별주문번호, **합포장번호(발주 탭 제외)**, 쇼핑몰/상점, 주문일/등록일, 주문처리상태, 상품코드/명, 수량, 금액, 할인금액, 수령인, **수령인 연락처, 수령인 휴대전화, 주소, 상세주소, 우편번호**. (생성일시 컬럼은 미표시.) 주문번호·행 클릭 시 상세 모달.
- **배송 컬럼 조건 노출**
  - 발송준비(`DELIVERY_READY`): 송장번호, 출고지시일시, 출고지시자
  - 배송중(`SHIPPING`): 송장번호, 출고지시일시, 출고지시자, 출고완료일
  - 배송완료(`DELIVERED`): 송장번호, 출고지시일시, 출고지시자, 출고완료일, 배송완료일
- **그리드 인라인 편집**
  - **수령인 관련**(수령인(이름), 수령인 연락처, 휴대전화, 주소, 상세주소, 우편번호): 셀 편집 후 자동 저장 → 주문 마스터 **order_info** 반영(상세 조회 후 해당 필드만 병합해 PUT). 저장 성공 시 알림 없음, 실패 시에만 SweetAlert 에러.
  - **수량·금액·할인금액**: 셀 편집 후 자동 저장 → 해당 **라인(om_order_item_m)** 만 반영(상세 조회 후 해당 lineNo 라인만 병합해 PUT). 수량은 0 이상 정수만 허용. 저장 성공 시 알림 없음, 실패 시 에러 표시 후 셀 되돌림(undoCellEditing).

### 3.2 주문 상세 (모달)

- **진입**: 목록에서 주문번호 또는 행 클릭 → 상세 모달. `GET /api/v1/orders/{orderId}?registDt=...&corporationCd=...(필수)&storeCd=...(선택)`.
- **표시**: 주문 마스터(주문번호, 상점, 상태, 주문일, 수령인·주소·연락처 등) + 주문 라인(상품, 수량, 단가, 금액). 탭: 주문 마스터 / 주문 라인.
- **수정**: 마스터·라인 편집 후 **저장** 시 `PUT /api/v1/orders/{orderId}`. 취소/저장 버튼 영역은 푸터 스타일(위 구분선 유지, 아래 여백 축소).
- **소속 검증**: 상세 조회·수정 시 `corporation_cd`(필수), `store_cd`(선택) 로 본인 소속 주문만 접근.

### 3.3 주문 일괄등록 (엑셀 양식)

- **용도**: 발주주문 > 주문등록 > **주문 일괄등록** 시 사용할 엑셀 양식. 수기 주문 등록 API(`ManualOrderCreateRequest`) 구조에 맞춤.
- **양식 파일**
  - 경로: `frontend/public/order_import_template.xlsx`
  - 생성: `cd frontend && node scripts/create-order-import-template.js` 실행 시 위 경로에 생성됨.
- **시트**: `주문일괄등록` 1개. 한 행 = 주문 **라인** 1건. 같은 **주문번호** 값을 가진 행들이 하나의 주문으로 묶여 등록되며, 해당 값이 **order_no** 컬럼에 저장됨.
- **화면에서 적용되는 값 (양식에 없음)**
  - **법인코드**: 업로드 시 선택된 법인으로 일괄 적용.
  - **판매유형코드**: 업로드한 메뉴(국내 B2C 등)의 판매유형으로 일괄 적용.
  - **등록일**: 엑셀 업로드한 날짜로 저장.
- **컬럼 요약**
  - **주문번호**(필수): 같은 값 = 한 주문. DB **order_no**에 저장. 예: ORD-001, ORD-001, ORD-002 → 주문 2건. B2C/B2B(sales_type_cd)별로 동일 주문번호 허용.
  - **상점코드**(필수): 해당 법인에 등록된 상점만 허용. **쇼핑몰코드**는 엑셀 양식에서 제거·상점으로 mall_cd 자동 설정. **주문일**(선택, yyyy-MM-dd).
  - 수령인/주문자/메모/배송비/**결제방법코드**: 주문 공통(선택). 결제방법코드는 공통코드 **PAYMENT_METHOD** 전체 사용 가능(CARD, CASH, BANK_TRANSFER, MOBILE, ETC).
  - **상품코드**(필수), **상품명**(참고용), **수량**(필수, 기본 1), **금액**(필수, 기본 0), **할인금액**(선택, 기본 0).
- **일괄등록 업로드**: 발주(접수) 탭 툴바 **주문등록 > 주문 일괄등록** 클릭 시 엑셀 파일 선택 → `POST /api/v1/orders/bulk-import` (multipart: file, corporationCd, salesTypeCd). 응답: successOrderCount, failOrderCount(필수값 누락으로 미등록된 주문 수), successLineCount, errors. 일부만 등록된 경우에도 등록된 주문 수를 메시지로 표시(§9.5).

## 4. 프로세스 요약

1. **목록 보기**: 법인 선택 → GET `/api/v1/orders`(corporationCd 필수, **salesTypeCd**(메뉴별), orderProcessStatus·dateType·orderDtFrom/To·showDeletedOnly·page·size) → 그리드 표시. 탭별 건수는 GET `/api/v1/orders/counts-by-status`(동일 쿼리 파라미터).
2. **상세 진입**: 주문번호(또는 행) 클릭 → GET `/api/v1/orders/{orderId}?registDt=...&corporationCd=...&storeCd=...` → 주문 마스터 + 라인 목록 표시·수정.
3. **일괄 처리**: 그리드에서 행 선택 후 툴바 선택처리 → 출고보류/보류 해제(주문 단위) 또는 선택 삭제(라인 단위, 확인 후). 요청 body에 `corporationCd`(필수), `storeCd`(선택) 포함.
4. **엑셀 다운로드**
  - **목록 주문 다운로드**: 현재 그리드 로드 데이터(현재 페이지) 기준.
  - **전체 주문 다운로드**: 현재 필터 조건의 전체 데이터 기준(전용 API 호출).

| POST | `/api/v1/orders/bulk-delete`                           | 선택 라인 삭제               | body: **corporationCd(필수)**, storeCd(선택), items: [{ orderId, registDt, **lineNo** }]. **라인 단위** 소프트 삭제                                                                                                                                      |


- **주문서 처리·합포장**: 선택 주문을 PROCESSING으로 변경 시, **상점코드(store_cd)** 와 **수령인(수령인명·주소·연락처)** 이 같은 주문끼리 동일한 **합포장번호(combined_ship_no)** 를 부여. 합포장번호 형식: **yyyyMMdd** + **'-'** + **LPAD(시퀀스, 8, '0')** (예: 20260301-00000001). 시퀀스 seq_combined_ship_no로 유일 보장. docs/ddl/seq_combined_ship_no.sql.
- **전체 엑셀 다운로드 성능 개선**: 서버 메모리 사용량 완화를 위해 `StreamingResponseBody + SXSSFWorkbook` 기반으로 스트리밍 응답. DB도 cursor 조회(`fetchSize`, `FORWARD_ONLY`)로 대량 데이터를 순차 처리.
- **동시성 제어(상태 전이)**: 주요 `...처리` UPDATE는 기대 현재 상태 조건(`expectedCurrentStatus`)을 포함해 경합 시 중복 처리 방지.
- **처리 결과 메시지**: 요청 건수와 실제 반영 건수를 구분해 안내(경합으로 일부 미반영 시 스킵 건수 표시).
- **인증**: 세션 없으면 401, 프론트는 로그인 리다이렉트.
- **소속 검증**: 목록·상세·일괄 처리 시 WHERE에 **corporation_cd**(필수), **store_cd**(선택) 적용.

## 9. 최근 변경 분석/정리

### 9.1 엑셀 다운로드 UX/기능 분리

- 엑셀 버튼을 하위 그룹으로 분리:
  - `목록 주문 다운로드`: 현재 그리드 페이지/로드 데이터
  - `전체 주문 다운로드`: 현재 필터 전체(전용 API)
- 전체 다운로드는 확인 팝업/진행 중 중복 클릭 방지(버튼 비활성) 적용.

### 9.2 Full Export 서버 성능 개선

- 이전: 대용량 파일을 `byte[]`로 메모리에 모두 적재 후 응답.
- 현재: 응답 스트림으로 직접 write하는 방식으로 변경.
  - `ResponseEntity<byte[]>` -> `ResponseEntity<StreamingResponseBody>`
  - `ByteArrayOutputStream` 제거, `SXSSFWorkbook` 임시파일 압축 사용
- DB 조회도 `LIMIT/OFFSET 반복`에서 **cursor 기반 순차 조회**로 변경.

### 9.3 일괄 상태 처리 동시성 개선

- 상태 전이 UPDATE에 기대 현재 상태 조건 추가:
  - `ORDER_RECEIVED -> HOLD`
  - `HOLD -> ORDER_RECEIVED`
  - `ORDER_RECEIVED -> PROCESSING`
  - `PROCESSING -> SHIP_READY`
  - `SHIP_READY -> PROCESSING`
- 다단계 처리 메서드는 서비스 트랜잭션(`@Transactional`)으로 묶어 원자성 강화.
- 프론트 성공 메시지는 “요청/실반영/스킵”을 구분해 오해를 줄이도록 개선.

### 9.4 그리드/모달 정책 정리

- 전체/삭제주문 탭은 체크박스 열 미표시.
- 주문 수정 모달의 편집 가능 여부는 탭이 아니라 **해당 주문의 주문처리상태 기준**으로 판단.

### 9.5 주문 엑셀 등록·수기 등록·상태 로그 정책

- **주문번호 중복·UK**
  - 중복 검사 및 DB 유니크 제약에 **sales_type_cd** 포함. B2C와 B2B는 동일 주문번호를 가질 수 있음.
  - 앱: `selectExistingOrderNosForBulk(registDt, corporationCd, salesTypeCd, list)`. DB UK: (regist_dt, corporation_cd, sales_type_cd, mall_cd, store_cd, order_no). 기존 DB는 `docs/ddl/om_order_m_uk_add_sales_type_cd.sql` 1회 실행.
- **엑셀 일괄등록**
  - **상점코드**: 엑셀에는 쇼핑몰코드 없음. 상점코드로 `om_mall_store_m` 조회해 해당 법인에 등록된 상점만 허용, 아니면 `orders.bulkImport.invalidStoreCd` 오류로 전체 미등록.
  - **필수값 누락**: 주문번호·상점코드·상품코드·수량·금액 등 필수값이 없는 **해당 주문 전체**는 등록 대상에서 제외. 나머지 주문만 등록 후, 응답에 등록 성공 건수(successOrderCount, successLineCount)와 미등록 건수(failOrderCount), 검증 오류 목록(errors) 반환.
  - **결과 표시**: 일부만 등록된 경우 "등록된 주문 N건(라인 M건)입니다. K건은 필수값 누락으로 미등록되었습니다." 및 상세 오류(행/필드별) 표시.
  - **DB 제약 위반**: 주문 마스터 UK 위반 시 GlobalExceptionHandler에서 `orders.bulkImport.duplicateOrderNo` 메시지 반환(상품코드 중복 메시지와 구분).
- **상태 로그(om_order_status_log)**
  - **등록 시(수기 등록·엑셀 일괄등록)**: **ETC/REGISTERED** 로그는 남기지 않음. **ORDER_PROCESS/ORDER_RECEIVED** 만 기록.
  - 그 외 ETC(UPDATED, LINE_DELETED 등)는 기존대로 기록.
