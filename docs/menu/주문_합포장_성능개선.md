# 주문서 처리(합포장번호 채번·업데이트) 속도 개선

> 합포장번호 채번 및 일괄 UPDATE 시 적용 가능한 개선 방안 요약.

## 현재 구조

1. **마스터 조회** `selectOrderMastersByKeys`: 선택 (orderId, registDt)에 해당하는 행만 조회. `INNER JOIN (VALUES (...), ...)` 사용.
2. **채번** `getNextCombinedShipNoValues(groupCount)`: 그룹 수만큼 시퀀스 nextval 1회 쿼리로 조회.
3. **업데이트** `updateOrderProcessStatusAndCombinedShipNoBulk`: `UPDATE ... FROM (VALUES ...)` 를 **배치 크기(현재 1000건)** 단위로 반복.

## 적용된 개선

| 항목 | 내용 |
|------|------|
| **배치 크기** | UPDATE를 500 → **1000건** 단위로 수행해 round-trip 횟수 감소. (`OrderService.BULK_ORDER_PROCESS_BATCH_SIZE`) |
| **채번** | 그룹 수만큼 nextval을 **한 번에** `generate_series(1, count)` 로 조회. (기존 유지) |

## 추가로 고려할 수 있는 개선

### 1. 배치 크기 조정

- **상수**: `OrderService.BULK_ORDER_PROCESS_BATCH_SIZE` (현재 1000).
- **조정**: 2000~3000으로 올리면 UPDATE 호출 횟수는 더 줄지만, 쿼리·파라미터 크기와 DB 부하가 커질 수 있음. DB/네트워크 상황에 맞춰 튜닝.

### 2. 마스터 조회 대량일 때 (수만 건)

- **현상**: `items` 가 수만 건이면 `VALUES (..., ..., ...)` 가 매우 길어져 파싱·전송 비용 증가.
- **방안 A**  
  - 마스터 조회를 **여러 번** 나눠서 호출(예: 3000건씩), 결과를 Java에서 합친 뒤 그룹핑·채번·UPDATE는 그대로.  
  - VALUES 크기만 줄이고, round-trip은 증가.
- **방안 B**  
  - PostgreSQL **배열 + unnest** 사용: `WHERE (order_id, regist_dt) IN (SELECT * FROM unnest(#{orderIds}::bigint[], #{registDts}::date[]))`  
  - 파라미터는 배열 2개만 전달. JDBC에서 `java.sql.Array` 또는 MyBatis TypeHandler로 배열 바인딩 필요.

### 3. DB 함수로 한 번에 처리 (최대 단일 트랜잭션 절감)

- **아이디어**: plpgsql 함수 하나에서 (1) 전달받은 (order_id, regist_dt) 배열로 마스터 조회 → (2) 그룹 키로 그룹핑 → (3) 그룹 수만큼 nextval → (4) 합포장번호 형식(yyyyMMdd-LPAD(시퀀스,8,0)) 적용 후 UPDATE.
- **효과**: 애플리케이션–DB **왕복 1회**, DB 내부에서만 처리해 네트워크·파싱 비용 감소.
- **단점**: 로직이 DB로 이전되어 유지보수·버전 관리·테스트 방식을 팀 정책에 맞게 정해야 함.

### 4. 인덱스·파티션

- **om_order_m**: UPDATE 조건이 `(order_id, regist_dt)` + `corporation_cd`(, `store_cd`) 이므로 PK로 이미 단일 행 접근. 추가 인덱스는 효과 제한적.
- **파티션**: `regist_dt` 월별 파티션이면 해당 월 파티션만 스캔되므로 유리. 유지.

### 5. 트랜잭션

- 현재 한 트랜잭션에서 마스터 조회 → 채번 → 배치 UPDATE 수행. 롤백 시 전체 롤백으로 일관성 유지. 유지 권장.

---

**요약**: 배치 크기 1000 적용으로 UPDATE round-trip이 이미 절감된 상태. 더 필요한 경우 (2) 마스터 조회 분할/배열 또는 (3) DB 함수를 검토하면 됨.
