# 주문 그리드 표시 속도 검토

> 목적: 주문 그리드가 “최대 N건”까지 뿌릴 때 느리게 느껴지는 구간을 측정할 수 있는지, 개선 여지가 있는지 검토. **소스 변경은 지시 시에만 진행.**

---

## 1. 현재 구조 요약


| 구간      | 내용                                                                                                                                                                                      |
| ------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **프론트** | 기본 `pageSize=1000`, 옵션 `[100, 1000, 5000, 10000]`. 한 번에 한 페이지만 요청.                                                                                                                      |
| **백엔드** | `size` 최대 5000으로 제한. 한 번의 목록 API 호출 시 **DB 쿼리 2회** (COUNT + 목록 SELECT).                                                                                                                 |
| **API** | `GET /api/v1/orders?corporationCd=...&page=0&size=1000&...` → `PagedData` (items, total, totalPages 등).                                                                                 |
| **쿼리**  | `om_order_m` JOIN `om_order_item_m` JOIN `om_mall_m` JOIN `om_mall_store_m`, WHERE/날짜/상태/검색, ORDER BY combined_ship_no, order_no, order_id DESC, regist_dt DESC, line_no, LIMIT/OFFSET. |
| **그리드** | AG Grid, `pagination={false}`(서버 페이징만), `rowData={items}`. 컬럼 수 많음(30개 이상), 여러 컬럼에 `cellRenderer`(React 컴포넌트) 사용.                                                                       |


---

## 2. 속도 측정 방법 (도입 가능한 지표)

소스에 타이밍을 넣지 않고도 **지금** 할 수 있는 것과, **측정 코드를 한 번 넣으면** 잴 수 있는 것을 구분해 정리했습니다.

### 2.1 지금 바로 가능 (도구만 사용)

- **브라우저 DevTools → Network**
  - `/orders` 요청 선택 후 **Time / Waiting (TTFB) / Content Download** 확인.
  - TTFB: 서버에서 응답을 시작할 때까지 시간 (백엔드 + DB 포함).
  - Content Download: 응답 본문(JSON) 전송 시간. 응답 크기가 크면 여기서 시간이 길어짐.
- **Chrome DevTools → Performance**
  - 주문 목록 화면 로드 후 녹화 → “그리드가 보이기까지” 구간을 타임라인에서 확인.
  - Main 스레드에서 긴 작업(JS 실행, 레이아웃, 페인트)이 어디서 나오는지 확인 가능.

### 2.2 측정 코드를 넣었을 때 잴 수 있는 것 (지시 시 구현)

- **API 구간**
  - 프론트: `fetchOrderList` 호출 직전/직후 `performance.now()` → **클라이언트 기준 전체 목록 로딩 시간** (네트워크 + 파싱 + normalize 포함).
  - 백엔드: `getOrderList` 안에서 `selectOrderListCount` / `selectOrderList` 호출 전후 로그(또는 AOP) → **DB 쿼리별 소요 시간**.
- **그리드 표시 구간**
  - AG Grid `onGridReady` 또는 `api.addEventListener('firstDataRendered', ...)` 에서 `performance.now()`와 조합 → **첫 데이터가 그리드에 그려진 시점**까지 시간.
- **정리하면**
  - “요청 시작 ~ 목록 데이터 수신” = API 측정.
  - “데이터 수신 ~ 그리드 첫 렌더” = 그리드 측정.
  - “DB COUNT 소요”, “DB SELECT 소요” = 백엔드 로그로 구분 가능.

## 3. 병목 가능성이 있는 구간


| 구간                | 설명                                                                                                                  | 개선 시 검토할 것                                                                                          |
| ----------------- | ------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **DB COUNT**      | 매번 목록 조회 시 `selectOrderListCount`로 전체 조건에 맞는 건수를 셈. 데이터가 많으면 비용이 큼.                                                 | (1) COUNT를 캐시하거나 주기적으로만 갱신, (2) 또는 “대략 건수”만 보여주고 정확한 total은 필요 시에만 조회.                              |
| **DB SELECT**     | `selectOrderList`가 JOIN 4개 + ORDER BY + LIMIT/OFFSET. 정렬/필터에 맞는 인덱스가 없으면 정렬 비용이 큼.                                  | `docs/ddl/om_order_indexes_analysis.md` 등에 따라 인덱스 적용 여부 확인. ORDER BY와 WHERE 조건에 맞는 복합 인덱스.          |
| **응답 크기**         | 1000건 × 컬럼 수 많음 → JSON 용량 증가.                                                                                       | (1) 화면에 안 쓰는 컬럼은 API에서 제외하거나 별도 엔드포인트로 분리, (2) 페이지 크기 기본값을 100~500 등으로 낮춰서 체감 속도 개선.                |
| **프론트 normalize** | `fetchOrderList` 내부에서 `items.map(normalizeOrderListItem)`로 1000건 객체 생성.                                             | 건수가 클 때는 비용이 쌓임. 필요 시 워커나 청크로 나누는 방안 검토(구현 복잡도 대비).                                                 |
| **그리드 초기 렌더**     | AG Grid는 가상화로 “보이는 행”만 DOM에 두지만, **처음 rowData를 세팅할 때** 내부 모델 구축과 컬럼 정의 처리 비용이 있음. 컬럼 수가 많고 `cellRenderer`가 많을수록 부담. | (1) `getRowId`로 행 식별 제공(이미 되어 있으면 유지), (2) 불필요한 컬럼은 기본 숨김 등으로 초기 렌더 부담 감소, (3) 페이지 크기를 줄이면 당연히 빨라짐. |

## 4. 개선 시 검토할 항목 (우선순위 예시)

1. **백엔드**
  - 주문 목록용 **인덱스**가 실제 쿼리(WHERE, ORDER BY)와 맞는지 `om_order_indexes_analysis.md` 기준으로 재점검.
  - **COUNT**가 매 요청마다 꼭 필요한지 검토. 필요하다면 쿼리 최적화 또는 캐시/대략 건수 전략.
  - 목록 API에서 **필요한 컬럼만** SELECT하거나, “목록용 경량 DTO”를 두어 응답 크기 축소 검토.
2. **프론트**
  - **기본 pageSize**를 1000 → 100 또는 500으로 낮추면 “첫 화면”이 빨라짐. (이미 100 옵션이 있음.)
  - **측정 코드**를 한 번 넣어서 “API 구간” vs “그리드 첫 렌더 구간” 중 어디가 더 오래 걸리는지 수치로 확인 후, 그에 맞춰 백엔드/그리드 중 어디를 손댈지 결정.
3. **그리드**
  - 컬럼 수·`cellRenderer` 수가 많으므로, “표시할 컬럼을 탭/상태별로 최소화”하면 초기 렌더가 가벼워질 수 있음. (이미 상태별 hide가 있으면 유지.)
  - AG Grid 옵션 중 `rowBuffer`, `maxBlocksInCache` 등은 기본값으로 두고, **먼저 데이터 건수·API/DB 시간**을 줄이는 것이 효과가 큰 경우가 많음.

## 5. 응답 JSON 압축 및 기타 방법

### 5.1 Gzip 압축 (적용됨)

- **설정**: `application.yml`에 `server.compression.enabled: true` 적용.  
  응답이 1KB 이상일 때 `application/json` 등에 gzip 압축을 사용합니다.
- **동작**: 클라이언트가 `Accept-Encoding: gzip`을 보내면 서버가 압축해 응답하고, 브라우저/axios가 자동으로 해제합니다. **프론트 수정 불필요**.
- **효과**: JSON은 보통 70~85% 정도 줄어들어, 전송 시간·체감 속도 개선이 기대됩니다. (이미 압축된 스트림 등은 제외.)

### 5.2 그 외 방법

| 방법 | 설명 |
|------|------|
| **목록 전용 경량 DTO** | 그리드에 안 쓰는 컬럼을 API에서 제외해 JSON 크기 축소. (예: 상세용 필드는 목록에서 제거) |
| **페이지 크기** | 10,000건 대신 1,000~2,000건 단위로 요청하면 응답·파싱·그리드 부담이 줄어듦. |
| **COUNT 생략/캐시** | 전체 건수를 매번 안 구하고, “이상 N건”만 표시하거나 주기적으로만 갱신. (요구사항에 따라 검토) |

---

## 6. 요약

- **속도 “계산”**: 브라우저 Network/Performance로 대략 잴 수 있고, **API 전후·그리드 firstDataRendered에 `performance.now()`를 넣으면** “목록 API 구간”과 “그리드 표시 구간”을 나눠서 수치로 볼 수 있음.
- **개선 포인트**: (1) DB 인덱스·COUNT/SELECT 부담, (2) 응답 크기·페이지 크기, (3) 그리드 초기 로드(컬럼/건수) 순으로 점검하는 것을 권장.  
- **실제 수정**은 말씀해 주실 때만 반영하면 됨.
- **응답 압축**: `server.compression.enabled: true`로 gzip 적용해 두었음. 대용량 목록(수천~1만 건) 시 전송량 감소로 체감 속도 개선 기대.
