# 사용자 정보 메뉴 기능 명세

> **대상 독자**: 유지보수 개발자, QA, 매뉴얼 작성자  
> **목적**: 프로세스·기능 파악, 테스트 시나리오, API·DB 연계 설계의 단일 참고

---

## 1. 개요

- **메뉴명**: 사용자 정보 (기초정보)
- **역할**: `OM_USER_M` 기준 BO 사용자 목록을 검색·필터하고, 그리드에서 일부 필드를 직접 수정하거나 모달로 상세 수정·신규 등록한다. 등급·상태 코드는 공통코드, 권한그룹은 권한그룹 마스터와 연동한다.
- **인증**: 로그인(세션) 필요. 미인증 시 API 401, 프론트는 로그인 리다이렉트 처리.
- **UI 정렬**: 화주(법인) 정보와 동일한 툴바 패턴(`orders-filter-row`, `orders-search-group`, `FiFilter` / 검색어 / `FiSearch`+검색 / 보조 버튼). 등록·수정 모달은 화주(법인)와 동일한 폼 패턴(`FloatingRow`, 필수 `.required`, `form-control-sm`, 푸터 `product-modal__footer--compact`, `btn-phoenix-secondary`·`btn-phoenix-primary btn-sm`).
- **구현 상태(요약)**: `/basic/users`에 사용자 목록·필터·엑셀·그리드 수정·등록/수정 모달이 연결됨. API: `GET /api/v1/users`(레거시), `GET /api/v1/users/manage`, `GET/PUT` detail·field, `POST` register, `GET` export, `GET /api/v1/auth-groups`. DB에 **`om_auth_group_m`** DDL 적용 및 권한그룹 데이터(예: `docs/dml/om_auth_group_m_seed.sql`)가 있어야 권한 필터·등록 검증이 동작한다.

## 3. 화면별 기능

### 3.1 필터·옵션 데이터 소스

| UI | 소스 | 비고 |
| --- | --- | --- |
| 등급 `select` | `GET /api/v1/codes/list/USER_GRADE?lang=` | 공통코드 **`CM_CODE_M`**, `main_cd = USER_GRADE`. 표시명은 `CodeItem`(subCd, codeNm). 시드 미등록 시 빈 목록 가능. |
| 권한그룹 `select` | `GET /api/v1/auth-groups` (제안) | **`OM_AUTH_GROUP_M`**. `is_deleted = false`, 필요 시 `is_active = true`만. 정렬 `sort_seq`, `auth_group_cd`. |

> 참고: 초기 요구에서 공통코드 테이블을 `om_code_m`으로 표기한 경우, 본 프로젝트 DDL·API는 **`CM_CODE_M`** / **`/api/v1/codes/list/{mainCd}`** 를 사용한다.

### 3.2 사용자 목록 — `/basic/users`

- **데이터(구현 후)**: `GET /api/v1/users/manage` — `keyword`, `gradeCd`, `authGroup`, `page`, `size`.
- **정렬**: 서버 **`created_at DESC`** (등록일시 역순). 그리드 컬럼 정렬은 끔(`defaultColDef.sortable: false`) 권장(화주 관리와 동일).
- **삭제 제외**: `is_deleted = false` 행만 조회.
- **번호 컬럼**: 전역 역순(첫 행=전체 건수 기준 큰 번호), `colId: 'rowNum'`.
- **검색어**: `user_id`, `user_nm`, `user_info->>'email_id'` **ILIKE** 부분 일치 (OR).
- **필터**: `gradeCd`가 있으면 `user_info->>'grade_cd'` 일치, `authGroup`이 있으면 `user_info->>'auth_group'` 일치.
- **컬럼**: 번호, 아이디, 이름, 이메일, 등급, 권한그룹, 상태, 최종 로그인 일시, 등록일시.  
  - 등급·권한·상태 **표시명**: 그리드는 코드값 + (조회 시 서버가 내려준 `gradeNm`, `authGroupNm` 등) 또는 클라이언트에서 옵션 맵으로 해석.
- **아이디**: 링크 → **사용자 수정 모달** (`GET /api/v1/users/detail?userId=` → `PUT /api/v1/users/detail`).
- **그리드 직접 편집**  
  - **이름(`user_nm`), 이메일(`user_info.email_id`), 등급(`grade_cd`), 권한그룹(`auth_group`), 상태(`user_status`)** → `PUT /api/v1/users/field`.  
  - 이름·이메일 빈 값 불가. 실패 시 SweetAlert + AG Grid `undoCellEditing`.  
  - JSONB 키 변경 시 기존 `user_info`와 **병합(merge)** 후 저장·`password` 등 미변경 키 유지.
- **페이징**: `DataGridPaginationFooter`, `pageSize` 50/100/500/1000.
- **엑셀다운로드**: `GET /api/v1/users/export` — 현재 **적용된** `keyword`, `gradeCd`, `authGroup`과 동일 조건의 전체 행, 헤더 언어(`lang` / `X-Requested-Lang`).

### 3.3 사용자 등록 모달

- **필수**: 아이디(`user_id`), 이름(`user_nm`), 비밀번호(정책에 맞는 입력), 권한그룹(`auth_group`), 상태(`user_status`). 등급·이메일 등은 정책에 따라 필수 여부 결정.
- **저장**: `POST /api/v1/users/register` — `user_info`에 bcrypt 비밀번호·기본 JSON 키 채움. `auth_group_cd`는 `OM_AUTH_GROUP_M`에 존재하는지 검증.
- **닫기**: 헤더 `×`, 하단 **취소**.

### 3.4 사용자 수정 모달

- 상세 로드 후 편집. **아이디**는 읽기 전용.
- 비밀번호 변경은 **별도 필드·옵션**(변경 시에만 해시 갱신) 권장.
- 저장: `PUT /api/v1/users/detail` — 성공 시 모달 닫기 + 목록 무효화.

### 3.5 공통 모달 UI

- 화주(법인) 정보와 동일 — `FloatingRow`, `form-floating.required`, 푸터 `product-modal__footer--compact`.

## 4. 프로세스 요약

1. **목록**: 등급·권한 옵션 API 로드 → `users/manage` 조회 → 그리드 표시(`created_at DESC`).
2. **검색·필터**: 조건 적용·1페이지.
3. **셀 편집**: 값 변경 → `PUT /users/field` → 성공 시 무효화, 실패 시 원복.
4. **아이디 클릭**: 상세 로드 → 모달 수정 → `PUT /users/detail`.
5. **등록**: 모달 → `POST /users/register` → 목록 무효화.
6. **엑셀**: `GET /users/export` 동일 검색·필터.

## 5. API 명세

### 5.1 요약 표

| 메서드 | 경로 | 용도 | 상태 |
| ------ | ---- | ---- | ---- |
| GET | `/api/v1/codes/list/USER_GRADE` | 등급 select 옵션 (`lang`) | 기존 |
| GET | `/api/v1/auth-groups` | 권한그룹 select 옵션 | 구현됨 |
| GET | `/api/v1/users` | 단순 페이징 목록(`keyword`, `page`, `size`) | 기존(레거시) |
| GET | `/api/v1/users/manage` | 관리 그리드 페이징 | 구현됨 |
| GET | `/api/v1/users/export` | 엑셀 (`keyword`, `gradeCd`, `authGroup`, `lang`) | 구현됨 |
| GET | `/api/v1/users/detail` | 모달 상세 (`userId`) | 구현됨 |
| PUT | `/api/v1/users/field` | 그리드 단건 필드 수정 | 구현됨 |
| PUT | `/api/v1/users/detail` | 모달 일괄 수정 | 구현됨 |
| POST | `/api/v1/users/register` | 신규 등록 | 구현됨 |

### 5.2 `GET /api/v1/users/manage`

**Query**

| 파라미터   | 타입   | 필수 | 설명 |
| ---------- | ------ | ---- | ---- |
| keyword    | string | N    | 아이디·이름·이메일 부분 검색 |
| gradeCd    | string | N    | `user_info.grade_cd` 정확 일치 |
| authGroup  | string | N    | `user_info.auth_group` 정확 일치 |
| page       | int    | N    | 0-based, 기본 0 |
| size       | int    | N    | 기본 100, 최대 5000 |

**응답** `ApiResponse.data`: `PagedData<UserManageRow>`

**UserManageRow (제안 필드)**

| 필드명           | 타입   | 설명 |
| ---------------- | ------ | ---- |
| userId           | string | PK |
| userNm           | string | 이름 |
| emailId          | string | 이메일 |
| gradeCd          | string | 등급 코드 |
| gradeNm          | string | 등급 표시명(서버 조인·보강, 없으면 null) |
| authGroup        | string | 권한그룹 코드 |
| authGroupNm      | string | 권한그룹명(OM_AUTH_GROUP_M 조인) |
| userStatus       | string | 상태 코드 |
| userStatusNm     | string | 선택, 공통코드명 |
| lastLoginDtm     | string | ISO8601 또는 DB 문자열 그대로 |
| createdAt        | string | 등록일시 (TIMESTAMPTZ → ISO 권장) |

### 5.3 `PUT /api/v1/users/field`

**Body** `application/json`

```json
{
  "userId": "admin",
  "field": "user_nm",
  "value": "관리자"
}
```

**field 허용값**

| field 값     | 매핑 | value 제약 |
| ------------ | ---- | ------------ |
| `user_nm`    | `om_user_m.user_nm` | 빈 문자열 불가 |
| `email_id`   | `user_info.email_id` | 이메일 형식(선택 검증) |
| `grade_cd`   | `user_info.grade_cd` | `USER_GRADE`에 존재하는 sub_cd |
| `auth_group` | `user_info.auth_group` | `OM_AUTH_GROUP_M.auth_group_cd` 존재 |
| `user_status`| `user_info.user_status` | 허용 열거(공통코드 또는 고정) |

**응답**: `ApiResponse<Void>` 성공 시 200.

**오류 메시지 키(예시)**: `users.invalid_field`, `users.user_nm_required`, `users.grade_not_found`, `users.auth_group_not_found`, `users.not_found`.

### 5.4 `GET /api/v1/users/detail`

**Query**: `userId` (필수)

**응답**: 모달용 DTO — `userId`, `userNm`, `userInfo`에서 파싱한 이메일·등급·권한·상태·휴대전화 등. **`password` 해시는 절대 미포함.**

### 5.5 `PUT /api/v1/users/detail`

**Body**: 상세 폼 전체. 서버는 JSONB merge + `user_nm`·`updated_by`·`updated_at` 갱신. 비밀번호 필드가 비어 있으면 기존 해시 유지.

### 5.6 `POST /api/v1/users/register`

**Body (예시)**

```json
{
  "userId": "newuser01",
  "userNm": "신규",
  "password": "********",
  "emailId": "a@example.com",
  "gradeCd": "USER",
  "authGroup": "ORDER_MANAGER",
  "userStatus": "ACTIVE"
}
```

**응답**: 201 또는 200 + 생성된 `userId`. 중복 아이디 시 `users.duplicate_user_id`.

### 5.7 `GET /api/v1/users/export`

**Query**: `manage`와 동일 필터 + `lang`.

**응답**: `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`, 파일명 예 `users_export.xlsx`. 컬럼은 그리드와 동일 권장.

### 5.8 `GET /api/v1/auth-groups` (제안)

**응답**: `List<{ authGroupCd: string, authGroupNm: string }>`  
조건: `is_deleted = false`, `is_active = true`(정책에 따라 false 포함 가능).

## 8. QA 체크리스트

1. 진입 시 목록이 **등록일시 내림차순**인지 확인.  
2. 검색어·등급·권한그룹 조합 AND 조건·1페이지 동작.  
3. **전체** 버튼으로 초기화 확인.  
4. 아이디 링크 → 수정 모달 로드·저장 → 목록 반영.  
5. 그리드 셀 편집(이름·이메일·등급·권한·상태) → API 성공/실패 시 원복·알림.  
6. 편집 후 `password`·기타 `user_info` 키가 의도치 않게 변하지 않았는지 DB 샘플 확인.  
7. **사용자등록** → 필수값·중복 아이디·권한그룹 미존재 오류.  
8. **엑셀다운로드** 행 수·필터가 목록과 일치.  
9. 등급/권한 select 옵션과 그리드 표시명 일관성.  
10. 툴바·모달 스타일이 화주(법인) 정보 화면과 시각적으로 일관한지 확인.
