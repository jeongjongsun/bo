# 쇼핑몰관리 메뉴 기능 명세

> **대상 독자**: 유지보수 개발자, QA, 매뉴얼 작성자  
> **목적**: 프로세스·기능 파악, 테스트 시나리오, 사용자 매뉴얼 기초 자료

---

## 1. 개요

- **메뉴명**: 쇼핑몰 관리 (Apps > 쇼핑몰 관리)
- **역할**: 주문 수집 대상 쇼핑몰·상점 목록 조회·등록·수정, 상점별 API 접속정보(1:N) 관리.
- **인증**: 로그인(세션) 필요. 미인증 시 API 401, 프론트는 로그인 리다이렉트 처리.

## 2. 메뉴·화면 구성

| 메뉴(사이드바) | 경로 | 화면 | 비고 |
|----------------|------|------|------|
| 쇼핑몰/상점 관리 | `/mall` | 쇼핑몰·상점 목록(그리드) | 상점 단위 행, 접속정보 링크 |
| 접속정보 관리 | `/mall/connection` | 접속정보 관리 | 별도 화면(준비 중) |

## 3. 화면별 기능

### 3.1 쇼핑몰·상점 목록 — `/mall`

- **표시**: 쇼핑몰(om_mall_m) + 상점(om_mall_store_m) 조인 목록. **상점 단위** 한 행(한 행당 한 상점). AG Grid, 페이징(50/100/500/1000/5000), 엑셀 내보내기 파일명 `malls`.
- **툴바**: 판매구분 필터(전체/B2C 국내 등), **상점 등록** 버튼.
- **컬럼**: 번호, 쇼핑몰코드, 쇼핑몰명, 상점코드(클릭 시 수정 모달), **접속정보**(링크), 상점명, 상점 구분, WMS 연동, 통화코드, GMT, 배송타입, 수집타입, 판매구분, 사용여부, 생성자, 생성일시. 상점코드·상점명·상점구분·WMS·통화·GMT·사용여부는 그리드 셀 편집 가능.
- **접속정보**: 행별 **접속정보** 링크 클릭 시 해당 상점의 API 접속정보 패널 오픈(3.3 참고).

### 3.2 상점 등록·수정

- **상점 등록**: 툴바 **상점 등록** → 모달(쇼핑몰, 상점명, 상점 구분, WMS 연동, 통화코드, GMT, 서비스 상태). 법인 선택 필수(상단 법인 선택 후 등록).
- **상점 수정**: 그리드에서 **상점코드** 클릭 시 수정 모달 오픈. 쇼핑몰, 상점명, 상점 구분, WMS 연동, 통화코드, GMT, 서비스 상태 편집 후 저장. 또는 그리드 셀 직접 편집(상점명·상점구분·WMS·통화·GMT·사용여부) 시 API로 즉시 반영.

### 3.3 상점별 API 접속정보 (1:N)

- **진입**: 목록 그리드에서 해당 행의 **접속정보** 링크 클릭.
- **패널**: 제목에 상점명만 표시. 접속정보 목록 테이블(접속별칭, clientId 마스킹, siteCode, 수정/삭제 버튼), **접속정보 추가** 버튼.
- **추가/수정 폼**: 접속별칭(필수), 아이디, 비밀번호, clientId, siteCode, redirectUri, clientSecret, scope. form-floating 스타일. API테스트 버튼(현재 비활성). 저장 후 패널 목록 갱신, 성공 알림(SweetAlert) 표시.
- **삭제**: 행별 삭제 버튼 → 확인 후 삭제.

### 3.4 접속정보 관리 — `/mall/connection`

- (준비 중) 상점별 API 접속정보를 목록·전용 화면에서 관리하는 기능은 향후 구현 예정. 현재는 쇼핑몰/상점 관리 그리드의 **접속정보** 링크로 행 단위 패널에서 조회·추가·수정·삭제 가능(3.3 참고).

## 4. API 요약

| 메서드 | 엔드포인트 | 용도 |
|--------|------------|------|
| GET | `/api/v1/malls` | 쇼핑몰+상점 목록(상점 단위) 페이징. query: keyword, corporationCd, page, size. 응답: ApiResponse.data = PagedData&lt;MallStoreListItem&gt; |
| GET | `/api/v1/malls/options` | 쇼핑몰 옵션 목록(드롭다운). 응답: ApiResponse.data = List&lt;MallOption&gt; |
| POST | `/api/v1/malls/stores` | 상점 등록. body: mallCd, corporationCd, storeCd(선택), storeNm, storeInfo(선택), isActive(선택). |
| PATCH | `/api/v1/malls/stores/{storeId}` | 상점 수정. body: mallCd, storeNm, storeInfo, isActive. |
| GET | `/api/v1/malls/stores/{storeId}/connections` | 해당 상점의 API 접속정보 목록. 응답: ApiResponse.data = List&lt;StoreConnectionItem&gt; |
| POST | `/api/v1/malls/stores/{storeId}/connections` | 접속정보 등록. body: connectionAlias(필수), apiId, apiPassword, clientId, siteCode, redirectUri, clientSecret, scope. |
| PUT | `/api/v1/malls/stores/{storeId}/connections/{connectionId}` | 접속정보 수정. body: 동일. |
| DELETE | `/api/v1/malls/stores/{storeId}/connections/{connectionId}` | 접속정보 삭제. |

## 5. DB/테이블 참고

- **쇼핑몰**: `om_mall_m` (docs/ddl/om_mall_m.sql). mall_cd(PK·비즈니스키), mall_nm, mall_info(JSONB), is_active, created_at, updated_at, created_by, updated_by.
- **상점**: `om_mall_store_m` (docs/ddl/om_mall_store_m.sql). store_id(PK), mall_cd(FK), corporation_cd(FK), store_cd, store_nm, store_info(JSONB), is_active, created_at, updated_at, created_by, updated_by. UK (mall_cd, corporation_cd, store_cd).
- **상점 API 접속정보 (1:N)**: `om_store_connection_m` (docs/ddl/om_store_connection_m.sql). connection_id(PK), store_id(FK → om_mall_store_m ON DELETE CASCADE), connection_alias, api_id, api_password, client_id, site_code, redirect_uri, client_secret, scope, created_at, updated_at, created_by, updated_by.
- **적용 순서**: om_mall_m.sql → om_mall_store_m.sql → om_store_connection_m.sql. 샘플/시드 데이터는 docs/dml/ 참고(om_mall_m_data_sample.sql, om_store_connection_m_data_sample.sql 등).

## 8. 최적화·성능 참고 (소스 분석)

- **프론트**
  - 접속정보 조회(`useStoreConnections(storeId)`)는 **패널 오픈 시에만** 호출되며, `enabled: storeId != null && storeId > 0`으로 불필요 호출 없음. 상점 목록 행마다 N+1 요청하지 않음.
  - 목록·옵션·공통코드는 React Query 캐시·staleTime 활용. 접속정보 mutation 성공 시 해당 storeId의 connections 쿼리만 invalidate.
  - 패널 제목·버튼: 긴 상점명 말줄임(`text-truncate`, `min-w-0`), 버튼 줄바꿈 방지(`text-nowrap`, `flex-shrink-0`).
- **백엔드**
  - 상점 목록: `selectMallStoreListCount` + `selectMallStoreList` 2회 호출(페이징 총 건수·목록). 표준 패턴.
  - 접속정보: 단일 `selectByStoreId`로 해당 상점 N건 일괄 조회. connection_id 순 정렬.
  - 서비스 레이어: 단일 테이블 CRUD, 불필요한 루프·할당 없음. 필요 시 `@Transactional` 추가 검토(현재 단일 DML이라 선택 사항).

추가 개선 여지: 목록 화면에서 “총 건수”를 첫 페이지만 조회하거나, 무한 스크롤로 전환 시 count 생략 등은 요구사항에 따라 검토.

## 9. 쿼리·인덱스 점검

### 9.1 현재 인덱스 요약

| 테이블 | 인덱스 | 용도 |
|--------|--------|------|
| om_mall_m | uk_mall_cd, idx_mall_is_active | mall_cd 유일·활성 필터 |
| om_mall_store_m | idx_mall_store_mall_cd, idx_mall_store_corporation_cd, idx_mall_store_is_active, uk_mall_store_corp_store | 조인·법인 필터·활성 필터·유일 |
| om_store_connection_m | PK(connection_id), idx_store_connection_store_id | store_id 조건 조회 |

### 9.2 쿼리별 점검

- **selectMallStoreList / selectMallStoreListCount**  
  - 조건: `corporation_cd = ?`(선택), `mall_cd|mall_nm|store_cd|store_nm` ILIKE `'%keyword%'`.  
  - 정렬: `m.mall_cd, s.store_cd`.  
  - `corporation_cd` 단일 조건 시 `idx_mall_store_corporation_cd` 사용 가능.  
  - **ILIKE '%...%'** 는 B-tree 인덱스로는 한계 있음. 데이터 많을 때만 **pg_trgm** 확장 + GIN 인덱스 검토(예: `store_nm`, `mall_nm`).

- **selectByStoreId (om_store_connection_m)**  
  - `WHERE store_id = ? ORDER BY connection_id`.  
  - **idx_store_connection_store_id** 로 충분. connection_id는 PK라 정렬 비용 낮음.

- **insert/update/delete (om_store_connection_m)**  
  - PK 또는 store_id FK 기준. 현재 인덱스로 충분.

### 9.3 인덱스 추천·변경

- **필수 변경 없음.** 현재 DDL만으로도 목록·접속정보 조회에 무리 없음.
- **선택(대용량 시)**  
  - 상점 목록 키워드 검색이 느려지면: PostgreSQL `pg_trgm` 확장 후 `store_nm`, `store_cd`, `mall_nm`, `mall_cd` 등에 GIN 트라이그램 인덱스 검토.  
  - 법인+키워드 동시 조건이 많다면: 복합 인덱스 `(corporation_cd, mall_cd)` 또는 `(corporation_cd, store_cd)` 후보(실제 실행 계획 확인 후 결정 권장).

적용 시 인덱스 DDL은 `docs/ddl/` 에 추가하고, docs/08-DB-표준.md 네이밍 규칙(idx_테이블_컬럼) 준수.
