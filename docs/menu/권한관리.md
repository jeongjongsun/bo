# 권한관리 메뉴 기능 명세

> 대상 독자: 유지보수 개발자, QA, 운영자
> 목적: 권한 그룹별 메뉴 접근 제어와 감사 로그를 일관되게 구현/운영하기 위한 단일 기준

---

## 1. 개요

- 메뉴명: 권한관리
- 역할: 권한그룹(`om_auth_group_m.auth_group_cd`)별로 접속 가능한 메뉴를 저장한다.
- 사용자 연결: `om_user_m.user_info ->> 'auth_group'` 값과 `om_auth_group_m.auth_group_cd`가 매핑된다.
- 반영 방식: 로그인 시 허용 메뉴만 세션에 저장하고, 프론트는 해당 메뉴만 노출한다.
- 범위: 하나의 권한그룹에서 OM/BO 메뉴 권한을 함께 관리한다.

---

## 2. 화면 구성

### 2.1 권한그룹 목록 그리드

- 컬럼:
  - 번호
  - 권한그룹 코드
  - 권한그룹 명 (편집 가능)
  - 사용여부 (`is_active`)
  - 소속 사용자 수 (`om_user_m`에서 `auth_group` 일치 건수)
  - 비고 (`remark`, 편집 가능)
  - 삭제
  - 등록일, 등록자, 수정일, 수정자
- 삭제 컬럼:
  - badge 형태 `삭제` 버튼
  - 소속 사용자 수가 1 이상이면 2차 확인:
    - `소속 사용자에 부여된 권한은 모두 삭제됩니다.<br>계속하시겠습니까?`
  - 확인 시:
    - 해당 그룹의 메뉴 매핑 삭제
    - `om_user_m.user_info.auth_group = ''` 처리
    - 감사 로그 저장

### 2.2 메뉴 권한 모달

- 권한그룹 코드 클릭 시 모달 오픈
- 탭:
  - OM
  - BO
- 각 탭에서 메뉴 트리/목록 표시 + 체크박스 선택
- 편의 기능 권장:
  - 검색
  - 전체 선택/해제
  - 변경 건수 표시
- 저장 버튼 클릭 시 탭별 메뉴 권한 일괄 저장(Replace)

## 3. DB 저장 모델

## 3.1 권한 매핑 저장

- 테이블: `om_auth_group_menu_r`
- 키:
  - `auth_group_cd`
  - `menu_id`
  - 시스템 필드: `system_main_cd`, `system_sub_cd`
- 방식:
  - 시스템 탭 단위로 기존 매핑을 삭제(soft delete 또는 물리 삭제 정책 선택) 후 신규 일괄 insert
- DDL: `docs/ddl/om_auth_group_menu_r.sql`

## 3.2 감사 로그 저장

- 테이블: `om_auth_group_menu_audit_h`
- 핵심 필드:
  - 누가: `created_by`
  - 언제: `created_at`
  - 무엇을:
    - `action_type`
    - `before_menu_ids`
    - `after_menu_ids`
    - `affected_user_count`
    - `change_reason`
  - 추적:
    - `request_id`
    - `request_ip`
    - `user_agent`
- DDL: `docs/ddl/om_auth_group_menu_audit_h.sql`

## 4. API 계약(초안)

> **베이스 경로**: `AuthGroupManageController`는 `@RequestMapping("/api/v1/auth-groups/manage")` 이므로, 아래 엔드포인트는 모두 `/api/v1/auth-groups/manage` 접두어를 사용한다.

### 4.1 권한그룹 목록 조회

- `GET /api/v1/auth-groups/manage`
- 응답: 그리드 행 목록(사용자 수 포함)

### 4.1b 권한그룹 기본 정보 수정

- `PUT /api/v1/auth-groups/manage/{authGroupCd}`
- 요청 body: `authGroupNm`(필수), `remark`(선택, null 허용)
- 응답: `ApiResponse<Void>`

### 4.2 권한그룹별 메뉴 권한 조회

- `GET /api/v1/auth-groups/manage/{authGroupCd}/menus?systemSubCd=OM|BO&systemMainCd=SYSTEM` (`systemMainCd` 기본값 `SYSTEM`)
- 응답:
  - 시스템 메뉴 목록(트리 구성용 플랫 또는 트리)
  - 해당 그룹의 체크된 메뉴 ID 목록

### 4.3 권한그룹별 메뉴 권한 저장

- `PUT /api/v1/auth-groups/manage/{authGroupCd}/menus`
- 요청 예시:

```json
{
  "systemMainCd": "SYSTEM",
  "systemSubCd": "BO",
  "menuPermissions": [
    {
      "menuId": "basic-users",
      "canView": true,
      "canCreate": true,
      "canUpdate": true,
      "canDelete": false,
      "canExcelDownload": true,
      "canApprove": false
    }
  ],
  "changeReason": "BO 주문 운영 권한 조정"
}
```

- 처리:
  - before 목록 조회
  - replace 저장
  - after 목록 조회
  - 감사 로그 insert (`action_type=UPSERT`)

### 4.4 권한그룹 삭제

- `DELETE /api/v1/auth-groups/manage/{authGroupCd}`
- 처리:
  - 소속 사용자 수 조회
  - 사용자 `auth_group` 초기화
  - 그룹-메뉴 매핑 제거
  - 그룹 삭제(또는 비활성)
  - 감사 로그 insert
    - `action_type=DELETE_GROUP`
    - 필요 시 `action_type=CLEAR_USERS_AUTH_GROUP` 분리 저장

### 4.5 감사 로그 조회

- `GET /api/v1/auth-groups/manage/{authGroupCd}/audits?page=0&size=20` (`size` 상한 500)
- 응답: `ApiResponse<PagedData<AuthGroupMenuAuditRow>>` — 행(`items`) 필드 예:
  - `id`, `actionType`, `systemMainCd`, `systemSubCd`
  - `beforeMenuIds`, `afterMenuIds` (JSONB 문자열)
  - `affectedUserCount`, `changeReason`
  - `requestId`, `requestIp`, `userAgent`
  - `createdAt`, `createdBy`

## 5. 트랜잭션/동시성 기준

- 메뉴 권한 저장/권한그룹 삭제는 Service 계층 `@Transactional` 적용
- 처리 순서:
  1. 입력/존재 검증
  2. 변경 전 데이터 조회
  3. 본 변경 수행
  4. 변경 후 데이터 조회
  5. 감사 로그 저장
- 실패 시 전체 롤백
- 동시 수정 대응(권장):
  - `updated_at` 기반 낙관적 잠금(If-Match 유사 정책) 또는
  - 최근 변경 감지 경고

---

## 6. 보안/인가

- 메뉴 노출 제어만으로 끝내지 않고 API 인가도 반드시 검사
- 401: 미인증
- 403: 인증되었으나 권한 없음
- 감사 로그에는 비밀번호/토큰/민감정보 저장 금지

---

## 7. QA 체크리스트

1. 권한그룹별 OM/BO 탭 저장 후 재로그인 시 메뉴 노출이 정확한가
2. 권한 없는 URL 직접 접근 시 403이 내려오는가
3. 소속 사용자가 있는 그룹 삭제 시 confirm 문구/동작이 정확한가
4. 삭제 후 사용자 `auth_group`이 비워지는가
5. 감사 로그에 누가/언제/무엇(before/after)이 남는가
6. 저장 실패 시 매핑/로그가 부분 반영되지 않고 롤백되는가
