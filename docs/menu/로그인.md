# 로그인 (BO)

백오피스(React + Vite)와 Spring Boot API 간 **세션 기반** 로그인의 정책·처리 순서·데이터 갱신을 정리한다. 구현 기준 경로는 저장소 기준이다.

---

## 1. 범위

| 구분 | 내용 |
|------|------|
| API | `POST /api/v1/auth/login`, `POST /api/v1/auth/logout`, `GET /api/v1/auth/me` |
| 인증 저장소 | 서버 HTTP 세션 (`JSESSIONID`, `credentials: include`) |
| 사용자 데이터 | `om_user_m` + `user_info` JSONB |
| 설정 | `om_config_m.max_password_fail_count` (단일 행 `id=1`) |

---

## 2. 정책 요약

| 항목 | 규칙 |
|------|------|
| 허용 등급 | `user_info.grade_cd` 가 **`ADMIN`** 또는 **`MANAGER`** 인 경우만 로그인 허용 |
| 계정 상태 | `user_info.user_status` 가 **`ACTIVE`** 인 경우만 이후 검증 진행 (`INACTIVE`, `LOCKED` 등은 차단) |
| 삭제 여부 | `om_user_m.is_deleted = false` 인 행만 조회. **삭제됨·미존재**는 DB에서 동일하게 “없음”으로 처리하고, API는 **동일 메시지** (`error.login_failed`, 401) |
| IP 제한 | `user_info.access_ip_limit = 'Y'`(대소문자 무시)일 때만 적용. 클라이언트 IPv4가 `access_ip` 규칙에 포함될 때만 허용. 불가 시 **403**, 코드 `ERR_ACCESS_IP_NOT_ALLOWED`, 메시지 키 `error.access_ip_not_allowed` |
| 비밀번호 | `user_info.password` bcrypt와 평문 비교 |
| 연속 실패 상한 | `om_config_m`에서 `max_password_fail_count` 조회. **null·0 이하·조회 실패 시 기본값 5** |
| 잠금 | 비밀번호 불일치 시 `password_fail_cnt` 를 1 증가. **증가 후 값 ≥ 상한** 이면 동시에 `user_status = LOCKED` 로 갱신. 응답은 `ERR_ACCOUNT_LOCKED` / `error.account_locked` (401) |
| 성공 시 DB | `password_fail_cnt = 0`, `last_login_dtm` = 현재 시각 (**Asia/Seoul**, ISO-8601 문자열) |

---

## 3. 처리 프로세스 (서버)

한 트랜잭션(`AuthService.login` · `@Transactional`) 안에서 다음 순서로 진행한다.

1. **입력**  
   `userId`·`password` 공백만인 경우 → 401, `ERR_UNAUTHORIZED`, `error.login_failed`
2. **`selectByUserIdForUpdate`**  
   `user_id` 일치 **且** `is_deleted = false`. 없으면 → 401, `error.login_failed` (삭제·미존재 동일)
3. **상태**  
   `user_status != ACTIVE` → 403, `ERR_ACCOUNT_NOT_ACTIVE`, `error.account_not_active`
4. **등급**  
   `grade_cd` 가 `ADMIN`/`MANAGER` 가 아님 → 403, `ERR_LOGIN_GRADE_NOT_ALLOWED`, `error.login_grade_not_allowed`
5. **IP** (`access_ip_limit` 이 `Y` 일 때만)  
   `ClientIpAllowlist` 로 IPv4 단일 주소 또는 CIDR 목록 매칭. 실패 시 → 403, `ERR_ACCESS_IP_NOT_ALLOWED`, `error.access_ip_not_allowed`  
   - 규칙 목록은 JSON 배열 또는 콤마 구분 문자열로부터 파싱  
   - **IPv4만** 대상. 규칙이 비어 있으면 허용 IP 없음으로 처리
6. **비밀번호**  
   - 일치: `user_info` merge 후 `updateUserInfoJson` — `password_fail_cnt=0`, `last_login_dtm` 갱신 → 성공  
   - 불일치: `password_fail_cnt` 증가, 상한 도달 시 `LOCKED` 반영 후 `updateUserInfoJson` → 401 및 위 잠금/일반 실패 메시지
7. **JSON 갱신 실패** (직렬화/저장 예외) → 500, `ERR_INTERNAL`, `error.internal`

성공 후 **컨트롤러**에서 세션에 `userId`, `userNm`, `auth_group`(키명은 세션 상수 참고)을 넣고, 응답 바디에 `userId`, `name`, `roles`(현재는 `auth_group` 기반)를 내려준다.

---

## 4. 클라이언트 IP 판별

`AuthController.resolveClientIp`:

- `X-Forwarded-For` 가 있으면 **첫 번째 hop**만 사용 (프록시 뒤 환경 전제).
- 없으면 `request.getRemoteAddr()`.

운영에서는 신뢰할 수 있는 프록시에서만 `X-Forwarded-For` 를 세팅하는 것이 전제다.

---

## 5. API 응답 규격

실패 시 `ApiResponse`: `success=false`, `code`, `message`(i18n 키 문자열). HTTP 상태는 아래와 같이 분기한다.

| HTTP | code | message 키 (예) |
|------|------|-----------------|
| 401 | `ERR_UNAUTHORIZED` | `error.login_failed` (미존재·삭제·비밀번호 불일치·상한 미만 실패) |
| 401 | `ERR_ACCOUNT_LOCKED` | `error.account_locked` (이번 시도로 잠금된 경우) |
| 403 | `ERR_ACCOUNT_NOT_ACTIVE` | `error.account_not_active` |
| 403 | `ERR_LOGIN_GRADE_NOT_ALLOWED` | `error.login_grade_not_allowed` |
| 403 | `ERR_ACCESS_IP_NOT_ALLOWED` | `error.access_ip_not_allowed` |
| 500 | `ERR_INTERNAL` | `error.internal` |

코드 상수는 `ErrorCodes` 참고.

## 7. 프론트엔드

| 항목 | 내용 |
|------|------|
| 화면 | `frontend/src/pages/LoginPage.tsx` |
| 클라이언트 검증 | Bootstrap 5 패턴: `form` 에 `noValidate`, `needs-validation` / 제출 시도 후 `was-validated`, 필드 `required` + `.invalid-feedback` (이메일 `type` 검증 없음, 아이디는 `type="text"`) |
| 서버 오류 표시 | SweetAlert `showError` (`frontend/src/utils/swal.ts`). `getApiErrorMessage` 로 `message`·`code` 기반 번역 |
| 다국어 | `frontend/src/locales/*/translation.json` 의 `login.*`, `error.*` |

CORS 허용 출처는 `WebConfig` 에서 Vite 개발 포트(예: 8081, 5173) 등을 설정한다.

## 9. 운영 체크리스트

1. `om_config_m` DDL 적용 및 `id=1` 행 존재 확인  
2. BO 로그인 대상 사용자의 `user_info` 에 **`grade_cd`**, **`user_status`**, **`password`**, 필요 시 **`access_ip_limit` / `access_ip`** 설정  
3. 리버스 프록시 사용 시 **클라이언트 실 IP** 가 `resolveClientIp` 와 정책에 맞게 전달되는지 확인

## 10. 세션·보호 API

- 로그인 제외 경로: `WebConfig` 에서 `/api/v1/auth/login`, `/api/v1/auth/logout` 은 인터셉터 제외.  
- 그 외 `/api/v1/**` 는 `SessionAuthInterceptor` 로 세션 검사.  
- 프론트 axios는 `withCredentials: true` 로 쿠키 전송.

사전 조건: 백엔드·프론트 기동, DB에 `om_config_m`(선택) 및 로그인용 사용자(`grade_cd`, `user_status`, `password` 등) 준비. 네트워크 탭 또는 API 클라이언트로 `code`·HTTP 상태를 확인할 것.

### 11.1 UI·클라이언트 검증 (로그인 페이지)

| # | 시나리오 | 기대 결과 |
|---|----------|-----------|
| U1 | 아이디·비밀번호 비운 채 **로그인** 클릭 | 제출 후 `was-validated` 적용, 각 필드 아래 `.invalid-feedback` 표시 (브라우저 기본 풍선 없음) |
| U2 | 아이디만 입력 후 제출 | 비밀번호 필드에만 필수 오류 표시 |

### 11.2 정상 로그인·세션

| # | 시나리오 | 기대 결과 |
|---|----------|-----------|
| S1 | `grade_cd` 가 `ADMIN` 또는 `MANAGER`, `user_status=ACTIVE`, 올바른 비밀번호로 로그인 | **200**, `success=true`, 응답에 `userId`·`name`·`roles`. 메인(또는 `/`)으로 이동 |
| S2 | 로그인 직후 동일 브라우저에서 `GET /api/v1/auth/me` | **200**, 세션 기반 사용자 정보 |

### 11.3 미존재·삭제·비밀번호 오류 (동일 메시지 계열)

| # | 시나리오 | 기대 결과 |
|---|----------|-----------|
| A1 | DB에 없는 `user_id`로 로그인 시도 | **401**, `code=ERR_UNAUTHORIZED`, `message=error.login_failed` |
| A2 | `is_deleted=true` 인 계정과 동일한 `user_id`로 시도 | **A1과 동일** (미존재와 구분 불가) |
| A3 | 존재·ACTIVE·등급 OK이나 **비밀번호만 틀림** | **401**, 동일 `error.login_failed`. DB `password_fail_cnt` 가 1 증가 |
| A4 | 연속으로 비밀번호만 틀림 (상한 미만) | 매번 **401** `error.login_failed`, `password_fail_cnt` 누적 |

### 11.4 계정 잠금 (실패 상한)

| # | 시나리오 | 기대 결과 |
|---|----------|-----------|

| L2 | L1 직후 DB `user_info` | `user_status=LOCKED`, `password_fail_cnt` 가 상한 이상으로 반영 |
| L3 | 잠금된 계정으로 올바른 비밀번호 입력 시도 | **403** `ERR_ACCOUNT_NOT_ACTIVE` (비밀번호 검증 전에 상태 차단) |

### 11.6 IP 접근 제한

| # | 시나리오 | 기대 결과 |
|---|----------|-----------|
| I1 | `access_ip_limit=N`(또는 미설정) | 클라이언트 IP와 무관하게 등급·상태·비밀번호만 통과 시 로그인 성공 |
| I2 | `access_ip_limit=Y`, `access_ip` 에 현재 클라이언트 IPv4(또는 포함 CIDR) 설정 | 로그인 성공(그 외 조건 충족 시) |
| I3 | `access_ip_limit=Y`, 허용 목록에 **없는** IP에서 시도 | **403**, `ERR_ACCESS_IP_NOT_ALLOWED`, `error.access_ip_not_allowed` |
| I4 | `access_ip_limit=Y` 이고 `access_ip` 가 **빈 배열/빈 목록** | **I3과 동일** (허용 IP 없음) |
| I5 | (선택) 프록시 뒤에서 `X-Forwarded-For` 첫 값이 허용 목록에 맞는지 | `resolveClientIp` 기준으로 I2/I3 판정이 기대와 일치 |
