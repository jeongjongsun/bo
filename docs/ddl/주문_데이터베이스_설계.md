# 주문 정보 데이터베이스 설계

> **전제**: 일 10만 건 이상 주문 생성, 목록/그리드 조회 시 SELECT 비용 최소화, 다양한 상태 값 및 이력 관리

---

## 1. 설계 원칙

| 원칙 | 적용 |
|------|------|
| **대용량 대비** | 주문 마스터는 `regist_dt`(등록일) 기준 **월 단위 RANGE 파티셔닝**. 파티션 프루닝으로 검색 범위 축소. |
| **좁은 마스터** | 목록/그리드 **검색·정렬·필터에 필요한 컬럼만** 마스터에 두고, 수령인/주문자/메모 등은 **JSONB(order_info)** 한 컬럼으로 관리. SELECT 시 불필요한 컬럼 로딩 방지. |
| **상태 처리** | **현재 상태**만 마스터에 3컬럼(상태코드, 일시, 처리자). **전체 이력**은 **별도 테이블(om_order_status_log)** 에 저장해 마스터 컬럼 수 비대 방지. |

---

## 2. 상태 컬럼: 마스터 vs 분리 테이블 검토

### 2.1 요구사항

- 주문처리상태: 발주, 처리, 출고준비, 발송준비, 배송중, 배송완료 등
- **각 처리 상태별** → **날짜** + **처리자 아이디** 관리 필요

### 2.2 옵션 비교

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A. 마스터에 모두** | process_dt, process_by, ship_ready_dt, ship_ready_by, delivery_ready_dt, ... | 조회 단순 | 컬럼 수 많음(상태 6개 × 2 ≈ 12개+), SELECT 비용·스키마 비대 |
| **B. 상태 이력 테이블** | 마스터에는 **현재 상태 3컬럼**만, 이력은 **om_order_status_log** | 마스터 컬럼 수 최소, 이력/감사 용이, 상태 종류 추가 시 유연 | 목록은 마스터만 조회하면 되고, 상세 이력은 로그 테이블 1회 조회 |

### 2.3 채택 방안: **B. 상태 이력 테이블**

- **마스터(om_order_m)**  
  - `order_process_status` (현재 주문처리상태)  
  - `order_process_status_dt` (해당 상태로 변경된 일시)  
  - `order_process_status_by` (처리자 아이디)  
  → 목록/그리드 필터·정렬은 마스터만 사용.

- **이력(om_order_status_log)**  
  - `order_id`, `status_kind`, `status_value`, `status_dt`, `status_by`  
  - 상태가 바뀔 때마다 INSERT.  
  → 상세 화면·감사·통계는 이력 테이블 조회.  
  - **등록 시**: ETC/REGISTERED 로그는 남기지 않고 ORDER_PROCESS/ORDER_RECEIVED만 기록. (docs/menu/주문관리.md §9.5)

- **추가 상태(출고상태, 주문서 처리 상태 등)**  
  - 같은 이력 테이블에 `status_kind`로 구분해 저장 가능 (예: 'ORDER_PROCESS', 'SHIP_STATUS' 등).  
  - 필요 시 마스터에 “현재값만” 컬럼 3개씩 추가하는 방식도 가능하나, 우선은 **주문처리상태만** 마스터에 두고 나머지는 이력 테이블로 확장 권장.

## 3. 테이블 구성 요약

| 테이블 | 역할 | 파티셔닝 | 비고 |
|--------|------|----------|------|
| **om_order_m** | 주문 마스터 **1주문 1행** | regist_dt 기준 월별 RANGE | order_id(seq_order_id 1회/주문), order_no, corporation_cd, mall_cd, store_cd, order_info(JSONB) 등. 라인 컬럼 없음. |
| **om_order_item_m** | 주문 라인 **1주문 N행**(상품 단위) | 없음 | order_id, regist_dt(FK), line_no, item_order_no, product_cd, product_nm, line_qty, line_amount, line_discount_amount, line_payload |
| **om_order_status_log** | 주문처리상태 등 상태 변경 이력 | status_dt 기준 월별 RANGE(선택) | 현재 상태는 마스터, 이력은 여기 |

- **그리드**: **om_order_m JOIN om_order_item_m** 로 조회. 같은 주문번호(order_no)에 대해 상품코드(product_cd)·상품명(product_nm)·상품별주문번호(item_order_no)를 행 단위로 표시. **1주문 N라인 = N행**.
- **order_id**: 시퀀스(seq_order_id)를 **주문 1건당 1회** 부여. 마스터 1행 INSERT 후, 라인은 **om_order_item_m**에 order_id, regist_dt, line_no 1,2,3... 으로 INSERT.

## 4. 마스터 테이블(om_order_m) 컬럼 — 1주문 1행

| 컬럼 | 타입 | 설명 |
|------|------|------|
| order_id | BIGINT | 주문 PK (seq_order_id 1회/주문) |
| corporation_cd | VARCHAR(20) | 법인코드 |
| mall_cd | VARCHAR(20) | 쇼핑몰코드 |
| store_cd | VARCHAR(50) | 상점코드 |
| order_dt | DATE | 주문일(yyyymmdd) |
| regist_dt | DATE | 등록일(yyyymmdd), **파티션 키·FK 보조** |
| order_no | VARCHAR(100) | 쇼핑몰 주문번호 |
| combined_ship_no | VARCHAR(100) | 합포장 번호(OM에서 동일 배송지 묶음 처리) |
| sales_type_cd | VARCHAR(20) | 판매구분(B2C 국내/해외, B2B, 기타) |
| order_process_status | VARCHAR(30) | 주문처리상태(발주, 처리, 출고준비, 발송준비, 배송중, 배송완료 등) |
| order_process_status_dt | TIMESTAMPTZ | 해당 상태로 변경된 일시 |
| order_process_status_by | VARCHAR(48) | 처리자 아이디 |
| order_type_cd | VARCHAR(20) | 주문타입(일반, 교환, 반품, 취소) |
| order_info | JSONB | 수령인/주문자/메모 등 주문 정보. 키: receiverNm, receiverTel, receiverMobile, receiverAddr, receiverAddr2, receiverZip, ordererNm, ordererUserId, ordererTel, ordererMobile, memo. 암호화 필요 필드는 앱 레이어에서 처리. 그 외 확장 데이터도 동일 JSONB에 저장. |
| created_at, updated_at, created_by, updated_by | | 감사 컬럼 |

- **PK**: (order_id, regist_dt)  
- **유일 제약**: (regist_dt, corporation_cd, mall_cd, store_cd, order_no)  
- **인덱스**: corporation_cd, regist_dt, order_dt, order_process_status, order_id, mall_cd, store_cd, combined_ship_no 등.

---

## 4-1. 라인 테이블(om_order_item_m) 컬럼 — 1주문 N행

| 컬럼 | 타입 | 설명 |
|------|------|------|
| order_id | BIGINT | 주문 PK (FK → om_order_m) |
| regist_dt | DATE | 등록일 (FK → om_order_m, 파티션 보조) |
| line_no | INTEGER | 라인 번호 (주문 내 순번, PK 구성) |
| item_order_no | VARCHAR(100) | 상품별 주문번호 (쇼핑몰에서 생성) |
| product_cd | VARCHAR(50) | 상품코드 |
| product_nm | VARCHAR(200) | 상품명 |
| line_qty | INTEGER | 라인 구매 수량 |
| line_amount | NUMERIC(15,2) | 라인 구매 금액 |
| line_discount_amount | NUMERIC(15,2) | 라인 할인 금액 |
| line_payload | JSONB | 라인별 옵션·쇼핑몰 원본 등 확장용 |
| created_at, updated_at, created_by, updated_by | | 감사 컬럼 |

- **PK**: (order_id, line_no)  
- **FK**: (order_id, regist_dt) → om_order_m(order_id, regist_dt) ON DELETE CASCADE  
- **인덱스**: (order_id, regist_dt), product_cd 등.

## 5. order_info JSONB 구조

**om_order_m.order_info** 에 수령인/주문자/메모 및 확장 데이터를 저장.

- **필수/권장 키**: receiverNm, receiverTel, receiverMobile(수령인 휴대전화), receiverAddr, receiverAddr2, receiverZip, ordererNm, ordererUserId(주문자 쇼핑몰 아이디), ordererTel, ordererMobile(주문자 휴대전화), memo.
- **암호화 필요**: receiverTel, receiverMobile, ordererTel, ordererMobile 는 앱 레이어에서 암호화 후 저장 권장.
- **확장**: 그 외 mall_order_id, total_qty 등 주문 단위 확장 데이터도 동일 JSONB에 키로 추가 가능.

**om_order_item_m.line_payload** 는 라인별 옵션·쇼핑몰 원본 등 확장 정보만 저장.

```json
{ "option_nm": "색상:빨강", "mall_product_cd": "MALL-P-001", "mall_sku_id": "SKU-001" }
```
