# 액션 권한 적용 표준

## 목적

- 메뉴 접근 권한(`view`)과 화면 액션 권한(`create`, `update`, `delete`, `excel_download`, `approve`)을 분리해 일관되게 적용한다.
- BO/OM 모두 동일한 권한 코드 규칙을 사용한다.

## 권한 코드 규칙

- 기본 형식: `{resource}:{action}`
- action 집합: `view`, `create`, `update`, `delete`, `excel_download`, `approve`
- resource 기본 규칙: `menu_id`의 `-`를 `_`로 치환
  - 예: `basic-shipper` -> `basic_shipper`
- 예외 매핑:
  - `basic-users` -> `user`

## 프론트엔드 적용 규칙

- 권한 확인 훅:
  - `useHasActionPermission(permissionCode)`
  - `useHasMenuActionPermissionByPath(pathname, action)`
  - `useHasMenuActionPermissionByMenuId(menuId, action)`
- 버튼/링크/셀 편집 모두 권한을 적용한다.
  - 생성 버튼: `create`
  - 수정 모달 링크, 그리드 editable, 저장 버튼: `update`
  - 삭제 버튼: `delete`
  - 엑셀 다운로드 버튼: `excel_download`
  - 승인/상태전이/일괄처리: `approve`
- 권한 없음 처리:
  - 기본은 "노출하지 않음" 또는 `disabled`.
  - 데이터 변경 이벤트(예: `onCellValueChanged`)에서도 한 번 더 방어한다.

## 백엔드 적용 규칙

- `GET /api/v1/auth/me`의 `permissions`는 권한 코드 목록을 내려준다.
- UI 제어만으로 끝내지 않고, CUD/엑셀/승인 API는 서버에서도 동일 권한을 검증한다.
- BO 전용 API라도 향후 OM 확장을 위해 권한 코드는 공통 규칙으로 저장/조회한다.

## 신규 개발 체크리스트

- 새 화면 경로가 `boAppRoutes`에 등록되어 있는가
- 화면의 모든 버튼/링크/편집 포인트에 액션 권한이 적용되어 있는가
- 해당 API가 서버에서 동일 액션 권한을 검증하는가
- 권한명 예외 매핑이 필요한 경우 문서에 추가했는가
