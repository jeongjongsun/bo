# 개발 표준 (Development Standards)

Spring Boot + React 통합 시스템(Shopeasy4 고도화)을 위한 개발 표준 문서입니다.  
백엔드·프론트엔드 동일 도메인/서버, PostgreSQL, 세션 기반 인증을 전제로 합니다.

---

## 아키텍처: 백엔드와 프론트엔드, “두 프레임워크”에 대해

### 왜 Spring Boot + React로 문서를 썼는가

- **기존 시스템(Shopeasy4)**이 이미 **Spring Boot + JSP/Thymeleaf**이기 때문에, “백엔드는 그대로 두고 화면만 React로 전환”하는 구성을 전제로 표준을 작성했습니다.  
  따라서 **프레임워크를 임의로 2개로 나눈 것이 아니라**, 현재 도입/고도화 방향에 맞춘 것입니다.

### “백엔드도 React 기반”으로 할 수 있나

- **React는 브라우저에서 동작하는 UI 라이브러리**입니다. DB 접속, 세션, HTTP API 서버 역할은 할 수 없습니다.  
- **백엔드까지 JavaScript/TypeScript로 통일**하고 싶다면, 다음 같은 선택이 가능합니다.
  - **Node.js(Express, Nest 등)** 로 API 서버를 구현 → 프론트(React)와 같은 언어/생태계 사용.
  - **Next.js** 같은 React 기반 풀스택 프레임워크 사용 → API 라우트 + 서버 렌더링까지 한 프로젝트에서 처리, **배포도 하나의 앱**으로 가능.

즉, “백엔드를 React로”가 아니라 **“백엔드를 JS/TS 기반(Node 또는 Next.js)으로”** 바꾸는 선택입니다.

### 두 스택으로 나누면 개발/배포가 불편한가

- **배포**는 반드시 2개로 나눌 필요는 없습니다.  
  **동일 도메인/서버**를 쓰는 경우, **Spring Boot 하나**가 다음을 모두 담당할 수 있습니다.
  - `/api/**` → API 처리 (기존 Spring Boot)
  - 그 외 → React 빌드 결과(정적 파일) 서빙  
  → **실제 운영 시 서버/배포 단위는 1개**로 할 수 있습니다.

- **개발** 측면에서는 다음이 “두 스택”의 비용입니다.
  - 백엔드(Java)와 프론트(JS/TS)를 각각 다뤄야 함.
  - 로컬에서 백엔드 실행 + 프론트 dev 서버 등, 스크립트/도구가 둘로 나뉠 수 있음.

현재 문서의 API·에러·페이징·세션 규칙은 **백엔드 구현체가 Spring Boot가 아니어도**(예: Node, Next API 라우트) 그대로 적용 가능합니다. 백엔드를 Node/Next로 옮기면 **언어·빌드·배포를 하나로** 맞출 수 있고, Spring Boot를 유지하면 **기존 자산·운영 방식을 그대로** 활용할 수 있습니다.

---

## React 입문자를 위한 안내

이 프로젝트의 **React 개발자는 입문자**를 전제로 합니다. 아래 원칙을 지키면 **찾기 쉽고, 수정하기 쉬운** 코드가 됩니다.

- **일관성**: 같은 일은 항상 같은 방식·같은 위치에서 처리한다.
- **단순함**: 한 파일·한 컴포넌트는 한 가지 역할만 한다. 고급 패턴은 필요할 때만 도입한다.
- **이름으로 찾기**: 파일명·폴더명·함수명만 봐도 역할을 알 수 있게 짓는다.
- **복사해서 쓰기**: 부록(타입, API 호출, 훅 예시)을 그대로 복사해 조금만 바꿔서 재사용한다.

문서에 나오는 **React Query**, **훅**, **라우팅** 등이 낯설다면 해당 절을 먼저 읽고, 부록의 코드를 참고해 동일한 패턴으로 구현하면 됩니다.

## 0. 프로젝트 구조 및 폴더 규칙

**목적**: "이 코드는 어디 있을까?"를 최소한의 학습으로 예측 가능하게 만듭니다.

### 0.1 권장 폴더 구조 (단순·일관)

```
src/
├── api/              # API 호출 함수만 (axios 호출, URL, 파라미터)
├── components/       # 여러 화면에서 쓰는 공통 UI (버튼, 모달, 레이아웃)
├── features/         # 화면/기능 단위 (사용자, 주문, 대시보드 등)
│   └── users/
│       ├── UserList.tsx      # 목록 화면
│       ├── UserDetail.tsx    # 상세 화면
│       ├── hooks.ts          # useUserList, useUserDetail 등
│       └── types.ts          # User, UserListParams 등 해당 기능만의 타입
├── hooks/            # 전역 공용 훅 (useAuth, usePagination 등)
├── pages/            # 라우트와 1:1 매칭되는 페이지 (필요 시 features 조합)
├── types/            # API 공통 타입 (ApiResponse, PagedData, ApiError)
└── utils/            # 공통 유틸 (날짜 포맷, 검증 등)
```

### 0.2 "어디에 뭘 넣을지" 한 줄 요약

| 넣을 것 | 넣는 곳 | 이유 |
|--------|---------|------|
| 서버 요청 함수 (GET/POST 등) | `api/` 또는 `features/xxx` 내 | API 주소·파라미터가 한곳에 모여 유지보수 쉬움 |
| useQuery / useMutation 사용 코드 | `features/xxx/hooks.ts` 또는 `hooks/` | 데이터 가져오기/변경 로직을 화면과 분리 |
| 한 화면에서만 쓰는 UI | 해당 `features/xxx/` 안 | 화면 단위로 묶여 있어 찾기 쉬움 |
| 두 개 이상 화면에서 쓰는 UI | `components/` | 재사용·일관성 |
| API 응답/요청 공통 형태 | `types/` | 표준 문서와 동일한 이름 사용 권장 |

### 0.3 네이밍 규칙 (찾기 쉽게)

- **파일명**: PascalCase (컴포넌트), camelCase (hooks, utils). 예: `UserList.tsx`, `useUserList.ts`
- **폴더명**: 소문자 또는 camelCase. 예: `features/users`, `api`
- **컴포넌트/훅 이름**: 역할이 드러나게. 예: `UserList`, `useUserList`, `useAuthMe`

## 1. API 응답 포맷 표준

### 1.1 공통 응답 래퍼 (단일/목록 공통)

**성공 응답**

```json
{
  "success": true,
  "data": { ... },
  "message": null,
  "code": "SUCCESS"
}
```

**필드 정의**

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| success | boolean | O | 요청 성공 여부 |
| data | object/array | 조건부 | 성공 시 응답 본문 (에러 시 null) |
| message | string | X | 사용자용 메시지 (에러/성공 모두) |
| code | string | O | 결과 코드 (SUCCESS, 에러코드 등) |

### 1.2 목록(페이징) 응답 포맷

목록 API는 `data` 안에 아래 구조를 사용합니다.

```json
{
  "success": true,
  "data": {
    "items": [ ... ],
    "page": 0,
    "size": 20,
    "total": 100,
    "totalPages": 5,
    "first": true,
    "last": false,
    "sort": [
      { "property": "createdAt", "direction": "DESC" }
    ]
  },
  "message": null,
  "code": "SUCCESS"
}
```

**필드 정의**

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| items | array | O | 현재 페이지 데이터 목록 |
| page | number | O | 0-based 페이지 번호 |
| size | number | O | 페이지 크기 |
| total | number | O | 전체 레코드 수 |
| totalPages | number | O | 전체 페이지 수 |
| first | boolean | O | 첫 페이지 여부 |
| last | boolean | O | 마지막 페이지 여부 |
| sort | array | X | 정렬 정보 (property, direction: ASC/DESC) |

### 1.3 요청 규칙 (목록 API)

- **쿼리 파라미터**
  - `page`: 0-based (기본값: 0)
  - `size`: 1~100 (기본값: 20, 최대 100)
  - `sort`: `property,direction` (예: `createdAt,DESC`), 복수일 경우 `sort=name,ASC&sort=id,DESC`

## 2. 에러 포맷, 페이징 포맷, 코드/메시지 규칙

### 2.1 에러 응답 포맷

```json
{
  "success": false,
  "data": null,
  "message": "사용자에게 보여줄 한글 메시지",
  "code": "ERR_BAD_REQUEST",
  "errors": [
    {
      "field": "email",
      "message": "올바른 이메일 형식이 아닙니다.",
      "rejectedValue": "invalid"
    }
  ],
  "timestamp": "2025-02-20T10:00:00Z",
  "path": "/api/v1/users"
}
```

**필드 정의**

| 필드 | 타입 | 설명 |
|------|------|------|
| success | boolean | 항상 false |
| data | null | 에러 시 null |
| message | string | 클라이언트 표시용 메시지 |
| code | string | 에러 코드 (표준 코드 사용) |
| errors | array | 필드별 검증 오류 (선택) |
| timestamp | string | ISO 8601 |
| path | string | 요청 경로 |

### 2.2 표준 에러 코드 규칙

- **형식**: `ERR_` 접두어 + 대문자 스네이크 케이스
- **HTTP 상태 매핑**

| HTTP Status | 코드 예시 | 용도 |
|-------------|-----------|------|
| 400 | ERR_BAD_REQUEST | 잘못된 요청/파라미터 |
| 401 | ERR_UNAUTHORIZED | 미인증(로그인 필요) |
| 403 | ERR_FORBIDDEN | 권한 없음 |
| 404 | ERR_NOT_FOUND | 리소스 없음 |
| 409 | ERR_CONFLICT | 비즈니스 충돌(중복 등) |
| 422 | ERR_VALIDATION | 검증 실패 (errors 배열 사용) |
| 500 | ERR_INTERNAL | 서버 내부 오류 |

### 2.3 코드/메시지 규칙

- **코드**: 영문 상수, 클라이언트 분기/로깅용
- **message**: 한글, 사용자 노출용 (다국어 시 메시지 키만 전달하고 클라이언트에서 치환 가능)
- **에러 메시지**: 비개발자도 이해 가능한 문구 사용

### 2.4 페이징 규칙 (요청)

- `page`: 0-based 일관 사용
- `size`: 기본 20, 최대 100 (초과 시 100으로 제한 또는 400 에러)
- `sort`: 백엔드 허용 필드만 적용, 기본 정렬 명시 (예: `id,DESC`)

## 3. 권한/메뉴 제공방식

### 3.1 전제

- 인증은 **세션 기반** (동일 도메인이므로 쿠키 기반 JSESSIONID 유지)
- React는 SPA이므로 **메뉴/권한은 API로 제공**하고, 초기 로딩 또는 로그인 후 1회 조회

### 3.2 권한/메뉴 API

**권장 엔드포인트**

- `GET /api/v1/auth/me`  
  - 현재 로그인 사용자 정보 + **메뉴 트리** + **권한 코드 목록** 포함

**응답 예시**

```json
{
  "success": true,
  "data": {
    "user": {
      "userId": "user01",
      "name": "홍길동",
      "roles": ["USER", "ADMIN"]
    },
    "menus": [
      {
        "id": "M001",
        "name": "대시보드",
        "path": "/dashboard",
        "icon": "Dashboard",
        "children": []
      },
      {
        "id": "M002",
        "name": "관리",
        "path": null,
        "icon": "Settings",
        "children": [
          { "id": "M002-01", "name": "사용자", "path": "/admin/users", "icon": "User", "children": [] }
        ]
      }
    ],
    "permissions": ["USER_READ", "USER_WRITE", "ORDER_READ"]
  },
  "message": null,
  "code": "SUCCESS"
}
```

### 3.3 메뉴/권한 사용 규칙

- **메뉴**: 서버에서 내려준 `menus` 기준으로 사이드바/탑메뉴 렌더링 (역할별 메뉴 자동 반영)
- **권한(permissions)**: 버튼/기능 노출·비노출, API 호출 전 체크용
- **라우팅**: React에서 인증 여부는 `/api/v1/auth/me` 성공 여부로 판단, 메뉴에 없는 path는 권한 없음으로 처리 가능

### 3.4 동일 도메인 고려사항

- API 요청 시 `credentials: 'include'` 사용하여 세션 쿠키 전송
- CORS는 same-origin이므로 별도 설정 최소화 (필요 시 `SameSite` 등 쿠키 정책만 정리)

## 4. 권한 만료/세션 만료 시 규칙

### 4.1 백엔드 동작

- 세션 만료/미인증 요청 시: **HTTP 401** + 본문에 표준 에러 포맷
- 권한 없음(인증됐으나 권한 부족): **HTTP 403** + 표준 에러 포맷

```json
{
  "success": false,
  "data": null,
  "message": "세션이 만료되었습니다. 다시 로그인해 주세요.",
  "code": "ERR_SESSION_EXPIRED",
  "errors": null,
  "timestamp": "2025-02-20T10:00:00Z",
  "path": "/api/v1/orders"
}
```

**권장 코드**

- `ERR_SESSION_EXPIRED` 또는 `ERR_UNAUTHORIZED`: 401
- `ERR_FORBIDDEN`: 403

### 4.2 프론트엔드 동작 (React)

1. **전역 HTTP 인터셉터(axios 등)**  
   - 401 수신 시: 저장된 사용자/메뉴/권한 정보 초기화 → 로그인 페이지로 리다이렉트 (또는 모달 후 로그인 페이지)  
   - 403 수신 시: “권한이 없습니다” 메시지 표시, 필요 시 이전 페이지 또는 대시보드로 이동

2. **리다이렉트 주소**  
   - 로그인 페이지: 기존 Shopeasy4 로그인 URL 사용 (동일 도메인)  
   - 예: `/login` 또는 `/sso/login` (실제 경로에 맞게 정의)

3. **재시도**  
   - 401/403 발생 요청은 자동 재시도하지 않음 (한 번 로그인 페이지로 이동)

4. **동시 요청**  
   - 여러 API가 동시에 401을 반환해도 로그인 페이지로 한 번만 이동하도록 플래그/큐 사용 권장

## 5. React Query 표준

### 5.1 사용 라이브러리

- **@tanstack/react-query** (React Query v5) 사용
- 동일 도메인 + 세션 기반이므로 별도 토큰 관리 불필요, `credentials: 'include'`만 보장

### 5.2 쿼리 키(Query Key) 규칙

- **배열 기반**, 계층 구조로 관리
- 규칙: `[도메인, 식별자?, 쿼리 파라미터?]`

**예시**

```ts
// 목록 (페이징/필터 포함)
['users', 'list', { page: 0, size: 20, sort: 'createdAt,DESC' }]
['orders', 'list', { page, size, status }]

// 단건
['users', 'detail', userId]
['orders', 'detail', orderId]

// 권한/메뉴 (전역 1회)
['auth', 'me']
```

### 5.3 목록(페이징) 쿼리 표준

- **useQuery** 사용
- **queryKey**: 위와 같이 `['도메인', 'list', { page, size, sort, ...필터 }]`
- **queryFn**: API 호출 후 `data.data` (items, page, size, total 등) 반환
- **keepPreviousData** (또는 v5의 유사 옵션): 페이지 전환 시 이전 데이터 유지로 UX 개선

```ts
// 예시 (타입/경로는 프로젝트에 맞게 조정)
useQuery({
  queryKey: ['users', 'list', { page, size, sort }],
  queryFn: () => fetchUserList({ page, size, sort }),
  keepPreviousData: true,
});
```

### 5.4 뮤테이션(Mutation) 표준

- **useMutation** 사용
- **onSuccess**에서:
  - 목록 무효화: `queryClient.invalidateQueries({ queryKey: ['도메인', 'list'] })`
  - 단건 무효화: `queryClient.invalidateQueries({ queryKey: ['도메인', 'detail', id] })`
- 필요 시 **optimistic update** 적용 (동일 도메인이라 응답 빠를 경우 선택)

### 5.5 전역 설정 권장

- **staleTime**: 목록 1~2분, 단건 2~5분, `auth/me` 5분 등 (세션 유효 시간과 조율)
- **retry**: 401/403은 재시도하지 않도록 **retry** 함수에서 status 확인 후 false 반환
- **refetchOnWindowFocus**: 목록은 true, `auth/me`는 true (세션 유효성 확인용)

### 5.6 API 레이어 규칙

- **baseURL**: 동일 도메인이므로 상대 경로 `/api/v1` 또는 환경변수
- **공통 헤더**: `Content-Type: application/json`
- **credentials**: `include` (세션 쿠키 전송)
- **에러 처리**: queryFn 내에서 4xx/5xx 시 표준 에러 포맷 파싱 후 throw (Error 또는 커스텀 타입), React Query가 onError 등으로 처리

### 5.7 타입 공유

- 응답 포맷(공통 래퍼, 페이징, 에러)은 TypeScript 타입으로 정의하고 백엔드와 가능하면 공유 (모노레포 또는 별도 패키지)
- 예: `ApiResponse<T>`, `PagedResponse<T>`, `ApiError`

### 5.8 React Query 입문자용 요약

| 하고 싶은 일 | 사용할 것 | 데이터 갱신 시 |
|-------------|-----------|----------------|
| 목록 조회 (페이징) | `useQuery` + 쿼리키 `['도메인', 'list', params]` | 자동(캐시 만료 시) 또는 invalidate |
| 단건 조회 | `useQuery` + 쿼리키 `['도메인', 'detail', id]` | 동일 |
| 저장/수정/삭제 | `useMutation` | onSuccess에서 `invalidateQueries`로 해당 목록/단건 무효화 |

- **queryFn**에서는 API 호출 후 **`response.data`**(우리 표준의 `data` 필드)만 반환하도록 하면, 화면에서는 `data.items`, `data.total` 등만 쓰면 됩니다.
- 복잡한 옵션(optimistic update 등)은 나중에 필요할 때 도입해도 됩니다. 먼저 위 패턴만 통일하는 것을 권장합니다.

## 6. React 개발 시 유지보수 규칙

입문자도 **수정·추가 시 흐름을 따라가기 쉽게** 하기 위한 규칙입니다.

### 6.1 한 파일은 한 역할

- 한 파일에 너무 많은 컴포넌트나 로직을 넣지 않습니다. 200~300줄 넘어가면 "목록/상세/폼" 등으로 파일을 나누는 것을 검토합니다.
- API 호출은 **훅(hooks)** 또는 **api/** 폴더의 함수로 두고, 화면(TSX)에서는 "데이터 가져오기/전송" 결과만 사용합니다.

### 6.2 화면(TSX)에서는 "표시"와 "사용자 동작"만

- **표시**: `data?.items`, `isLoading`, `error?.message` 등으로 로딩/에러/목록만 렌더링.
- **사용자 동작**: 버튼 클릭 시 `mutate(...)` 호출, 페이지 변경 시 `setPage` 등 상태만 변경. 실제 API 호출은 훅/API 레이어에 두세요.
- 이렇게 하면 "화면 수정"과 "API/비즈니스 로직 수정"을 나눠서 할 수 있어 유지보수가 쉬워집니다.

### 6.3 반복되는 패턴은 훅/공통 컴포넌트로

- 같은 형태의 목록+페이징이 여러 화면에 있으면 `usePagination` 또는 공통 `ListPage` 컴포넌트를 두고 재사용합니다.
- 에러 메시지 표시, 로딩 스피너도 한 곳에서 정의해 두면 나중에 디자인 변경 시 한 번만 수정하면 됩니다.

### 6.4 타입은 꼭 두기 (any 최소화)

- API 응답은 `types/`의 `ApiResponse<T>`, `PagedData<T>` 등을 사용해 타입을 지정합니다. 자동 완성과 오타 방지에 도움이 됩니다.
- 부록에 있는 타입을 복사해 두고, 도메인별로 `User`, `Order` 등만 추가해 나가면 됩니다.

### 6.5 주석은 "왜" 위주로

- "무엇을 하는 코드인지"는 이름으로 드러나게 하고, **이상한 비즈니스 규칙이나 예외 처리 이유**만 주석으로 남깁니다. 나중에 같은 곳을 수정할 사람이 이해하기 쉽습니다.

## 7. 새 화면/기능 추가 시 체크리스트

새 메뉴/화면을 추가할 때 아래 순서로 진행하면 표준을 지키기 쉽습니다.

1. **라우트**: 해당 path를 라우터에 추가하고, 필요 시 메뉴(`auth/me`의 menus)에 반영되었는지 확인.
2. **폴더**: `features/기능명/` 생성 후 `XXXList.tsx` 또는 `XXXDetail.tsx` 등 역할이 드러나는 파일명 사용.
3. **API**: 목록/단건/저장 등 필요한 API를 `api/` 또는 `features/기능명/` 안에 함수로 정의. 응답은 `ApiResponse`/`PagedData` 타입 사용.
4. **훅**: `useQuery`/`useMutation`은 `features/기능명/hooks.ts`에 정의. 쿼리키는 `['도메인', 'list'|'detail', params]` 규칙 준수.
5. **화면**: TSX에서는 훅 결과(`data`, `isLoading`, `mutate`)만 사용해 UI 구성. 에러 메시지는 `error?.response?.data?.message` 또는 표준 필드 사용.
6. **권한**: 버튼/메뉴 노출은 `auth/me`의 `permissions`로 제어. 403 시 표준대로 "권한이 없습니다" 처리.

### 7.1 자주 하는 실수와 권장 사항

| 실수 | 권장 |
|------|------|
| API 호출을 화면 컴포넌트 안에 직접 작성 | `api/` 함수 + `useQuery`/`useMutation`으로 분리해 두면 재사용·테스트가 쉬움 |
| 쿼리키를 문자열 하나로만 사용 (`['users']`) | 목록/단건/필터가 바뀌어도 캐시가 구분되도록 `['users','list',{page,size}]` 형태 사용 |
| mutation 성공 후 목록을 다시 안 불러옴 | `onSuccess`에서 `invalidateQueries({ queryKey: ['도메인', 'list'] })` 호출 |
| 에러 메시지를 하드코딩 | 서버가 주는 `message` 또는 `errors[].message` 사용 (표준 에러 포맷) |
| 컴포넌트가 너무 커져서 수정이 두려움 | "한 파일 한 역할" 원칙으로 잘라 두고, 공통 부분은 hooks/components로 올리기 |

## 8. 정리 체크리스트

| 구분 | 표준 요약 |
|------|-----------|
| **API 응답** | success, data, message, code 래퍼; 목록은 data.items, page, size, total, totalPages, first, last, sort |
| **페이징** | page(0-based), size(기본 20, 최대 100), sort(property,direction) |
| **에러** | 동일 래퍼 + code, message, errors[], timestamp, path; 코드는 ERR_ 접두어 |
| **권한/메뉴** | GET /api/v1/auth/me 로 user, menus, permissions 제공 |
| **세션 만료** | 401 + ERR_SESSION_EXPIRED → 로그인 페이지 이동, 403 → 권한 없음 메시지 |
| **React Query** | 쿼리키 [도메인, 타입, params], credentials include, 401/403 no retry, invalidation 규칙 |
| **유지보수** | 폴더/파일 역할 일관, API·훅·화면 분리, 타입 사용, 새 기능은 체크리스트 순서로 |
