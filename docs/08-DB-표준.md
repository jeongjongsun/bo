## 1. 원칙

- **일관성**: 테이블·컬럼·인덱스 이름을 통일된 규칙으로 짓는다. 팀 전체가 같은 규칙을 따르면 스키마 이해와 쿼리 작성이 쉬워진다.
- **가독성**: 이름만 봐도 역할과 관계를 알 수 있게 한다. 약어는 팀에서 정의한 것만 사용한다.
- **변경 가능성**: 스키마 변경은 **마이그레이션 스크립트**로만 수행한다. 직접 DDL을 실행하지 않고, 스크립트를 버전 관리·검토 후 적용한다.

## 2. 네이밍 규칙

### 2.1 테이블

| 규칙 | 설명 | 예시 |
|------|------|------|
| **단수형** | 테이블명은 단수 명사. (한 행이 하나의 엔티티를 나타냄) | `user`, `order`, `order_item` |
| **소문자** | 모두 소문자 사용. | `user`, `product_category` |
| **단어 구분** | 여러 단어는 **언더스코어(_)** 로 연결. | `order_item`, `product_category` |
| **도메인 접두사** | 필요 시 도메인 구분용 접두사 사용 (팀 정책). | `biz_order`, `sys_config` — 선택 사항 |

- **JPA/Spring Data** 사용 시 엔티티 클래스명(PascalCase)과 테이블명을 `@Table(name = "order")` 등으로 매핑하면, 코딩 표준의 클래스 네이밍과 자연스럽게 대응된다.

### 2.2 컬럼

| 규칙 | 설명 | 예시 |
|------|------|------|
| **snake_case** | 소문자 + 언더스코어. | `user_id`, `created_at`, `order_status` |
| **의미 있는 이름** | 역할이 드러나게. 약어는 팀 정의만 사용. | `user_id`, `phone_no`, `is_active` |
| **불리언** | `is_~`, `has_~`, `can_~` 접두어 권장. | `is_active`, `is_deleted` |

### 2.3 기본키(PK)

- **컬럼명**: `id` 단일 컬럼을 사용하는 것을 기본으로 한다.
- **타입**: 자동 증가 정수(`BIGSERIAL`) 또는 **UUID** 중 팀에서 하나로 통일.  
  - 분산·외부 노출이 많으면 UUID, 단순·조회 성능 우선이면 BIGSERIAL.
- **복합 PK**가 필요한 경우만 컬럼명을 `테이블1_id`, `테이블2_id` 등으로 구체적으로 짓는다.

### 2.4 외래키(FK)

- **컬럼명**: `참조하는_테이블명_id` 형태. (단수 테이블명 + `_id`)  
  예: `user_id`, `order_id`, `product_id`
- **동일 테이블 참조**가 여러 개일 때**: 역할을 붙인다.  
  예: `created_by_user_id`, `approved_by_user_id`

### 2.5 인덱스

- **이름**: `idx_테이블명_컬럼1_컬럼2` 또는 `idx_테이블_용도`.  
  예: `idx_order_user_id`, `idx_order_created_at`, `idx_order_item_order_id`
- **유니크 인덱스**: `uk_테이블명_컬럼1_컬럼2`.  
  예: `uk_user_login_id`

### 2.6 제약조건

- **PK**: `pk_테이블명` — 예: `pk_order`
- **FK**: `fk_테이블명_참조테이블명` — 예: `fk_order_user`
- **Unique**: `uk_테이블명_컬럼` — 예: `uk_user_login_id`
- **Check**: `chk_테이블명_설명` — 예: `chk_order_amount_positive`

## 3. 공통 컬럼

**대부분의 비즈니스 테이블**에 아래 컬럼을 두어 감사(audit)·정렬·소프트 삭제를 일관되게 다룹니다.

| 컬럼명 | 타입 | 필수 | 설명 |
|--------|------|------|------|
| **id** | BIGINT 또는 UUID | O | PK (BIGSERIAL 또는 기본값 gen_random_uuid()) |
| **created_at** | TIMESTAMPTZ | O | 최초 생성 시각. 기본값 `CURRENT_TIMESTAMP` |
| **updated_at** | TIMESTAMPTZ | O | 최종 수정 시각. 갱신 시 매번 업데이트 |
| **created_by** | VARCHAR 또는 BIGINT | X | 생성자 식별자 (user_id 등). 감사용 |
| **updated_by** | VARCHAR 또는 BIGINT | X | 수정자 식별자. 감사용 |
| **is_deleted** | BOOLEAN | X | 소프트 삭제 시 true. 기본값 false. 삭제 정책이 있을 때만 사용 |

- **타임존**: 본 시스템은 **한국 시간(Asia/Seoul, UTC+9)** 으로 통일한다. 백엔드 JVM 기본 타임존과 PostgreSQL JDBC 세션 `TimeZone`을 `Asia/Seoul`로 설정하여, `CURRENT_TIMESTAMP`, Java `new Date()`/`LocalDateTime.now()` 등 모든 시각이 한국 시간으로 저장·해석된다. DB 컬럼은 `TIMESTAMPTZ` 사용 권장.
- **created_by / updated_by**: FK로 `user` 테이블을 참조하거나, 로그인 ID 문자열을 넣을 수 있다. (팀 정책)
- **소프트 삭제**: 물리 삭제 대신 `is_deleted = true`로 두고, 조회 시 `WHERE is_deleted = false` 조건을 기본으로 적용한다. 인덱스에 `is_deleted`를 포함해 성능을 고려한다.

### 3.1 시스템 타임존(한국 시간)

- **정책**: 모든 시각은 **한국 시간(Asia/Seoul, UTC+9)** 으로 저장·처리한다.
- **백엔드**: `TimeZoneConfig`에서 JVM 기본 타임존을 `Asia/Seoul`로 설정. `application.yml`(및 프로필별) datasource URL에 `options=-c TimeZone=Asia/Seoul` 로 PostgreSQL 세션 타임존 지정.
- **효과**: SQL `CURRENT_TIMESTAMP`, Java `new Date()`·`LocalDateTime.now()`·`Instant` 변환, DB 조회 시 반환 시각이 모두 한국 시간 기준으로 동작한다.

## 4. 데이터 타입

PostgreSQL 기준으로 **자주 쓰는 타입**을 통일해 둡니다.

| 용도 | 타입 | 비고 |
|------|------|------|
| **PK (자동 증가)** | BIGSERIAL | 8바이트 정수, 대용량에도 안전 |
| **PK (UUID)** | UUID | 기본값 `gen_random_uuid()` |
| **외래키** | BIGINT 또는 UUID | 참조하는 PK 타입과 동일 |
| **일시** | TIMESTAMPTZ | 타임존 포함 |
| **날짜만** | DATE | 시간 불필요할 때 |
| **문자열 (고정/짧음)** | VARCHAR(n) | n은 최대 길이. 코드·로그인 ID 등 |
| **문자열 (길이 가변)** | TEXT 또는 VARCHAR(n) | PostgreSQL에서는 TEXT도 성능 차이 거의 없음. n 상한이 있으면 VARCHAR(n) |
| **불리언** | BOOLEAN | |
| **정수** | INTEGER 또는 BIGINT | 크기에 따라 선택 |
| **금액·정밀 수** | NUMERIC(p,s) | 반올림 오류 방지. 예: NUMERIC(15,2) |
| **JSON** | JSONB | 검색·인덱스 필요 시 JSONB 권장 |

- **문자열 길이**: 개발 표준의 API·검증 규칙과 맞춘다. (예: 로그인 ID 50자, 이름 100자 등)
- **ENUM**: PostgreSQL 네이티브 ENUM 또는 **VARCHAR + check 제약** 중 팀에서 선택. VARCHAR이면 마이그레이션·확장이 쉽다.

## 5. 제약조건·기본값

### 5.1 NOT NULL

- **필수 비즈니스 데이터**는 NOT NULL로 정의한다. 선택 항목만 NULL 허용.
- **공통 컬럼**: `id`, `created_at`, `updated_at`, `is_deleted`(사용 시)는 NOT NULL.

### 5.2 기본값

- **created_at**: `DEFAULT CURRENT_TIMESTAMP` (또는 동등한 표현)
- **updated_at**: 최초는 created_at과 동일하게, 이후 트리거나 애플리케이션에서 갱신.
- **is_deleted**: `DEFAULT false`
- **버전/낙관적 잠금**: `version INTEGER NOT NULL DEFAULT 1` 등 필요 시 사용.

### 5.3 UNIQUE

- **비즈니스상 유일해야 하는 조합**만 UNIQUE 제약 또는 유니크 인덱스로 둔다.  
  예: 로그인 ID, 주문번호, (order_id + line_no) 등.
- **NULL**: PostgreSQL에서는 UNIQUE 컬럼에 NULL 여러 개 허용. “NULL은 서로 다른 값”으로 취급되므로, “하나만 NULL 허용”이 필요하면 부분 유니크 인덱스 등으로 처리한다.

### 5.4 CHECK

- **도메인 규칙**을 CHECK로 보강한다. 예: `amount >= 0`, `status IN ('PENDING', 'CONFIRMED', 'CANCELLED')`.

## 6. 인덱스

### 6.1 생성 기준

- **PK**: 자동 생성. 추가 인덱스 불필요.
- **FK**: 참조하는 컬럼에 **인덱스**를 두어 JOIN·삭제 시 성능·잠금을 고려한다. (필수는 아니나 대부분 권장)
- **조회 조건**: WHERE, ORDER BY에 자주 쓰는 컬럼 조합에 인덱스 검토.  
  예: 목록 조회가 `user_id`, `created_at DESC`면 `idx_order_user_id_created_at` 등.
- **유니크 보장**: UNIQUE 제약 또는 유니크 인덱스.

### 6.2 네이밍

- **일반 인덱스**: `idx_테이블명_컬럼1_컬럼2` (컬럼 순서 = 인덱스 정의 순서)
- **유니크 인덱스**: `uk_테이블명_컬럼1_컬럼2`
- **복합 인덱스**: 자주 쓰는 **조건·정렬 순서**에 맞춰 컬럼 순서를 정한다. (왼쪽 프리픽스가 조건에 사용될 수 있도록)

### 6.3 지양할 것

- **과도한 인덱스**: INSERT/UPDATE 부담이 커진다. 쓰는 쿼리 기준으로 필요한 것만.
- **중복 인덱스**: (A), (A, B)처럼 왼쪽이 같은 경우, (A, B)만 두고 (A)는 제거 검토.
- **긴 문자열 전체 인덱스**: 필요 시 앞부분만 사용하는 prefix 인덱스 또는 전문 검색 등 다른 수단 검토.

## 7. 마이그레이션

### 7.1 도구

- **Flyway** 또는 **Liquibase** 중 하나를 사용하고, **DDL·DML 변경은 모두 마이그레이션 스크립트**로만 적용한다.
- **직접 DB에서 DDL 실행**은 하지 않는다. (환경 간 차이·이력 누락 방지)

### 7.2 파일 명명 (Flyway 예시)

- **형식**: `V{버전}__{설명}.sql`  
  예: `V1__create_user_table.sql`, `V2__add_order_status_to_order.sql`
- **버전**: 순차 증가 숫자(1, 2, 3…) 또는 **타임스탬프**(`V20250220100000__...`)로 중복·순서 충돌을 피한다.
- **구분자**: 버전 번호와 설명 사이 **더블 언더스코어 `__`** (Flyway 기본).
- **설명**: 영문 소문자 + 언더스코어. “무엇을 하는지” 한 줄로 드러나게.

### 7.3 스크립트 내용

- **한 파일에 한 가지 목적** 권장. (테이블 생성, 컬럼 추가, 인덱스 추가 등을 나누면 롤백·검토가 쉽다)
- **Idempotent(재실행 안전)** 가 되도록 할 수 있으면 유리.  
  예: `CREATE TABLE IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`(PostgreSQL 9.5+). 도구에 따라 지원이 다르므로 팀 정책으로 결정.
- **데이터 변경(DML)** 이 필요할 때: 백업·롤백 계획을 세우고, 가능하면 **다운 마이그레이션** 또는 보상 스크립트를 정의해 둔다.

### 7.4 순서·호환성

- **적용 순서**: 버전 번호 순서대로 한 번만 적용된다. 이미 적용된 버전은 변경하지 않는다. (수정이 필요하면 새 버전 파일 추가)
- **하위 호환**: 컬럼 삭제·타입 변경은 **다음 버전**에서 하고, 필요 시 단계적 마이그레이션(새 컬럼 추가 → 데이터 이전 → 구 컬럼 제거)으로 진행한다.

### 7.5 롤백

- Flyway Community는 **자동 롤백**을 지원하지 않는다. 롤백이 필요하면 **수동 롤백 스크립트**를 별도 관리하거나, Liquibase 등 롤백을 지원하는 도구를 검토한다.
- **운영 적용 전**: 스테이징 등에서 먼저 실행해 보고, 백업 후 적용한다.

### 7.6 DML·임시 데이터 스크립트 위치

- **DDL**(테이블/인덱스/제약 생성·변경)은 `docs/ddl/` 폴더에 둔다.
- **DML**(INSERT, UPDATE 등 데이터 생성·시드·마이그레이션용 쿼리)은 **`docs/dml/`** 폴더에 둔다.  
  - 예: 시드 데이터, 테스트/개발용 임시 데이터, 수동 적용용 INSERT/UPDATE 스크립트.  
  - 파일명 예: `om_product_m_data_100.sql`, `cm_code_m_seed.sql`.
- Flyway/Liquibase로 관리하는 **버전 마이그레이션**은 각 도구가 정한 경로에 두고, **문서화·수동 실행용** DML은 `docs/dml/`에 두어 구분한다.

### 7.7 테이블·인덱스 변경 시 단일 DDL 파일 정책

- **테이블 컬럼 추가/수정, 인덱스 추가** 등 스키마 변경이 있을 때 **별도 add·마이그레이션·권장 인덱스용 파일을 새로 만들지 않고**, 해당 테이블의 **DDL 파일 한 곳에 반영**한다.
- 예: `om_order_m` 수정·인덱스 추가 → **`docs/ddl/om_order_m.sql`** 에만 추가. `om_order_m_add_xxx.sql`, `om_order_indexes_recommended.sql` 같은 별도 DDL 파일은 새로 만들지 않는다.
- 목적: DDL 파일 수가 늘어나 관리가 어려워지는 것을 막고, 테이블당 한 파일에서 현재 스키마를 파악할 수 있게 한다.
- 이미 존재하는 add/권장 인덱스용 파일은 참고·과거 적용 이력용으로 둘 수 있으나, **이후 변경은 해당 테이블 DDL 파일에만 추가**한다.

## 8. 권한·보안

- **애플리케이션 계정**: [06 보안 표준](06-보안-표준.md)에 따라 **필요한 권한만** 부여한다. DDL, 다른 스키마 변경, 슈퍼유저 권한은 주지 않는다.
- **개발/테스트 DB**: 운영과 분리. 테스트 시 H2 또는 Testcontainers(PostgreSQL) 사용 시에도, **네이밍·타입 규칙**은 이 표준과 맞추면 이식성이 좋다.
- **비밀**: DB 비밀번호 등은 코드·마이그레이션 파일에 넣지 않고, 환경 변수·시크릿 관리로 주입한다.

## 9. 정리 체크리스트

| 구분 | 표준 요약 |
|------|-----------|
| **테이블** | 단수형, 소문자, 단어 구분 언더스코어 (예: `order`, `order_item`) |
| **컬럼** | snake_case, 불리언은 is_/has_/can_ 접두어 |
| **PK** | 컬럼명 `id`, 타입 BIGSERIAL 또는 UUID 통일 |
| **FK** | `참조테이블_id` (예: `user_id`, `order_id`) |
| **인덱스** | idx_테이블_컬럼, uk_테이블_컬럼 (유니크) |
| **제약** | pk_/fk_/uk_/chk_ 접두어 + 테이블·컬럼 반영 |
| **공통 컬럼** | id, created_at, updated_at (TIMESTAMPTZ), created_by, updated_by, is_deleted(선택) |
| **타입** | 일시 TIMESTAMPTZ, 금액 NUMERIC, 문자열 VARCHAR(n)/TEXT, 코드·상태 VARCHAR |
| **마이그레이션** | Flyway/Liquibase, V버전__설명.sql, 직접 DDL 금지, 적용 순서·롤백 정책. **DML(데이터 생성/시드)** → `docs/dml/`. **테이블/인덱스 변경은 별도 파일 없이 해당 테이블 DDL 파일에 추가** (7.7) |
| **권한** | 앱 계정 최소 권한, 비밀은 환경변수·시크릿 |
